寻味地理作为一款面向大众的商业级 WebGIS 平台，底层架构必须能扛住高并发的 C 端读取，并提供极速的瓦片渲染和空间计算体验。相比于重度写入的学术型系统，商业应用的数据库方案需要将重心放在**读取优化、空间索引（GiST）以及 JSONB 的高效检索（GIN）**上。

以下是为 Postgres 16 + PostGIS 定制的数据库详细建库方案：

### 一、 数据库全局与环境规划

1. **扩展启用**：必须在对应数据库下显式启用 `postgis` 扩展，以激活所有的空间类型和 `ST_` 函数集。

2. **坐标系基准**：全库统一采用 **EPSG:4326**（WGS 84）存储。虽然前端通常使用 Web 墨卡托（EPSG:3857），但 4326 更易于人类理解和外围数据对接，Web 墨卡托的转换应交给 MVT 直出函数在内存中动态完成。

---

### 二、 核心实体表设计方案

#### 1. 生态地理单元表（eco_geo_unit）
* **数据结构**：
  * `eco_id`: 主键，定长字符（CHAR(10)）。
  * `name`: 变长字符（VARCHAR）。
  * `climate`: **JSONB** 类型。用于存储温度带、降水模式等非结构化或层级气候数据。
  * `adjacent_ecos`: **TEXT[]**（文本数组）。存储相邻单元的 ID 列表。
  * `geom`: **GEOMETRY(MultiPolygon, 4326)**。用于存储该生态区的空间面状数据。
* **设计意图**：引入 `geom` 字段是 MVT 底图分发的核心，前端的生态区高亮、点击拾取均依赖此字段直出矢量瓦片。

#### 2. 原材料表（ingredient）
* **数据结构**：
  * `ingredient_id`: 主键，定长字符（CHAR(8)）。
  * `name`: 变长字符（VARCHAR(50)）。
  * `type`: **ENUM** 枚举类型（香辛料、油脂、谷物等）。使用枚举可以在大量数据下节省存储并加快查询。
  * `flavor_tags`: **TEXT[]**（文本数组）。存储诸如“麻”、“辣”、“醇厚”等标签。
  * `origin_ecos`: **TEXT[]**。用于实现 ER 图中“原产于”的 M:N 关系，直接记录原产地生态单元 ID。

#### 3. 成品风味基因型表（flavor_genotype）
* **数据结构**：
  * `genotype_id`: 主键，定长字符（CHAR(12)）。
  * `dish_name`: 变长字符（VARCHAR(50)）。
  * `category`: 变长字符（VARCHAR(30)）。
  * `eco_id`: 外键，关联 `eco_geo_unit(eco_id)`。
  * `genome_vector`: **JSONB** 类型。存储七维风味向量，例如 `{"spicy": 0.8, "numbing": 0.9, ...}`。
  * `coordinates`: **GEOMETRY(Point, 4326)**。确切的发源地坐标点，用于前端在特定层级渲染图标。

#### 4. 传播事件表（dispersal_event）
* **数据结构**：
  * `event_id`: 主键，定长字符（CHAR(12)）。
  * `ingredient_id`: 外键，关联 `ingredient`。
  * `from_eco_id`: 外键，起点生态单元。
  * `to_eco_id`: 外键，终点生态单元。
  * `time_label`: 变长字符（如“明末清初”）。
  * `vehicle`: 变长字符（如“湖广填四川”）。
  * `route_geom`: **GEOMETRY(LineString, 4326)**。记录传播的光流路径线。

---

### 三、 关联实体表设计方案

#### 5. 配方连结表（recipe_link）
* **数据结构**：
  * `genotype_id`: 复合主键之一，兼外键。
  * `ingredient_id`: 复合主键之一，兼外键。
  * `role`: 变长字符或枚举（锅底、蘸料、主涮等）。
  * `importance`: 数值型（NUMERIC 介于 0 到 1 之间），表示该材料对风味向量的贡献权重。
  * `is_native`: 布尔型（BOOLEAN）。

#### 6. 基因溯源链表（genotype_lineage）
* **数据结构**：
  * `ancestor_id`: 复合主键之一，外键指向 `flavor_genotype`。
  * `descendant_id`: 复合主键之一，外键指向 `flavor_genotype`。
  * `mutation_desc`: 文本型（TEXT），记录详细的变异描述。
* **设计意图**：防止循环引用的业务逻辑（例如 A 是 B 的祖先，B 就不能是 A 的祖先）需要在 Python 后端通过图算法约束，数据库层面主要保障双 ID 联合唯一即可。

---

### 四、 面向高并发的极致索引策略

数据库跑得快，全靠索引带。结合 FastAPI 异步直出数据的需求，需建立以下三类核心索引：

1. **PostGIS 空间索引（GiST）**
   * 对 `eco_geo_unit(geom)`、`flavor_genotype(coordinates)`、`dispersal_event(route_geom)` 必须全部建立 **GiST (Generalized Search Tree) 索引**。
   * **作用**：当 FastAPI 接收到前端发来的 XYZ 瓦片请求时，通过计算 Bounding Box 转换后，只有 GiST 索引能做到毫秒级裁切出视口内的空间数据。
2. **JSONB 与数组检索索引（GIN）**
   * 对 `flavor_genotype(genome_vector)`、`eco_geo_unit(climate)` 建立 **GIN (Generalized Inverted Index) 索引**。对 `ingredient(flavor_tags)` 和 `ingredient(origin_ecos)` 数组也建立 GIN。
   * **作用**：当 C 端用户组合搜索“含有‘麻辣’标签且‘油脂’成分极高”的成品时，GIN 索引能让这类 NoSQL 风格的内层属性查询速度比肩纯结构化关系查询。
3. **关系索引（B-Tree）**
   * 对所有的外键（如 `eco_id`, `from_eco_id` 等）显式建立 B-Tree 索引。Postgres 默认不为外键自动建索引，这会在执行跨表 Join 时引发全表扫描。

---

### 五、 架构预留：瓦片直出视图（MVT Views）

为了进一步减轻 Python FastAPI 的运算负担，在 Postgres 中预留专用的**数据库视图 (Views)**。

将复杂的 Join（例如前端地图需要同时显示“生态区边界 + 内部特有的原产地材料名称”）在数据库内部写成只读视图。FastAPI 在响应 HTTP 请求时，直接基于该视图使用 `ST_AsMVTGeom` 提取二进制切片。这规避了在 Python 端组装业务逻辑，做到了“空间数据计算一律不出数据库”。
