# 网页开发第二阶段规划：从展示型网站到服务化工作台

本文档基于 `开发第二阶段规划.md` 的观点，并结合当前 FlavorScape / 寻味地理的实际页面、代码结构和两人协作情况，重新整理第二阶段网页开发路线。

第二阶段的核心目标不是继续堆叠“更酷的页面”，而是让用户感觉这个网站可以为自己的产品提供服务：能导入、能配置、能生成、能导出、能分享、能形成案例资产。

## 1. 第二阶段总体方向

### 1.1 当前问题

当前网站已经有比较强的展示能力：

- `/brand`：智慧大屏 / GeoAtlas，适合展厅级空间叙事。
- `/marketing`：营销海报，已经具备导出 PNG 的服务感。
- `/archive`：实证白皮书，适合做 B 端证明材料。
- `/spread`：传播图谱，适合讲食材历史传播。
- `/map`：风味底图，承担 WebGIS 探索能力。

但用户进入网站后仍然容易产生“我在看一个作品集”的感觉，而不是“我在使用一套服务”。主要问题是：

- 首页和导航更像功能入口集合，缺少“我能生成什么”的工作台心智。
- 用户控制权不足，大部分页面是被动浏览。
- 导出、分享、嵌入、生成报告等闭环还没有贯穿全站。
- 数据案例太少，目前主要依赖汉源花椒、五常大米，缺少更贴近市场和消费品牌的案例。
- 模块之间还没有形成统一的“产品资产包”概念。

### 1.2 第二阶段目标

第二阶段要把网站从“数字展厅”推进为“地方风味品牌空间资产工作台”。

一句话定位：

> 用户输入一个产品或选择一个案例，系统为它生成一套可展示、可传播、可证明、可嵌入的地理品牌资产。

具体目标：

- 建立一个服务化入口，让用户感觉自己是在使用工具。
- 强化每个页面的可操作性，而不是只展示结果。
- 把导出、分享、嵌入、报告生成做成全站共通能力。
- 拓展更多产品案例，让系统看起来不是只为花椒和大米定制。
- 明确两人分工，减少互相覆盖代码。

## 2. 页面模块规划

### 2.1 Brand Studio 工作台

建议新增页面：`/studio`

建议文件：

- `src/views/StudioView.vue`

功能定位：

- 作为首页之后的过渡页，不直接替换当前首页。
- 用户从首页进入后，看到的是“品牌资产工作台”，而不是单个展示页。
- 该页面承担服务化叙事：选择产品、选择输出类型、模拟导入数据、进入不同生成模块。

核心内容：

- “我的产品资产”区域：展示汉源花椒、五常大米、茶叶/奶茶茶底、火锅底料等案例卡片。
- “生成什么”区域：
  - 生成营销海报
  - 生成实证白皮书
  - 开启智慧大屏
  - 查看传播图谱
  - 复制嵌入代码
- “一键导入”区域：
  - 导入检测报告 PDF
  - 输入产品产地
  - 输入品牌名称
  - 模拟 3 秒生成过程

第一阶段可以前端写死，不做真实上传。

交互重点：

- 点击“导入报告”后出现流程动画：
  - 正在识别产品名称
  - 正在提取产地信息
  - 正在匹配地理节点
  - 正在生成空间资产
- 结束后跳转到 `/archive` 或 `/marketing`。

优先级：最高。

原因：

- 它能最快把网站心智从“展示页面集合”改成“服务工具”。
- 不需要复杂后端。
- 对评审、客户、搭档都容易解释。

### 2.2 GeoAtlas 智慧大屏模块

当前文件：

- `src/views/GeoAtlasView.vue`

当前问题：

- 视觉强，但用户只能看系统设定好的节点。
- 大屏适合展厅，但服务感不足。
- 地图依赖后端瓦片，加载失败时体验容易受影响。

第二阶段改造：

- 增加“镜头导播台”面板。
- 支持勾选/隐藏节点：
  - 产地
  - 加工
  - 仓储
  - 市场
  - 餐桌
- 支持切换镜头模式：
  - 自动巡航
  - 产地聚焦
  - 市场路径
  - 餐桌闭环
- 增加“获取嵌入代码”按钮。
- 增加“复制分享链接”按钮。
- 增加加载失败兜底提示：
  - 如果瓦片未加载，仍展示节点 HUD 和文本叙事。

建议新增状态：

- `visibleNodeIds`
- `cameraMode`
- `shareModalOpen`
- `embedCode`

优先级：高。

### 2.3 Marketing 营销海报模块

当前文件：

- `src/views/MarketingView.vue`
- `src/components/StaticGeoMap.vue`

当前优势：

- 已经最接近“服务”。
- 有导出 PNG。
- 有模板风格切换。
- 静态地图适合 html2canvas，不依赖 WebGL。

第二阶段改造：

- 强化模板选择：
  - 极简绿
  - 东方风物
  - 蓝染非遗
  - 新增：新消费品牌风
  - 新增：茶饮供应链风
- 增加“应用到当前产品”逻辑：
  - 从工作台选择产品后进入海报页。
  - 海报自动套用产品名、产区、风味卖点、地图节点。
- 增加导出状态：
  - 输出 PNG
  - 复制小红书文案
  - 复制电商详情页文案
- 增加空白模板预览：
  - 告诉用户这不是固定海报，而是可以替换品牌和产品的模板系统。

优先级：最高。

原因：

- 这是最容易形成用户价值的模块。
- 输出物明确，可带走。
- 可以最快支撑“我们提供服务”的叙事。

### 2.4 Archive 实证白皮书模块

当前文件：

- `src/views/ArchiveView.vue`

当前定位：

- 用于 B 端采购、证明材料、科研感展示。

第二阶段改造：

- 增加“导出 PDF 报告”按钮。
- 优先用浏览器打印样式实现，不急着引入复杂 PDF 库。
- 增加 Print CSS：
  - 隐藏导航
  - 固定 A4 宽度
  - 分页清晰
  - 避免图表跨页断裂
- 增加轻量编辑模式：
  - 修改指标数值
  - 修改检测结论
  - 修改产区说明
- 增加“报告生成中”模拟流程：
  - 正在读取检测指标
  - 正在生成产区证据
  - 正在生成白皮书目录

优先级：高。

原因：

- 该页面最适合建立 B 端可信度。
- PDF 报告比单纯网页更接近真实商业交付。

### 2.5 Spread 传播图谱模块

当前文件：

- `src/views/IngredientSpreadView.vue`

当前定位：

- 食材历史传播、路线、文化叙事。

第二阶段改造：

- 增加“嵌入到品牌故事”能力。
- 增加“传播故事摘要生成”面板：
  - 复制 100 字短文案
  - 复制 300 字品牌故事
  - 复制研学/展陈讲解稿
- 增加与工作台案例联动：
  - 选择某个产品后，传播图谱优先展示相关食材。
- 增加导出：
  - 复制 iframe
  - 导出当前图谱截图

优先级：中。

原因：

- 很有项目深度，但短期商业闭环不如海报和白皮书直接。

### 2.6 Map 风味底图模块

当前文件：

- `src/views/MapView.vue`

当前定位：

- WebGIS 探索底图，支撑项目技术深度。

第二阶段改造：

- 暂时不做重构。
- 重点做“跳转关系”：
  - 从地图节点跳转到对应品牌资产案例。
  - 从工作台进入地图时带筛选参数。
- 增加“保存当前视图为素材”按钮。
- 增加“地图服务状态提示”：
  - 后端未启动时提示用户需要启动 FastAPI。

优先级：中低。

原因：

- 它是技术底座，但不应在短期冲刺里继续扩大范围。

## 3. 案例与内容扩展规划

### 3.1 当前案例不足

目前两个主案例：

- 汉源花椒
- 五常大米

问题：

- 都偏传统农产品。
- 对年轻消费品牌、茶饮、餐饮品牌、地方文旅客户的吸引力不足。
- 叙事有样板感，但市场想象空间还没有完全打开。

### 3.2 第二阶段建议补充案例

优先补 3 类案例：

1. 茶饮供应链案例
   - 示例：某奶茶品牌茶底、茉莉花茶、乌龙茶、云南普洱。
   - 价值：贴近新消费品牌，市场感强。
   - 可输出：茶叶产区大屏、茶底白皮书、门店海报。

2. 火锅/餐饮品牌案例
   - 示例：火锅底料、郫县豆瓣、辣椒、花椒、牛油。
   - 价值：和川渝饮食文化、城市消费场景结合自然。
   - 可输出：供应链空间叙事、餐饮品牌详情页。

3. 地方文旅/非遗食材案例
   - 示例：龙井茶、安吉白茶、云南菌菇、潮汕牛肉丸、沙县小吃。
   - 价值：适合政府、文旅、展厅、研学场景。
   - 可输出：展陈大屏、研学材料、区域品牌报告。

### 3.3 案例数据结构建议

建议后续新增：

- `data/product_cases/*.json`

每个案例包含：

- `id`
- `name`
- `category`
- `brandScenario`
- `province`
- `heroImage`
- `nodes`
- `evidence`
- `marketingCopy`
- `archiveMetrics`
- `spreadStory`
- `recommendedOutputs`

短期可以先不接后端，只让前端页面共享这批 JSON。

## 4. 全站服务化能力规划

### 4.1 导出中心

建议新增统一导出概念：

- 海报导出 PNG
- 白皮书导出 PDF
- 智慧大屏复制链接
- 图谱复制 iframe
- 地图视图导出截图

可以先不做独立页面，只在工作台展示“最近生成的资产”。

### 4.2 嵌入与分享

每个核心页面增加：

- 复制分享链接
- 复制 iframe 代码
- 复制介绍文案

示例：

```html
<iframe src="https://flavorscape.example.com/brand?case=pepper" width="100%" height="720"></iframe>
```

即使现在只是模拟，也能显著提升服务感。

### 4.3 伪数据导入

短期不做真实文件解析，但做完整体验：

- 拖拽上传区域
- 进度动画
- 生成结果跳转
- 提示“当前为演示模式”

这可以解释未来商业模式：

> 用户只需要提供检测报告、品牌资料和产地信息，我们生成完整的空间品牌资产包。

### 4.4 统一 Loading 和空状态

全站统一：

- 地图加载中
- 报告生成中
- 海报导出中
- 数据导入中
- 后端未启动提示

加载动画不应使用像素风字体，统一使用项目全局字体和克制的视觉语言。

## 5. 两人协作分工

### 5.1 分工原则

两个人协作时，不要同时改同一个大文件。

建议按模块和职责分工：

- 一个人负责“服务化交互与前端页面”。
- 一个人负责“数据案例、地图底座、文档与验证”。

### 5.2 角色 A：前端体验与服务闭环

主要负责：

- `StudioView.vue`
- `MarketingView.vue`
- `ArchiveView.vue` 的导出和编辑模式
- 全站分享 / iframe / 导出弹窗
- UI 统一和动效

优先任务：

1. 新建 `/studio` 工作台。
2. 强化 `/marketing` 海报生成服务。
3. 给 `/archive` 增加导出 PDF / 打印样式。
4. 给 `/brand` 增加分享和嵌入弹窗。

避免同时修改：

- `MapView.vue`
- `backend/`
- GIS 数据加载逻辑

### 5.3 角色 B：数据、地图与案例内容

主要负责：

- `data/product_cases/*.json`
- `src/assets/`
- `src/views/MapView.vue`
- `src/views/IngredientSpreadView.vue`
- 后端健康检查和地图瓦片稳定性
- 文档和案例资料补充

优先任务：

1. 整理 3-5 个新增产品案例。
2. 给每个案例补节点、证据、营销文案和白皮书指标。
3. 检查 `/tiles/*`、`/api/*` 在开发环境是否稳定。
4. 给传播图谱补可复用故事摘要。

避免同时修改：

- `MarketingView.vue`
- `ArchiveView.vue`
- `StudioView.vue`

### 5.4 Git 协作建议

建议分支：

- `feature/studio-dashboard`
- `feature/marketing-export`
- `feature/archive-report`
- `feature/product-cases`
- `fix/map-service-status`

每次提交控制在一个模块内，避免一次提交同时改 5 个页面。

合并前至少跑：

```powershell
npm run build
```

涉及地图和后端时再跑：

```powershell
.\start.ps1
```

## 6. 冲刺优先级

### 6.1 立即推进，1 天内完成

1. 新建 `/studio` 工作台。
2. `/marketing` 增加更明确的输出动作：导出 PNG、复制文案。
3. `/archive` 增加打印/PDF 按钮和基础 print CSS。
4. `/brand` 增加复制分享链接和嵌入代码。

### 6.2 第二轮，2-3 天内完成

1. 新增 3 个产品案例。
2. 建立 `data/product_cases/` 数据结构。
3. 工作台接入案例选择。
4. 智慧大屏支持节点显隐和镜头模式。
5. 传播图谱支持复制故事摘要。

### 6.3 第三轮，1 周内完成

1. 统一全站 Loading、弹窗、导出反馈。
2. 建立导出中心或最近生成资产区。
3. 增加案例库页面。
4. 把营销海报、白皮书、智慧大屏的数据来源统一到 `product_cases`。
5. 规划真实后端 CMS 或轻量 JSON 编辑器。

## 7. 第二阶段成功标准

如果第二阶段完成，用户应该能感受到：

- 这不是一个展示网站，而是一套品牌资产生成服务。
- 我可以选择自己的产品。
- 我可以看到系统如何生成海报、报告、大屏和图谱。
- 我可以导出或复制这些成果。
- 我可以想象把这套服务用于茶饮、火锅、文旅、地方农产品和非遗食材。

开发上应该达到：

- 页面职责清晰。
- 数据案例可复用。
- 两人分工不互相覆盖。
- 构建稳定。
- 服务化叙事贯穿首页、工作台和核心输出页面。

第二阶段最重要的判断：

> 不要再单纯追求“页面更震撼”，而要让用户相信“这套系统可以替我生成有用的东西”。
