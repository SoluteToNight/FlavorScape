# 伪数据导入与空间品牌资产包实现方案

## 1. 功能定位

“伪数据导入”不是简单的上传按钮，而是把网站从“展示型产品”推进到“服务型工作台”的入口。

当前短期目标是不解析真实文件，但要让用户完整体验一条商业服务链路：

> 用户提供检测报告、品牌资料和产地信息，系统生成完整的空间品牌资产包。

这里的“空间品牌资产包”可以理解为一组可交付成果：

- 营销海报：面向传播和销售场景。
- 实证白皮书：面向品牌背书、展会、招商、政府项目材料。
- 智慧大屏：面向空间叙事、产地展示、产业链讲解。
- 分享链接与嵌入代码：面向网页传播和客户交付。

因此，这个功能的价值不在于“上传文件本身”，而在于让用户感知：只要提供基础资料，平台就能把零散材料转化为空间化、可视化、品牌化的数字资产。

## 2. 当前演示版已经具备的框架

当前已新增 `/import` 页面，作为演示入口。

已实现内容：

- 拖拽上传区域。
- 本地文件选择。
- 文件清单展示。
- “当前为演示模式”提示。
- 生成进度动画。
- 无文件时自动使用示例资料。
- 生成完成后跳转到：
  - `/marketing` 营销海报
  - `/archive` 实证白皮书
  - `/brand` 智慧大屏

这个版本适合用于答辩、路演或产品原型展示。它不处理真实文件，也不会把文件上传到后端，因此风险低、实现成本低、对现有功能影响小。

## 3. 建议实现路径

### 3.1 第一阶段：前端演示闭环

目标：让用户相信“资料进入系统后，会生成一套成果”。

建议继续完善：

- 增加资料类型状态：
  - 检测报告
  - 品牌资料
  - 产地信息
  - 产品图片
- 增加资料完整度评分：
  - 缺少检测报告：提示“实证能力较弱”
  - 缺少产地信息：提示“空间叙事较弱”
  - 缺少品牌资料：提示“营销文案较弱”
- 增加生成结果总览页或结果弹窗：
  - 海报已生成
  - 白皮书已生成
  - 智慧大屏已生成
  - 分享链接已生成

这个阶段仍然不需要后端解析，所有数据都可以使用前端模拟。

### 3.2 第二阶段：伪解析与模板填充

目标：让导入结果和网站现有四个产品案例发生关系。

实现方式：

- 用户上传文件后，不读取文件内容，只根据文件名、文件类型、用户选择来判断资料类别。
- 页面提供一个“产品类型/案例模板”选择：
  - 汉源花椒
  - 生椰产品
  - 茉莉产品
  - 其他示例产品
- 根据选择，把用户带入对应案例的数据模板。
- 营销海报、实证白皮书、智慧大屏读取同一份案例数据。

这一阶段的重点是“统一数据入口”。用户看到的是导入资料，系统内部使用的是预设案例模板。

建议新增一个前端数据结构：

```js
const assetPackage = {
  productId: 'jasmine',
  sourceMode: 'demo',
  materials: [
    { type: 'report', name: '检测报告.pdf', status: 'mocked' },
    { type: 'brand', name: '品牌资料.docx', status: 'mocked' },
    { type: 'origin', name: '产地信息.xlsx', status: 'mocked' },
  ],
  outputs: {
    marketing: true,
    archive: true,
    atlas: true,
  },
}
```

后续 `/marketing`、`/archive`、`/brand` 可以通过 `productId` 或 query 参数读取同一案例。

### 3.3 第三阶段：后端接入真实上传

目标：开始支持真实文件进入系统，但不一定马上做复杂解析。

建议后端新增接口：

- `POST /api/import/session`
  - 创建一次导入任务。
- `POST /api/import/session/{id}/files`
  - 上传文件。
- `GET /api/import/session/{id}`
  - 获取导入进度和结果。
- `POST /api/import/session/{id}/generate`
  - 触发资产包生成。

文件可以先存为临时文件或对象存储记录，数据库只保存元信息：

- 文件名
- 文件类型
- 文件大小
- 上传时间
- 所属产品案例
- 当前处理状态

这一阶段的关键不是 AI 解析，而是把“导入任务”变成一个后端可以追踪的业务对象。

### 3.4 第四阶段：真实解析与半自动生成

目标：从文件中提取可用字段，减少人工录入。

可解析内容：

- 检测报告：
  - 指标名称
  - 指标数值
  - 检测机构
  - 检测时间
  - 结论摘要
- 品牌资料：
  - 品牌名称
  - 产品名称
  - 核心卖点
  - 目标人群
  - 包装文案
- 产地信息：
  - 产地名称
  - 经纬度或行政区
  - 工艺节点
  - 原料来源
  - 流通路径

但即使进入真实解析阶段，也不建议直接全自动发布。更适合做成“半自动工作台”：

- 系统提取字段。
- 用户确认或修改字段。
- 用户选择输出模板。
- 系统生成结果。

这样更符合真实商业场景，也能避免解析错误直接污染页面内容。

## 4. 前端页面建议

### 4.1 `/import` 导入页

定位：资料进入系统的入口。

应包含：

- 演示模式/正式模式标识。
- 拖拽上传区。
- 文件列表。
- 资料完整度检查。
- 产品案例模板选择。
- 生成按钮。
- 生成进度。
- 生成结果入口。

### 4.2 生成结果总览

可以单独做一个 `/package/:id` 页面，也可以先在 `/import` 页面内完成。

建议展示：

- 产品名称。
- 资产包编号。
- 资料完整度。
- 生成成果列表。
- 每个成果的操作按钮：
  - 查看
  - 编辑
  - 导出
  - 分享

### 4.3 与现有页面的连接

现有页面不需要大改，重点是统一入口参数。

- `/marketing?case=jasmine`
- `/archive?case=jasmine`
- `/brand?case=jasmine`

未来真实导入后可以变成：

- `/marketing?package=xxx`
- `/archive?package=xxx`
- `/brand?package=xxx`

这样既保留当前案例展示，也能逐渐接入用户自定义资产包。

## 5. 后端与数据设计

### 5.1 建议数据表

可以先设计三类表：

- `import_sessions`
  - 一次导入任务。
- `import_files`
  - 用户上传的文件。
- `asset_packages`
  - 生成后的空间品牌资产包。

核心字段示例：

```text
import_sessions
- id
- title
- product_name
- mode
- status
- created_at
- updated_at

import_files
- id
- session_id
- file_name
- file_type
- file_size
- material_type
- storage_path
- status

asset_packages
- id
- session_id
- product_id
- package_data
- status
- created_at
```

其中 `package_data` 可以先用 JSON 保存，等结构稳定后再拆成更细的数据表。

### 5.2 生成结果数据结构

建议把生成结果抽象成统一 JSON：

```json
{
  "product": {
    "name": "茉莉风味产品",
    "origin": "广西横州",
    "brandStory": "..."
  },
  "science": {
    "indicators": [],
    "evidenceSummary": "..."
  },
  "space": {
    "nodes": [],
    "routes": []
  },
  "outputs": {
    "poster": {},
    "whitepaper": {},
    "atlas": {}
  }
}
```

前端三个成果页面都读取这份数据，只是展示方式不同。

## 6. 商业模式表达

这个功能可以帮助网站解释未来服务能力：

- 客户不需要懂地图、设计、数据可视化。
- 客户只需要提交已有资料。
- 平台负责把资料转化为：
  - 空间叙事
  - 品牌内容
  - 实证材料
  - 可传播视觉资产

可以包装成三档服务：

- 基础版：上传资料，生成标准海报和基础白皮书。
- 专业版：加入空间路径、产地地图、检测指标可视化。
- 定制版：加入品牌视觉系统、工艺漫画叙事、展厅大屏和嵌入式网页。

## 7. 双人分工建议

你们两个人合作，可以这样推进：

### 开发者 A：前端与交互

- 完善 `/import` 页面。
- 增加资料完整度检查。
- 增加生成结果总览。
- 打通 `/marketing`、`/archive`、`/brand` 的参数跳转。
- 负责视觉风格和用户流程。

### 开发者 B：数据与后端

- 设计导入任务数据结构。
- 新增导入接口。
- 保存文件元信息。
- 准备案例模板数据。
- 后续研究 PDF、Word、Excel 的真实解析。

短期内，A 可以先把演示体验做完整；B 可以先不接真实文件，只整理数据结构和接口草案。这样不会因为后端解析难度拖慢前端产品化展示。

## 8. 风险与注意事项

- 不要过早做复杂真实解析。PDF、Word、Excel 的格式差异很大，容易消耗大量时间。
- 不要让用户误以为当前已经真实解析文件，页面必须持续标注“当前为演示模式”。
- 不要让上传文件进入不可控状态，正式上传前必须考虑隐私、存储、删除和权限。
- 不要让导入页变成孤立功能，它必须和海报、白皮书、智慧大屏形成闭环。
- 不要一次性重构现有三个成果页面，建议先通过 query 参数和前端模拟数据逐步接入。

## 9. 下一步建议

下一步最值得做的是完善第一阶段和第二阶段：

1. 在 `/import` 增加“产品案例模板选择”。
2. 增加“资料完整度评分”。
3. 生成完成后进入一个“资产包总览”。
4. 总览页跳转到现有三个成果模块，并带上 `case` 参数。

这样可以在不增加真实解析难度的情况下，把网站的服务逻辑完整讲出来。
