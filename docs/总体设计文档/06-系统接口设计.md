# 系统接口设计

本章对"寻味地理"风味博物志 WebGIS 系统的外部接口和内部接口进行详细设计。外部接口描述系统与外部数据源、浏览器运行环境的交互方式；内部接口描述前后端之间的 API 约定。

## 外部接口

### 与外部数据源的接口

#### Natural Earth 数据接口

系统通过以下方式消费 Natural Earth 数据：

| 数据 | 接口方式 | 说明 |
| --- | --- | --- |
| HYP 栅格底图 | 本地文件读取 | COG GeoTIFF 文件部署于服务器硬盘，通过 rio-tiler `Reader` 类按 XYZ 瓦片区域读取 |
| 10m 物理矢量 | 本地文件读取 + 内存常驻 | Shapefile 通过 fiona 在服务启动时一次性加载至内存 `vector_data` 字典 |

Natural Earth 数据无需网络请求，不依赖外部 API。

#### WWF TEOW 数据接口

| 数据 | 接口方式 | 说明 |
| --- | --- | --- |
| 生态区 Shapefile | 本地文件读取 → PostgreSQL | 通过 fiona + shapely 读取本地 Shapefile，seed.py 批量写入 `eco_geo_unit` 表 |

WWF TEOW 数据为一次性导入操作，运行时通过 PostgreSQL 查询获取，不依赖外部 API。

#### Wikimedia Commons 图片接口

| 数据 | 接口方式 | 说明 |
| --- | --- | --- |
| 菜品图片 | HTTP GET（远程 URL） | 17 张菜品图片以远程 URL 形式存储于 `app_data.py` 的 `FLAVOR_MEDIA` 字典中，前端通过 `<img src>` 直接加载 |

图片加载失败时，后端 `/api/flavors` 接口返回 SVG data URI 占位图像。

### 与浏览器的接口

系统运行于浏览器环境中，依赖以下浏览器能力：

| 接口/API | 用途 | 最低版本要求 |
| --- | --- | --- |
| WebGL 2.0 | MapLibre + Deck.gl 3D 渲染 | Chrome 90+, Firefox 88+, Safari 15+ |
| Canvas 2D | 首页粒子动画、叙事页路径缩略 | 所有现代浏览器 |
| Fetch API | HTTP 请求 | 所有现代浏览器 |
| CSS Transform / Animation | UI 动效、气泡节点定位 | 所有现代浏览器 |
| Touch Events | 移动端触摸交互（后续适配） | 所有现代浏览器 |

## 内部接口

### 前后端接口设计

前后端通信采用 RESTful 风格，所有接口统一前缀 `/api/` 或 `/tiles/`，请求和响应均使用 JSON 格式（瓦片接口除外）。

#### 数据格式约定

**成功响应格式**：

```json
{
  "data": [ ... ],
  "total": 17
}
```

或直接返回数组/对象（列表类接口）。

**错误响应格式**：

```json
{
  "detail": "错误描述信息"
}
```

**空间数据编码**：所有坐标和几何数据以 GeoJSON 格式编码（RFC 7946），坐标系为 WGS84 (EPSG:4326)。

##### GET /api/search

**说明**：全局搜索，模糊匹配风味节点和传播路线。

**请求参数**：

| 参数 | 类型 | 必需 | 说明 |
| --- | --- | --- | --- |
| `q` | string | 是 | 搜索关键词 |

**响应结构**：

```json
[
  {
    "type": "node",
    "id": "chengdu-hotpot",
    "label": "川渝红汤火锅",
    "region": "四川盆地 · 成都",
    "colorHex": "#C41E3A"
  },
  {
    "type": "route",
    "id": "silk-road",
    "label": "丝绸之路",
```

#### 瓦片服务接口

##### GET /tiles/raster/{z}/{x}/{y}.png

**说明**：返回 HYP 栅格底图的 XYZ 瓦片（PNG 格式，256×256 像素）。

**请求参数**：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `z` | int | 缩放等级（0~10） |
| `x` | int | 瓦片列号 |
| `y` | int | 瓦片行号 |

**响应**：`Content-Type: image/png`，`Cache-Control: max-age=86400`。

**实现机制**：通过 rio-tiler `Reader` 读取 COG 文件对应的瓦片区域，经 LRU 缓存（512 条目）加速重复请求。超出 COG 数据范围的请求返回 1×1 透明 PNG。

**性能要求**：首次渲染（冷读取）< 2 秒，缓存命中 < 50 ms。

##### GET /tiles/mvt/{z}/{x}/{y}.pbf

**说明**：返回生态区边界的 Mapbox Vector Tile（PBF 二进制格式）。

**请求参数**：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `z` | int | 缩放等级 |
| `x` | int | 瓦片列号 |
| `y` | int | 瓦片行号 |

**响应**：`Content-Type: application/x-protobuf`，`Content-Encoding: gzip`，`Cache-Control: max-age=3600`。

**实现机制**：通过 PostGIS `ST_AsMVTGeom` 将 `eco_geo_unit.boundary` 裁剪至瓦片包络并简化（容差 0.02°），`ST_AsMVT` 聚合为 MVT 瓦片。最大返回 200 个要素。

**性能要求**：响应时间 < 500 ms。

##### GET /tiles/vector/{layer}

**说明**：返回指定矢量图层的 GeoJSON FeatureCollection。

**支持的 layer 值**：`coastline`、`rivers`、`land`、`ecoregions`。

**响应**：`Content-Type: application/geo+json`，`Cache-Control: max-age=3600`。

**实现机制**：`coastline`、`rivers`、`land` 从内存中的 `vector_data` 字典直接返回；`ecoregions` 每次从 PostgreSQL 查询（数据量约 827 条记录，经 ST_Simplify 简化）。

**性能要求**：`coastline`/`rivers`/`land` < 50 ms（内存直读），`ecoregions` < 500 ms。

##### GET /tiles/vector

**说明**：返回所有可用矢量图层的元信息（名称和要素数量）。

**响应结构**：

```json
{
  "layers": ["coastline", "rivers", "land", "ecoregions"],
  "counts": {
    "coastline": 1,
    "rivers": 1,
    "land": 1,
    "ecoregions": 827
  }
}
```

##### GET /tiles/status

**说明**：返回瓦片服务的运行状态。

**响应结构**：

```json
{
  "rasterReady": true,
  "vectorLayers": ["coastline", "rivers", "land", "ecoregions"],
  "cacheSize": 512,
  "cacheEntries": 45,
  "cacheHitRate": 0.78
}
```

#### 系统接口

##### GET /health

**说明**：返回服务健康状态。

**响应结构**：

```json
{
  "status": "ok",
  "raster_ready": true,
  "vector_layers": ["coastline", "rivers", "land", "ecoregions"]
}
```

**实现机制**：检查 COG 文件是否存在、矢量图层是否加载完毕、数据库连接池是否初始化。FastAPI 的 lifespan handler 在所有启动任务（栅格提取、矢量加载、数据库连接池初始化、生态区数据加载）完成后才接受请求，因此 `/health` 返回 `"ok"` 即表示所有服务就绪。

### 子系统间的数据流设计

#### 数据查询流

```
用户操作 (浏览器)
   │
   ├── 页面加载 ──→ GET /api/flavors ──→ flavor_genotype + recipe_link + ingredient ──→ JSON
   ├── 全局搜索 ──→ GET /api/search?q= ──→ tsvector/ILIKE ──→ JSON
   ├── 地图浏览 ──→ GET /tiles/raster/{z}/{x}/{y}.png ──→ rio-tiler Reader ──→ PNG
   ├── 生态区渲染 ──→ GET /tiles/vector/ecoregions ──→ eco_geo_unit ──→ GeoJSON
   └── 生态区MVT ──→ GET /tiles/mvt/{z}/{x}/{y}.pbf ──→ ST_AsMVT ──→ PBF
```

#### 数据写入流

```
LLM 管线
   │
   ├── 风味向量提取 ──→ JSON (标准 schema) ──→ 人工审核 ──→ import_json.py ──→ PostgreSQL
   └── 传播事件录入 ──→ JSON (标准 schema) ──→ 人工审核 ──→ import_json.py ──→ PostgreSQL
```

### 接口错误码约定

| HTTP 状态码 | 说明 | 场景 |
| --- | --- | --- |
| 200 | 成功 | 正常响应 |
| 400 | 请求参数错误 | 缺少必需参数、参数类型错误 |
| 404 | 资源不存在 | 请求的瓦片超出 COG 范围（返回透明占位瓦片）、请求不存在的矢量图层 |
| 500 | 服务器内部错误 | 数据库连接失败、COG 文件读取失败 |
| 503 | 服务未就绪 | 启动过程中（栅格未解压、矢量未加载完毕）收到请求 |

### 前后端接口调用约定

1. **开发环境代理**：前端 Vite 开发服务器（:5173）通过 `vite.config.js` 中的 proxy 配置将 `/api/*` 和 `/tiles/*` 请求代理转发至 FastAPI（:8001）；
2. **生产环境代理**：Nginx 反向代理将 `/api/*` 和 `/tiles/*` 请求转发至 FastAPI（:8001），`/assets/*` 及页面请求由 Express（:3001）托管 `dist/` 静态文件；
3. **CORS**：后端配置 CORS 中间件，允许开发域名（`localhost:5173`、`localhost:4173`）的跨域请求；
4. **请求方式**：所有 API 请求使用 GET 方法，不接受 POST/PUT/DELETE（当前系统为只读，无数据写入 API）；
5. **超时设置**：前端 fetch 请求超时 30 秒，瓦片请求超时 10 秒。
