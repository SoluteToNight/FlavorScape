# 系统概况

本章主要对"寻味地理"风味博物志 WebGIS 系统建设的背景、目标和建设任务等内容作概要介绍，并对文档中出现的术语、缩写等进行解释，方便全文的阅读。

## 系统背景

饮食文化的空间分布一直缺乏直观的 GIS 表达工具。传统美食地图仅回答"哪里有什么"，却无法解释"味道为什么如此"——同样的食物（如火锅），在四川为麻辣红汤，在海南为椰青锅底，在北京为清汤涮肉。这种同源异构现象背后，是地理环境对风味进化的筛选与塑造。

本项目提出将食物风味视为一种**可遗传、可变异的生物基因**，将地理环境视为**自然选择的压力**：

- **同一种食物**（如"火锅"）视为一个属；
- **不同地域的版本**（如川渝红汤锅、海南椰子鸡锅）视为同一属下的不同基因型；
- **六维风味向量**（麻、辣、咸、酸、甜、鲜）是对基因型的数字化编码。

在上述理念下，"寻味地理"风味博物志 WebGIS 系统旨在利用 WebGIS 技术、PostgreSQL/PostGIS 空间数据库以及前端可视化技术（MapLibre、Deck.gl），建设一个交互式的中国饮食风味地理探索平台，实现对全国范围内地域风味节点的空间分布展示、历史传播路径的可视化追踪、生态地理单元与风味演化关系的直观呈现，最终让用户在一张地图上"亲眼看见风味进化的地理足迹"。

## 系统目标

本项目总的建设目标为：以 WWF 全球陆地生态区划（TEOW）为空间分析框架，以 Natural Earth 物理底图为视觉基底，建设一个面向大众的 WebGIS 风味博物志。系统应具备以下能力：

1. 支持交互式地图的缩放、漫游、投影切换（球面/墨卡托自适应切换）；
2. 提供四层地图叠加显示：物理底图（L0）、生态区边界（L1）、风味传播路径（L2）、风味节点（L3）；
3. 实现节点、路线、生态区三者之间的互斥选择与联动探索；
4. 提供全局搜索功能，模糊匹配风味节点和传播路线；
5. 以 ECharts 风味雷达图直观展示六维风味向量；
6. 以 Deck.gl 粒子流光动画表现历史传播路径的时空走向；
7. 提供风味基因库（卡片浏览）和时空叙事馆（章节阅读）两种非地图浏览模式；
8. 建立基于大语言模型（LLM）的数据智能化生产管线，从海量菜谱文献与历史古籍中自动提取结构化风味向量及传播事件，支撑系统数据的规模化扩展。

根据系统建设的总目标，建设任务主要包括四大部分：数据库建设、应用系统建设、数据智能化生产管线建设及基础数据准备与处理。

1. **数据库建设**

完成基于 PostgreSQL/PostGIS 的空间数据库设计与搭建，包括全球生态地理单元（827 个 WWF TEOW 生态区）、风味基因型、风味传播事件、原材料字典、叙事章节等核心数据表的建库，建立空间索引（GiST）、全文检索索引（GIN/GiST）及多表关联查询体系。当前系统以手动标注的 17 个风味节点与 5 条传播路径作为种子演示数据，完整数据内容由 LLM 数据管线规模化生产后导入。

2. **应用系统建设**
   - 建立基于 Vue 3 + Vite 的前端 SPA 框架；
   - 建立基于 MapLibre GL JS + Deck.gl 的交互式地图模块（四层数据叠加）；
   - 建立风味节点气泡聚类与联动选择机制；
   - 建立基于 ECharts 的风味雷达图与气候曲线图组件；
   - 建立风味基因库（LibraryView）卡片浏览页面；
   - 建立时空叙事馆（NarrativeView）章节阅读页面；
   - 建立基于 FastAPI + rio-tiler 的后端栅格瓦片服务；
   - 建立基于 PostGIS 的矢量瓦片（MVT）与 GeoJSON 服务；
   - 建立全局搜索 API（PostgreSQL tsvector 全文搜索）。

3. **数据智能化生产管线建设**
   - 建立基于 LLM 提示工程的风味向量自动提取管线：爬取开源菜谱文献及美食文本语料，通过 LLM 将非结构化描述文本转化为标准化六维风味向量（麻、辣、咸、酸、甜、鲜）及食材配方 JSON；
   - 建立基于 LLM 的结构化传播事件录入管线：从历史古籍、饮食人类学研究文献中自动抽取传播事件的起点、终点、传播媒介、时期标签及文献摘录，写入数据库 `dispersal_event` 表。

4. **基础数据准备与处理**
   - Natural Earth 栅格底图（HYP_HR_SR_W_DR）的 COG 格式转换；
   - WWF TEOW 生态区 Shapefile 的入库与中文名映射；
   - 手动标注 17 个风味节点的六维向量与食材配比（种子演示数据）；
   - 手动数字化 5 条历史传播路径（种子演示数据）；
   - 5 个叙事章节的编纂。

## 系统建设任务

"寻味地理"风味博物志 WebGIS 系统是一个前后端分离的单页应用，建设任务围绕 WebGIS 可视化核心、数据管理与服务、前端交互体验三条主线展开。

### 系统总体构架

系统采用 B/S 架构，前端为 Vue 3 单页应用，后端为 FastAPI RESTful 服务，数据库为 PostgreSQL 16 + PostGIS。开发环境由三个进程组成：FastAPI（:8001）、Express 开发代理（:3001）、Vite 热更新服务器（:5173）。浏览器访问 Vite 开发服务器，Vite 将 `/api/*` 和 `/tiles/*` 请求代理转发至 FastAPI。

核心架构如下：

```
Browser (Vite :5173)
  ├── Vue Router (5 路由)
  ├── Pinia Store (节点/路线/生态区互斥选择)
  ├── MapLibre GL JS (底图渲染)
  ├── Deck.gl MapboxOverlay (WebGL 数据图层)
  └── ECharts (雷达图/气候图)
         │ proxy /api/*, /tiles/*
FastAPI (:8001)
  ├── routers/api.py   → 业务数据 (flavors/routes/chapters/search)
  ├── routers/tiles.py → 瓦片服务 (栅格XYZ / 矢量GeoJSON / MVT)
  ├── tile_cache.py    → LRU (512 条目)
  └── startup.py       → 栅格提取 / 矢量加载
         │
PostgreSQL 16 + PostGIS
  ├── eco_geo_unit (827 生态区)
  ├── flavor_genotype (17 风味节点)
  ├── dispersal_event (5 传播路径)
  ├── ingredient / recipe_link / ingredient_origin
  ├── chapter (5 叙事章节)
  └── data_source (4 数据来源)
```

### 地图图层体系

地图页面（MapView.vue）是系统的核心交互界面，实现了四层空间数据的叠加显示：

| 图层 | 名称 | 数据类型 | 渲染引擎 | 交互 |
|------|------|----------|----------|------|
| L0 | 物理底图 | Natural Earth HYP 栅格 + 海岸线/河流矢量 | MapLibre raster/line layer | 背景，不可交互 |
| L1 | 生态区边界 | WWF TEOW 827 个生态区 MultiPolygon | MapLibre geojson line layer | 点击选择生态区 |
| L2 | 传播路径 | 5 条历史传播路线 LineString | Deck.gl PathLayer + TripsLayer | 悬停提示，点击选择 |
| L3 | 风味节点 | 17 个中国城市风味节点 Point | Deck.gl ScatterplotLayer + HTML 气泡 | 点击选择节点 + flyTo 飞行 |

地图支持球面/墨卡托投影自动切换——缩放等级 < 2.2 时为 globe 投影，2.2~2.8 为过渡 morph 投影，> 2.8 为 mercator 投影。低缩放等级下风味节点自动聚合成气泡簇。

### 页面功能模块

| 路由 | 页面 | 核心功能 |
|------|------|----------|
| `/` | 首页 (HomeView) | Canvas 粒子动画背景、品牌展示、"开启地图"入口按钮 |
| `/map` | 探索地图 (MapView) | 核心 WebGIS：四层地图 + 右侧详情面板 (MapInspector) + 全局搜索 + 图层开关 |
| `/library` | 风味基因库 (LibraryView) | 分类筛选 (辛辣/鲜甜/咸香/酸鲜) + 卡片网格 + 风味雷达图 |
| `/narrative` | 时空叙事馆 (NarrativeView) | 5 个历史章节阅读 + 路径动画缩略图 |
| `/about` | 关于 (AboutView) | 数据来源说明 + LLM 辅助数据清洗方法 |

### 数据智能化生产管线

系统设计了一套基于大语言模型的数据生产管线，用于将非结构化的自然语言文献自动转化为结构化数据库记录，支撑系统从种子演示数据向规模化内容的扩展。

**管线一：风味向量自动提取**

面向海量菜谱文献（包括开源菜谱网站、饮食人类学专著、地方美食志等），通过 LLM 提示工程自动完成以下转换：

- 输入：非结构化菜品描述文本（含口感、用料、烹饪技法等自然语言描述）；
- 输出：标准化 JSON 结构，包含六维风味向量（麻/辣/咸/酸/甜/鲜，数值 0~1）、食材列表与角色标注（锅底/主料/蘸料/配菜）、所属食物属分类；
- 入库目标：`flavor_genotype` 表（风味向量→`genome_vector` JSONB 字段）、`recipe_link` 表（食材→角色/贡献度）。

LLM 提示需要嵌入领域知识约束——例如四川盆地高湿环境与花椒碱之间的因果链条、海南椰子对牛油的生态替代逻辑——使提取结果不仅反映成品口味，也编码了地理成因。

**管线二：传播事件结构化录入**

面向历史古籍（如《齐民要术》《本草纲目》及地方志、饮食交流史研究文献），通过 LLM 自动抽取传播事件的结构化字段：

- 输入：古籍段落或研究论文摘要；
- 输出：起点生态单元（`from_eco_id`）、终点生态单元（`to_eco_id`）、传播媒介（海运/内河/陆路/人为引种）、时期标签（`time_label`）、时序排序值（`time_order`）、文献摘录（`literature_excerpt`）；
- 入库目标：`dispersal_event` 表。

该管线的关键挑战在于从含蓄的古文描述中精确锚定地理参照（如"自天竺经南海至交广"需映射至 TEOW 生态区编码），LLM 需配合地理知识库进行多跳推理。

> 当前系统中风味节点与传播路径为手动标注的种子演示数据（17 节点、5 路径）。上述两条管线设计完成、待实现后将接管数据生产流程。

### 数据库体系

系统采用单数据库单 Schema (PostgreSQL/PostGIS)，包含以下应用数据库（当前为种子演示数据，规模化数据由 LLM 管线生产后导入）：

1. **基础地理与生态区数据库**：WWF TEOW 全球陆地生态区划（827 个生态区，含 MultiPolygon 边界、气候带、面积等属性），作为风味演化的空间分析框架；
2. **风味基因型数据库**：存储各地域风味节点，包含六维风味向量（JSONB）、地理坐标（POINT）、菜品描述、食材配方（通过 `recipe_link` 关联）。当前录入 17 个手动标注的种子演示节点，数据生产管线设计目标为从菜谱文献中规模提取；
3. **传播事件数据库**：存储历史传播路径，含路径几何（LineString）、传播媒介、时期标签与文献摘录。当前录入 5 条手动数字化的种子演示路径，数据生产管线设计目标为从古籍中自动抽取；
4. **叙事内容数据库**：存储叙事章节（标题、年代标签、正文、文献引用），当前 5 章；
5. **系统元数据库**：存储数据来源说明（Natural Earth、WWF TEOW、Wikimedia Commons 图片等）。

## 术语、缩写和解释

| **缩写、术语** | **解释** |
| --- | --- |
| 寻味地理 / xunwei-dili | 本项目名称，意为在地理空间中探寻食物风味。 |
| 风味基因型 (FlavorGenotype) | 本项目的核心数据实体，指在某个特定生态单元下诞生的具体地方食物，携带完整六维风味向量（麻、辣、咸、酸、甜、鲜）。 |
| 风味向量 / genome_vector | 以 JSON 格式存储的六维数值（0~1），量化一道菜品在麻、辣、咸、酸、甜、鲜六个维度上的强度。 |
| 生态地理单元 (EcoGeoUnit) | WWF TEOW 定义的全球陆地生态区划基本单位，具有均质气候和物产特征。本系统使用其作为风味演化的空间分析容器。 |
| TEOW | Terrestrial Ecoregions of the World，世界自然基金会（WWF）编制的全球陆地生态区划数据。 |
| HYP | Natural Earth Cross-Blended Hypsometric Tints，一种综合高程渲染栅格底图。 |
| 传播事件 (DispersalEvent) | 某食材/风味在时空中的一次位移（如辣椒从美洲传入四川），包含起点、终点、路径几何和时期信息。 |
| COG | Cloud Optimized GeoTIFF，一种支持 HTTP 范围请求的 GeoTIFF 格式，允许按需读取瓦片区域而不需要下载整个文件。 |
| LRU | Least Recently Used，最近最少使用缓存淘汰策略。本系统瓦片缓存使用 LRU 策略，最大 512 条目。 |
| MVT | Mapbox Vector Tile，一种高效的矢量瓦片二进制格式，由 PostGIS ST_AsMVT 函数直接生成。 |
| SPA | Single Page Application，单页应用。前端通过 Vue Router 管理页面切换，无需刷新浏览器。 |
| rio-tiler | Python 库，用于从 COG GeoTIFF 高效读取并渲染栅格瓦片。本系统使用其 Reader API。 |
| Deck.gl | Uber 开源的 WebGL 数据可视化框架，本系统通过其 MapboxOverlay 在 MapLibre 地图上叠加 PathLayer、TripsLayer 和 ScatterplotLayer。 |
| MapLibre GL JS | MapLibre 社区维护的开源地图渲染引擎，本系统使用其渲染底图和矢量图层。 |
| Pinia | Vue 3 官方推荐的状态管理库。本系统使用 Pinia 管理节点/路线/生态区的互斥选择状态。 |
