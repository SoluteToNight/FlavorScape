## 1. 项目概况与实现度总览

### 1.1 项目定位
"寻味地理"是一个 WebGIS 风味博物志应用，探索地理环境如何塑造风味演化。当前版本已实现核心地图交互体验、后端数据服务、PostGIS 数据库层和五页面前端 SPA。

### 1.2 技术栈实现

| 层级 | 设计文档建议 | 实际实现 |
|:---|:---|:---|
| **前端框架** | Vue 3 或 React | **Vue 3.4**（Composition API + `<script setup>`） |
| **状态管理** | 轻量状态管理 | **Pinia 2.1.7**（单 Store） |
| **路由** | — | **Vue Router 4.4**（5 路由，懒加载） |
| **地图引擎** | MapLibre GL JS | **MapLibre GL JS 4.5** ✓ |
| **GIS 数据驱动** | Deck.gl | **Deck.gl 9.0**（PathLayer, TripsLayer, ScatterplotLayer, MapboxOverlay） ✓ |
| **图表库** | ECharts | **已安装但未使用** — 所有图表为手写 SVG/Canvas |
| **样式管理** | CSS 自定义属性 | **CSS Design Tokens**（global.css 变量系统） ✓ |
| **后端框架** | — | **FastAPI 0.111** |
| **数据库** | — | **PostgreSQL + PostGIS** |
| **瓦片服务** | — | **rio-tiler 5.0**（COG Reader） |
| **构建工具** | — | **Vite 5.3** |
| **生产服务器** | — | **Express 4.19** |

### 1.3 核心隐喻实现
- **"风味是基因"** → 实现为 `flavor_genotype` 表 + `genome_vector` JSONB（六维：麻 spicy / 辣 numbing / 咸 salty / 酸 sour / 甜 sweet / 鲜 umami）
- **"大地是编辑者"** → 实现为 `eco_geo_unit` 表 + `climate` JSONB + WWF TEOW 生态区叠加
- **"同源异构"** → 通过 `dish_family` 字段聚合同族基因型（如 "火锅" 属包含 10 种地域版本）

| 列 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|
| `eco_name` | VARCHAR(150) | **PK** | 生态区英文标准名（SHP ECO_NAME） |
| `eco_name_cn` | VARCHAR(100) | | 中文名（首批留空，待 LLM 翻译） |
| `eco_code` | VARCHAR(10) | | 紧凑编码，如 `"AA0101"` |
| `realm` | VARCHAR(20) | | 生物地理界全称（SHP REALM_1） |
| `biome` | VARCHAR(80) | | 生物群系英文名（BIOME 查表） |
| `biome_cn` | VARCHAR(30) | | 生物群系中文名（BIOME 查表） |
| `area_km2` | NUMERIC | | 面积（km²） |
| `climate` | JSONB | | 气候属性（暂 NULL） |
| `adjacent_ecos` | TEXT[] | | 相邻生态区列表（暂空） |
| `dominant_ingredients` | TEXT[] | | 本地优势原材料（暂空） |
| `description` | TEXT | | 生态环境中文描述（暂 NULL） |

| 类型 | 数量 | 覆盖 |

### 3.4 数据来源与管道

```
app_data.py (硬编码 17 FLAVOR + 5 ROUTE + 5 CHAPTER)
wwf_terr_ecos_merged.shp (827 生态区 + 边界几何)
  │
  ├─→ seed.py → PostgreSQL (schema.sql → INSERT → verify)
  │     ├─ eco_geo_unit: SHP 直接导入 827 行（含 boundary GEOMETRY）
  │     └─ 其余表: 从 app_data.py 生成
  │
  └─→ query.py (运行时合并 DB 行 + UI 元数据)
       │
       ├─ fetch_flavors(): DB JOIN + 按 dish 名匹配 app_data 颜色/标签
       ├─ fetch_routes(): DB 路径几何 + 按 event_id 顺序匹配 app_data 元数据
       ├─ fetch_chapters(): 纯 DB 查询
       └─ search(): 内存搜索 + 去重（最多 8 条）
```

> **⚠️ 关键耦合**：`fetch_routes()` 按 `event_id` 顺序与 `app_data.ROUTES` 位置索引对齐。重排 ROUTES 数组而不重新种子数据会导致名称/颜色/类型与几何体错位。

### 4.1 端点总览（10 端点）

| # | 方法 | 路径 | 描述 |
|:---|:---|:---|:---|
| 1 | GET | `/health` | 健康检查（光栅就绪状态 + 矢量图层名） |
| 2 | GET | `/api/flavors` | 全部 17 个风味节点（含坐标/原料/评分） |
| 3 | GET | `/api/routes` | 5 条传播路径（含 LineString 几何） |
| 4 | GET | `/api/chapters` | 5 篇历史叙事章节 |
| 5 | GET | `/api/data-sources` | 4 个数据来源引用 |
| 6 | GET | `/api/search?q=` | 模糊搜索（节点+路线，最多 8 结果） |
| 7 | GET | `/tiles/raster/{z}/{x}/{y}.png` | XYZ 光栅瓦片（HYP 地形，z 0-10） |
| 8 | GET | `/tiles/vector/{layer}` | 矢量图层 GeoJSON（coastline/rivers/land/ecoregions） |
| 9 | GET | `/tiles/vector` | 列出可用矢量图层及要素数 |
| 10 | GET | `/tiles/status` | 瓦片服务状态（光栅/矢量/缓存统计） |

### 4.2 瓦片渲染管道

```
HTTP GET /tiles/raster/{z}/{x}/{y}.png
  → 验证 z ∈ [0,10]
  → 检查 LRU 缓存（512 条目，线程安全 OrderedDict）
  → 缓存命中：直接返回 PNG 字节
  → 缓存未命中：ThreadPoolExecutor (max_workers=6)
      → rio_tiler.io.Reader(GeoTIFF)
      → .tile(x, y, z, tilesize=256)
      → .render(img_format="PNG")
      → 存入缓存
  → TileOutsideBounds：返回透明 256×256 PNG 兜底
  → 响应头：Cache-Control: max-age=86400
```

### 4.3 矢量数据流

```
启动时（startup.py）：
  ZIP 解压 → fiona 读取 SHP → shapely 简化（容差 0.05-0.08°）
  → 存入内存 vector_data 字典
    ├── coastline: X 个要素
    ├── rivers: X 个要素
    ├── land: X 个要素
    └── ecoregions: 800+ 个要素（含 BIOME_NAME 中文标签）

运行时（tiles.py）：
  GET /tiles/vector/{layer} → JSONResponse(vector_data[layer])
  Cache-Control: max-age=3600
```

### 4.4 中间件

| 顺序 | 中间件 | 参数 |
|:---|:---|:---|
| 1 | GZipMiddleware | minimum_size=1024 |
| 2 | CORSMiddleware | allow_origins=[localhost:5173, :4173 × 2], GET only |

### 5.2 四层地图系统（L0-L3）

| 层 | 名称 | 渲染引擎 | 数据来源 | 默认可见 | 控制方式 |
|:---|:---|:---|:---|:---|:---|
| **L0** | 纯净物理底图 | MapLibre (raster) + MapLibre (line) | `/tiles/raster/` + `/tiles/vector/coastline,rivers` | ✓ 是 | 开关可切换 |
| **L1** | 自然生态单元 | MapLibre (line) | `/tiles/vector/ecoregions` | ✓ 是（透明 0.15） | 开关 + 上下文强调 |
| **L2** | 脉冲流光传播 | Deck.gl (PathLayer + TripsLayer) | `/api/routes` | ✗ 否（上下文触发） | 上下文驱动 + 开关允许 |
| **L3** | 实体锚点 | Deck.gl (ScatterplotLayer) | `/api/flavors` | ✓ 是 | 开关可切换 |

**L2 上下文逻辑**：选择节点或路线时自动显示 L2；选择生态区时自动隐藏 L2。用户可通过开关禁用 L2 参与上下文触发。

**L1 强调逻辑**：选择生态区时 L1 不透明度从 0.15 增强到 0.85。

### 5.3 Deck.gl 图层详情

| 图层 ID | 类型 | 数据 | 配置 |
|:---|:---|:---|:---|
| `path-layer` | PathLayer | routes[] (5 路径) | 实线=陆路，虚线(6,3)=海路。颜色来自 route.color，可拾取 |
| `trips-layer` | TripsLayer | buildTripData (15 trips) | 每条路线 3 个光流粒子，非匀速（匀速=平原），trailLength=180 |
| `scatter-layer` | ScatterplotLayer | flavors[] (17 节点) | radius=6000m (radiusUnits: 'meters')，描边，可拾取 |

**动画循环**：`requestAnimationFrame` 驱动，`currentTime` 在 0-1800 循环，每帧重建所有 Deck.gl 图层。

## 10. 已知风险点

1. **路线位置耦合**：`query.py` 按 `dispersal_event.event_id` 顺序匹配 `app_data.ROUTES`——重排 ROUTES 而不重 seed 会导致几何/名称/颜色错位
2. **双数据源**：FastAPI（17 flavors, app_data.py）与 Express（8 flavors, api/data.js）数据不同——生产 Express 模式下风味节点更少
3. **生产瓦片缺失**：Express 不代理 `/tiles/*`——生产部署需额外 FastAPI 实例或反向代理
4. **种子依赖**：`app_data.py` 顺序变化会破坏 DB 种子一致性，需先 DROP 再迁移
5. **凭证安全**：`.env` 含真实密码——确认 `.gitignore` 生效且未提交至 git 历史
