# 总体设计

本章依据软件工程设计的原则，从宏观角度对"寻味地理"风味博物志 WebGIS 系统进行介绍，包括系统设计依据和原则、总体技术路线、系统建设架构以及系统整体解决方案等内容。

## 设计依据和原则

### 系统设计依据

#### 地理信息与空间数据标准

本系统在空间数据的组织、存储与交换过程中，参照以下国家标准和行业规范：

1. 《地理信息技术基本术语》（GB/T 17694-1999）
2. 《基础地理信息要素分类与代码》（GB/T 13923-2006）
3. 《信息分类和编码的基本原则与方法》（GB/T 7027-2002）
4. GeoJSON Format Specification (RFC 7946)
5. Mapbox Vector Tile Specification (v2.1)
6. Well-Known Text (WKT) / Well-Known Binary (WKB) — OGC Simple Feature Access

所有空间数据统一采用 WGS84 大地坐标系（EPSG:4326），以地理坐标（经度/纬度）存储，Web Mercator（EPSG:3857）投影由前端 MapLibre GL JS 在渲染时自动完成。

#### 软件工程标准

系统开发过程遵循以下软件工程规范：

1. 《计算机软件文档编制规范》（GB-T 8567-2006）
2. 《计算机软件需求规格说明规范》（GB-T 9385-2008）
3. 《计算机软件开发规范》（GB 8566-88）
4. 《计算机软件质量保证计划规范》（GB/T 12504-1990）

在系统实施过程中，按照面向对象的软件工程方法进行需求分析、系统设计、编码实现与测试，各阶段产出对应的设计开发文档。

#### 数据来源规范

1. **Natural Earth 数据**：遵循 Natural Earth 公共领域使用条款，标注数据来源；
2. **WWF TEOW 数据**：遵循 WWF 学术数据使用规范，引用 Olson et al. (2001) 原始文献；
3. **Wikimedia Commons 图片**：遵循各图片的 CC 许可协议（CC BY-SA / CC BY / 公共领域）。

### 系统设计原则

"寻味地理"风味博物志 WebGIS 系统的建设涉及 WebGIS 可视化、空间数据库、前后端分离架构、LLM 数据管线等多种技术，为实现系统的设计目标，在开发过程中遵循以下设计准则：

1. **视觉优先，内容驱动**

区别于传统 GIS 应用的工具栏密集风格，本系统以博物志视觉语言为核心设计准则：大地为画布、数据为内容、UI 为透明容器。所有交互控件以最小存在感融入界面，让用户注意力始终停留在地球与风味数据之上。

2. **分层解耦，渐进增强**

前端层（Vue 3 + MapLibre + Deck.gl）、API 层（FastAPI）、数据层（PostgreSQL/PostGIS）三者严格分层解耦。前端通过 RESTful API 获取业务数据，通过栅格/矢量瓦片获取地图数据，不直接访问数据库。各层可独立开发、测试、部署。

3. **标准先行，数据规范**

所有空间数据统一采用 WGS84（EPSG:4326）坐标系，关系数据遵循第三范式（3NF），JSONB 字段采用固定 schema。数据入库前进行拓扑校验（ST_MakeValid）与值域校验（风味向量 0~1 区间），确保数据质量。

4. **性能优先，缓存分层**

针对 WebGIS 应用的性能瓶颈，建立多级缓存策略：栅格瓦片 LRU 内存缓存（512 条目）、矢量数据内存常驻（海岸线/河流/陆地 在启动时加载）、数据库连接池（1~4 连接），确保常规操作响应时间在 500 ms 以内。

5. **可扩展，可持续**

数据库 schema 预留扩展字段（JSONB），支持未来新增风味维度或食材属性。LLM 数据管线设计为独立模块，与核心系统通过标准化的 JSON schema 耦合，可在不修改核心代码的情况下迭代管线逻辑。前端组件模块化，新增页面仅需新增路由和视图组件。

## 总体技术路线

针对"寻味地理"风味博物志 WebGIS 系统的建设需求，确定以下技术路线：

1. **前端渲染驱动**：采用 MapLibre GL JS 作为底图渲染引擎，Deck.gl 作为 WebGL 数据图层框架，将 GIS 渲染压力分布至客户端浏览器，后端仅提供数据服务和瓦片切图。此方案避免了服务端 GIS 服务器（如 GeoServer）的部署复杂性和性能瓶颈。

2. **纯 B/S 架构**：系统面向公众用户，采用纯 B/S 架构，用户仅需浏览器即可访问全部功能。无 C/S 客户端、无移动端独立 App（后续可考虑响应式适配移动浏览器）。

3. **PostgreSQL/PostGIS 为核心数据引擎**：选用 PostgreSQL 16 + PostGIS 3 作为数据库平台，利用其原生空间索引（GiST）、全文检索（tsvector）、JSONB 半结构化存储、矢量瓦片生成（ST_AsMVT）等能力，避免引入额外的 GIS 中间件。

4. **COG + rio-tiler 栅格瓦片方案**：栅格底图采用 Cloud Optimized GeoTIFF 格式存储，通过 rio-tiler 的 Reader API 按需读取渲染，配合线程池（6 workers）和 LRU 缓存（512 条目）保证瓦片响应速度。此方案较 GeoServer + GeoWebCache 方案更轻量，无需预切片。

5. **单版本数据库**：本系统数据更新频率低（批次导入为主），不区分现势库/历史库/工作库，采用单库单 Schema 设计，通过 pg_dump 定期全量备份。

6. **LLM 辅助数据生产**：风味向量提取与传播事件录入两条数据管线采用 LLM 提示工程实现，产出的标准化 JSON 经人工审核后通过 seed.py 批量导入数据库。管线与核心系统解耦，独立迭代。

7. **开发工具链**：
   - 前端：Vite（构建） + npm（包管理）+ ESLint（代码检查）
   - 后端：uv（Python 包管理）+ Uvicorn（ASGI 服务器）
   - 数据库：psql + pg_dump + seed.py
   - 版本控制：Git + GitHub
   - 编辑工具：Visual Studio Code

## 系统建设架构

### 体系规范建设

#### 数据规范体系

系统建立以下数据规范：

1. **空间数据入库规范**：所有空间数据统一 EPSG:4326 坐标系，Shapefile 通过 fiona + shapely 读取并做 ST_Simplify（容差 0.08°）及 ST_MakeValid 后入库；
2. **风味向量标注规范**：六维风味向量（麻/辣/咸/酸/甜/鲜）各维 0~1，标注时以该食物属内的相对强度为基准（如"川渝红汤火锅"的辣度为火锅属内最高，标记为 0.95），LLM 管线产出需附带置信度评分；
3. **传播事件录入规范**：起点/终点需映射至 TEOW 生态区编码，传播媒介限定枚举值（海运/内河/陆路/人为引种），文献摘录需标注来源章节；
4. **API 数据格式规范**：所有 API 响应统一 JSON 格式，空间几何以 GeoJSON 编码，时间字段以 ISO 8601 格式。

#### 质量保证体系

1. **代码质量**：前端 ESLint 检查，后端类型检查（Python type hints），Git 提交前人工 code review；
2. **数据质量**：入库前 ST_MakeValid 几何校验、风味向量 0~1 区间校验、外键完整性校验；
3. **服务可用性**：`/health` 端点持续监控栅格就绪状态和矢量图层加载状态，`/tiles/status` 端点上报缓存命中率。

### 标准建设

根据系统建设需要，建立以下标准：

1. 风味基因型数据建库标准：定义 `flavor_genotype` 表结构、JSONB 字段 schema、评分标度；
2. 传播事件数据建库标准：定义 `dispersal_event` 表结构、传播媒介枚举、路径几何精度要求；
3. 食材字典分类标准：定义食材类型枚举（香辛料/油脂/谷物/蛋白/蔬果/发酵/其他）；
4. API 接口标准：定义 RESTful 路由命名规范、响应结构、错误码约定；
5. 前端组件命名标准：Vue 组件采用 PascalCase 命名，页面组件置于 `views/`，复用组件置于 `components/`。

### 数据库建设

根据系统的数据管理和应用需求，需要建设的数据库包括：

1. **基础地理矢量数据库**：Natural Earth 10m 物理矢量（海岸线、河流、陆地）——以内存 GeoJSON 形式在服务启动时加载，不入库；
2. **生态区空间数据库**：WWF TEOW 全球 827 个生态区，含 MultiPolygon 边界几何、生物群系、气候属性（JSONB），存储于 `eco_geo_unit` 表；
3. **风味业务数据库**：风味基因型（`flavor_genotype`）、传播事件（`dispersal_event`）、食材字典（`ingredient`）及相关联表（`recipe_link`、`ingredient_origin`），存储核心业务数据；
4. **叙事内容数据库**：时空叙事章节（`chapter` 表），存储章节文本与引用信息；
5. **系统元数据库**：数据来源说明（`data_source` 表），记录各数据源的名称、描述与链接；
6. **栅格数据存储**：Natural Earth HYP COG GeoTIFF 文件，以文件形式存储于服务器硬盘，通过 rio-tiler 按需读取。

## 系统解决方案

### 整体架构

系统采用三层 B/S 架构，从下至上分为数据层、服务层和展示层。

**数据层**：由 PostgreSQL 16 + PostGIS 3 数据库和 COG 栅格文件组成。关系型业务数据（风味节点、传播路径、食材、叙事章节等）存储于 PostgreSQL 中，空间几何列利用 PostGIS 的 GiST 索引加速空间查询。栅格底图以单文件 COG GeoTIFF 格式存储于硬盘，通过 HTTP Range 请求实现按需读取。

**服务层**：由 FastAPI（Python）提供 RESTful API 服务和瓦片服务。业务 API（`/api/*`）通过 psycopg2 连接池查询 PostgreSQL，返回 JSON 数据；栅格瓦片 API（`/tiles/raster/*`）通过 rio-tiler 从 COG 文件实时渲染 PNG 瓦片，并经由 LRU 缓存加速重复请求；矢量瓦片 API（`/tiles/mvt/*`）通过 PostGIS ST_AsMVT 函数直接从数据库生成 MVT 二进制瓦片；矢量图层 API（`/tiles/vector/*`）从内存中返回启动时预加载的 GeoJSON 数据。

**展示层**：由 Vue 3 单页应用组成，运行于用户浏览器中。MapLibre GL JS 负责底图渲染（HYP 栅格 + 矢量图层），Deck.gl 通过 MapboxOverlay 叠加 WebGL 数据图层（PathLayer 传播路线、TripsLayer 流光粒子、ScatterplotLayer 节点光晕），ECharts 负责风味雷达图和气候图表渲染。Pinia Store 管理交互状态（节点/路线/生态区互斥选择），Vue Router 管理五个功能页面的路由切换。

```
┌─────────────────────────────────────────────────────┐
│  展示层 (Browser)                                     │
│  Vue 3 SPA                                           │
│  ├── MapLibre GL JS  (底图渲染)                       │
│  ├── Deck.gl         (WebGL 数据图层)                 │
│  ├── ECharts         (图表)                           │
│  ├── Pinia           (状态管理)                       │
│  └── Vue Router      (路由)                           │
├─────────────────────────────────────────────────────┤
│  服务层 (FastAPI :8001)                               │
│  ├── /api/*     → routers/api.py   (业务数据 CRUD)    │
│  ├── /tiles/*   → routers/tiles.py (瓦片服务)         │
│  ├── tile_cache.py  (LRU 512 条目)                    │
│  └── startup.py     (启动时加载矢量 + 初始化 DB 池)     │
├─────────────────────────────────────────────────────┤
│  数据层                                               │
│  ├── PostgreSQL 16 + PostGIS 3  (业务数据 + 空间数据)  │
│  └── COG GeoTIFF 文件            (栅格底图)           │
└─────────────────────────────────────────────────────┘
```

### 前端渲染方案

#### 地图渲染引擎选型

本系统选择 MapLibre GL JS（而非 Leaflet、OpenLayers 或 Cesium）作为底图渲染引擎，基于以下考量：

1. **WebGL 原生渲染**：MapLibre 基于 WebGL，支持球面投影（globe view）和 3D 地形，与 Deck.gl 的 WebGL 渲染管线无缝集成；
2. **矢量瓦片原生支持**：直接消费 MVT 格式矢量瓦片，无需中间格式转换；
3. **开源社区活跃**：MapLibre 是 Mapbox GL JS v1 的开源继承者，社区维护活跃，无商业授权限制；
4. **轻量**：核心库体积约 200 KB（gzipped），远小于 Cesium（~1 MB），适合面向公众的 Web 应用。

#### WebGL 图层框架选型

本系统选择 Deck.gl（而非 MapLibre 原生图层或 Three.js）作为 WebGL 数据图层框架，基于以下考量：

1. **图层类型丰富**：PathLayer（路径线）、TripsLayer（粒子流光）、ScatterplotLayer（散点）等开箱即用的图层类型，与需求高度匹配；
2. **MapboxOverlay 无缝集成**：通过 `@deck.gl/mapbox` 的 MapboxOverlay 接口直接嵌入 MapLibre 渲染管线，支持 interleaved 模式（Deck.gl 图层与 MapLibre 图层交替渲染），保证正确的遮挡关系；
3. **大数据量高性能**：Deck.gl 的实例化渲染管线在处理 827 个多边形+17 个节点时性能充裕，为 LLM 管线大规模导入数据后预留性能空间。

### 后端技术选型

#### Web 框架选型

本系统选择 FastAPI（Python）而非 Django、Flask 或 ASP.NET Core，基于以下考量：

1. **原生异步支持**：基于 Starlette 和 asyncio，适合 I/O 密集型场景（数据库查询、文件读取、瓦片渲染）；
2. **自动 API 文档**：基于 OpenAPI 规范自动生成 `/docs`（Swagger UI）和 `/redoc` 交互式文档，降低前后端联调成本；
3. **类型安全**：Pydantic 模型提供请求/响应数据的自动校验和序列化；
4. **GIS 生态兼容**：Python 拥有 rasterio、fiona、shapely、rio-tiler 等成熟的 GIS 库，在数据处理和瓦片渲染方面具有显著优势。

#### 数据库选型

本系统选择 PostgreSQL 16 + PostGIS 3 而非 MySQL 或 MongoDB，基于以下考量：

1. **PostGIS 空间扩展**：提供 GiST 空间索引、ST_AsMVT 矢量瓦片生成、ST_Simplify 几何简化、ST_AsGeoJSON 格式转换等丰富的空间 SQL 函数，无需额外的 GIS 服务器（如 GeoServer）；
2. **JSONB 半结构化支持**：风味向量、气候属性等非固定 schema 数据以 JSONB 存储，兼具 JSON 的灵活性和关系型数据库的查询能力（GIN 索引加速 JSONB 路径查询）；
3. **全文检索**：tsvector/tsquery 原生支持中文分词（需安装 zhparser 扩展或使用 simple 分词 + ILIKE 回退），满足全局搜索需求；
4. **成熟稳定**：PostgreSQL 是全球最先进的开源关系型数据库之一，在 GIS 行业被广泛使用。

### 部署方案

#### 开发环境

开发环境由三个进程组成：

| 进程 | 端口 | 用途 |
|------|------|------|
| FastAPI + Uvicorn | 8001 | API 服务 + 瓦片服务 |
| Express 开发代理 | 3001 | 转发 API 请求至 FastAPI |
| Vite 开发服务器 | 5173 | 前端热更新 + 代理转发 |

浏览器访问 Vite（:5173），Vite 将 `/api/*` 和 `/tiles/*` 请求代理转发至 FastAPI（:8001）。Express 开发代理（:3001）在 `npm run dev` 中自动启动，提供与生产环境一致的中间件行为。

#### 生产环境

生产环境由两个进程组成：

| 进程 | 端口 | 用途 |
|------|------|------|
| FastAPI + Uvicorn | 8001 | API 服务 + 瓦片服务 |
| Express (生产模式) | 3001 | 静态资源托管（`dist/`） |

生产环境下，建议在 Express 前增加 Nginx 反向代理：

- 静态资源（`/assets/*`）由 Express 直接托管；
- API 请求（`/api/*`）和瓦片请求（`/tiles/*`）由 Nginx 代理至 FastAPI（:8001）；
- Nginx 可配置 gzip 压缩、缓存策略、SSL 终止及速率限制。

```
Internet → Nginx (:80/:443)
                ├── /assets/*   → Express (:3001) → dist/
                ├── /api/*      → FastAPI (:8001)
                └── /tiles/*    → FastAPI (:8001)
```

#### 数据库部署

PostgreSQL 16 + PostGIS 3 部署于独立服务器或云数据库实例。数据库通过环境变量 `DATABASE_URL` 配置连接信息。建议配置：

- `shared_buffers`：2 GB（根据服务器内存调整）
- `effective_cache_size`：6 GB
- `work_mem`：64 MB
- 开启 `auto_explain` 用于慢查询分析
