# 数据库设计

数据库是"寻味地理"风味博物志 WebGIS 系统的核心基础。建立一个良好的数据组织结构和数据库，使整个系统能够迅速、准确、高效地调用和管理所需数据，是系统开发的必然要求。本章主要介绍数据库的数据内容、设计原则、数据库体系构成，并分别对开源生态区空间数据与系统业务数据，从概念模型、逻辑模型和物理模型等方面进行详细设计。

## 数据库内容

### 数据概况

本系统涉及的空间数据和业务数据现状如下：

| 序号 | 数据名称 | 格式 | 来源 | 备注 |
| --- | --- | --- | --- | --- |
| 1 | Natural Earth HYP 栅格底图 | COG GeoTIFF | Natural Earth | 全球高程渲染，~700 MB |
| 2 | Natural Earth 10m 物理矢量 | Shapefile → 内存 GeoJSON | Natural Earth | 海岸线、河流、陆地 |
| 3 | WWF TEOW 生态区划 | Shapefile → PostGIS | WWF | 827 个生态区 MultiPolygon |
| 4 | 风味基因型数据 | PostgreSQL 关系表 | 手动标注 + LLM 管线 | 17 个种子节点 |
| 5 | 传播事件数据 | PostgreSQL 关系表 | 手动数字化 + LLM 管线 | 5 条种子路径 |
| 6 | 食材字典数据 | PostgreSQL 关系表 | 菜谱文献整理 | ~60 种食材 |
| 7 | 叙事章节数据 | PostgreSQL 关系表 | 人工编纂 | 5 章 |
| 8 | 数据来源元数据 | PostgreSQL 关系表 | 人工整理 | 3 条 |

### 数据入库前处理

1. **TEOW Shapefile**：通过 fiona 读取，使用 shapely 做 ST_Simplify（容差 0.08°）降低顶点密度，ST_MakeValid 修正自相交拓扑错误，补充中文名称映射后批量写入 `eco_geo_unit` 表；
2. **Natural Earth 矢量**：通过 fiona 读取，shapely 简化后存入服务端内存（向量字典），不入数据库；
3. **风味/传播/食材/叙事数据**：从 Python 硬编码数据（`app_data.py`）经规范化后通过 `seed.py` 批量 INSERT，或通过 `import_json.py` 从 JSON 文件导入。

## 数据库设计原则

1. **空间与属性一体化**：利用 PostgreSQL + PostGIS 在同一数据库内同时管理空间数据（MultiPolygon / Point / LineString）和关系数据（外键关联、JSONB 文档），避免数据割裂。

2. **最小冗余**：通过外键约束和关联表实现多对多关系（如 ingredient_origin、recipe_link、genotype_lineage），遵循第三范式（3NF）。JSONB 字段仅用于半结构化数据（风味向量、气候属性），不用于可拆分为独立关系的数据。

3. **标准化命名**：表名采用小写蛇形命名（snake_case），主键统一为 `<表名简写>_id`，外键名与引用表主键名一致。空间几何列固定命名为 `boundary`（面）、`coordinates`（点）、`route_geom`（线）。

4. **索引驱动查询**：所有外键列建立 B-Tree 索引，所有空间几何列建立 GiST 索引，JSONB 字段建立 GIN 索引以支持路径查询，文本搜索列建立 tsvector 列和对应 GIN 索引。

5. **数据完整性**：通过 CHECK 约束保证风味向量各维度在 0~1 区间，通过外键约束保证引用完整性，通过 UNIQUE 约束保证业务主键唯一性。

## 数据库总体设计

### 数据库体系构成

系统数据库由一张 PostgreSQL 数据库承载（数据库名：`flavor`），通过 public Schema 组织，包含以下五组数据表：

```
PostgreSQL 16 + PostGIS 3 (Database: flavor, Schema: public)
│
├── 生态区空间数据组
│   └── eco_geo_unit              827 个 TEOW 生态区
│
├── 风味业务数据组
│   ├── flavor_genotype            风味基因型（地域食物）
│   ├── ingredient                 食材字典
│   ├── ingredient_origin          食材原产地（M:N）
│   └── recipe_link                配方连结（M:N）
│
├── 传播叙事数据组
│   ├── dispersal_event            传播事件（路径）
│   ├── chapter                    叙事章节
│   └── genotype_lineage           基因溯源链（自引用 M:N）
│
├── 系统元数据组
│   └── data_source                数据来源说明
│
└── 自定义枚举类型
    ├── ingredient_type            食材类型枚举
    ├── route_type                 路径类型枚举
    └── vehicle_type               传播媒介枚举
```

### 数学基础设计

所有空间数据统一采用以下数学基础：

| 属性 | 取值 |
| --- | --- |
| 坐标系 | WGS84 大地坐标系（EPSG:4326） |
| 坐标单位 | 十进制度（decimal degrees） |
| 几何类型 | Point / LineString / MultiPolygon |
| 空间参考标识符（SRID） | 4326 |
| 投影 | 前端 MapLibre GL JS 渲染时自动转 Web Mercator (EPSG:3857) |

### 数据库对象命名规则

| 对象类型 | 命名规则 | 示例 |
| --- | --- | --- |
| 数据库 | 小写英文 | `flavor` |
| Schema | `public` | — |
| 表 | 小写蛇形，复数或单数一致 | `eco_geo_unit`, `flavor_genotype` |
| 主键列 | `<表简写>_id` | `genotype_id`, `ingredient_id` |
| 外键列 | 与引用表主键同名 | `ingredient_id` → `ingredient.ingredient_id` |
| 空间几何列 | 固定名称 | `boundary`, `coordinates`, `route_geom` |
| 索引 | `idx_<表名>_<列名>` | `idx_flavor_genotype_coordinates` |
| 枚举类型 | 小写蛇形，`_type` 后缀 | `ingredient_type` |

### 数据库备份和恢复

本系统采用单版本数据库设计，不区分现势库/历史库/工作库。备份策略如下：

1. **全量备份**：通过 `pg_dump` 定期全量备份数据库，备份周期为每日一次（数据频繁变更期）或每周一次（稳定运行期）；
2. **逻辑备份**：`pg_dump -Fc flavor > flavor_YYYYMMDD.dump`，以 PostgreSQL 自定义格式存储，支持选择性恢复；
3. **恢复流程**：`pg_restore -d flavor flavor_YYYYMMDD.dump`，恢复后需重建物化视图（如有）并执行 `ANALYZE` 更新统计信息；
4. **备份保留**：保留最近 7 天的每日备份和最近 12 个月的每月备份。

## 生态区空间数据库设计

### 数据内容

WWF TEOW 全球陆地生态区划数据是系统的空间分析框架，包含 827 个生态区。每个生态区记录以下内容：

- 生态区英文名（`eco_name`）及中文译名（`eco_name_cn`）
- 生态区编码（`eco_code`）
- 所属生物地理界（`realm`：8 界）和生物群系（`biome`：14 类，含中文名 `biome_cn`）
- 面积（`area_km2`，单位 km²）
- 边界几何（`boundary`：MultiPolygon, EPSG:4326）
- 气候属性（`climate`：JSONB，含年均温、年降水量、年均湿度等）

### 概念模型设计

生态区实体（EcoGeoUnit）是系统的核心空间容器，与风味基因型为一对多关系（一个生态区内可诞生多个风味基因型），与传播事件为多对多关系（一个生态区可作为多条传播路线的起点或终点）。

### 逻辑模型设计

**表名**：`eco_geo_unit`

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `eco_name` | TEXT | PK | 生态区英文名（WWF 官方命名） |
| `eco_name_cn` | TEXT | — | 生态区中文译名 |
| `eco_code` | TEXT | — | WWF 生态区编码 |
| `realm` | TEXT | — | 生物地理界（Afrotropic / Nearctic 等） |
| `biome` | TEXT | — | 生物群系英文名 |
| `biome_cn` | TEXT | — | 生物群系中文名 |
| `area_km2` | NUMERIC | — | 面积（km²） |
| `climate` | JSONB | — | 气候属性（年均温、年降水量、湿度等） |
| `boundary` | GEOMETRY(MultiPolygon, 4326) | — | 生态区边界 |

**索引**：
- `idx_eco_geo_unit_boundary`：GiST 空间索引
- `idx_eco_geo_unit_climate`：GIN 索引（JSONB 路径查询）
- `idx_eco_geo_unit_biome`：B-Tree 索引（按生物群系筛选）

### 物理模型设计

- **存储引擎**：PostgreSQL heap storage
- **几何简化**：入库时对 MultiPolygon 进行 ST_Simplify（容差 0.08°），减少存储空间和渲染传输量
- **预计存储**：827 条记录，含 MultiPolygon 几何，约 50~100 MB
- **MVT 瓦片查询**：`ST_AsMVTGeom(boundary, ST_TileEnvelope(z, x, y), 4096, 0, true)` 配合 `ST_Simplify(boundary, 0.02)` 实时生成

## 系统业务数据库设计

### 数据内容

系统业务数据库包含四组核心实体及其关联表：

1. **风味基因型（FlavorGenotype）**：为核心业务实体，记录在特定生态区下诞生的具体地方食物；
2. **食材（Ingredient）**：记录构成风味的物质基础；
3. **传播事件（DispersalEvent）**：记录食材/风味在时空中的传播路径；
4. **叙事章节（Chapter）**：记录与传播事件对应的历史叙事。

### 概念模型设计

核心实体关系如下：

```
EcoGeoUnit (生态区)
     │ 1:N
     ▼
FlavorGenotype (风味基因型)
     │ M:N ──────────── Ingredient (食材)
     │ (recipe_link)        │ M:N ─── EcoGeoUnit
     │                      │ (ingredient_origin)
     │ M:N (自引用)
     ▼
GenotypeLineage (基因溯源链)
     
Ingredient (食材)
     │ 1:N
     ▼
DispersalEvent (传播事件)
     │ N:1 ─── EcoGeoUnit (起点 from_eco_id)
     │ N:1 ─── EcoGeoUnit (终点 to_eco_id)
     
DispersalEvent
     │ 1:N
     ▼
Chapter (叙事章节)
```

### 物理模型设计

**存储参数**：
- 数据库字符集：UTF-8
- 表空间：默认（pg_default）
- 预计总存储量：种子数据 < 100 MB，LLM 规模化后 < 500 MB

**空间索引**：

| 表 | 索引列 | 索引类型 |
| --- | --- | --- |
| `eco_geo_unit` | `boundary` | GiST |
| `flavor_genotype` | `coordinates` | GiST |
| `dispersal_event` | `route_geom` | GiST |

**GIN 索引**：

| 表 | 索引列 | 用途 |
| --- | --- | --- |
| `eco_geo_unit` | `climate` | JSONB 路径查询 |
| `flavor_genotype` | `genome_vector` | JSONB 路径查询 |
| `flavor_genotype` | `search_vector` | 全文检索 |
| `dispersal_event` | `search_vector` | 全文检索 |
| `ingredient` | `flavor_tags` | 数组查询 |

**全文检索实现**：

```sql
-- 风味基因型搜索向量
ALTER TABLE flavor_genotype 
  ADD COLUMN search_vector tsvector 
  GENERATED ALWAYS AS (
    to_tsvector('simple', 
      coalesce(dish_name,'') || ' ' || 
      coalesce(city_name,'') || ' ' || 
      coalesce(region_label,'') || ' ' || 
      coalesce(eco_label,'')
    )
  ) STORED;

-- 传播事件搜索向量
ALTER TABLE dispersal_event 
  ADD COLUMN search_vector tsvector 
  GENERATED ALWAYS AS (
    to_tsvector('simple', coalesce(route_name,''))
  ) STORED;
```

> 注：当前使用 `simple` 分词配置。中文分词可通过安装 `zhparser` 扩展或使用 `jieba` 分词后生成 tsvector 改进。

| 最大连接数 | 4 | ThreadedConnectionPool maxconn |

## 数据库建库与维护更新

### 建库流程

1. 确保 PostgreSQL 16 已安装，PostGIS 3 扩展已可用；
2. 创建数据库：`CREATE DATABASE flavor;`
3. 启用扩展：`CREATE EXTENSION postgis;`
4. 执行 schema.sql 创建所有表、索引和约束；
5. 执行 seed.py：导入 WWF TEOW Shapefile（827 条生态区记录）+ 导入 app_data.py 中的种子数据；
6. 执行 ANALYZE 更新统计信息。

### 数据维护更新

当前系统数据更新为手动批次导入模式：

1. **风味数据更新**：LLM 管线产出 JSON → 人工审核 → `import_json.py` 导入 → `REINDEX` 重建索引 → `ANALYZE`；
2. **传播事件更新**：同上流程；
3. **生态区更新**：TEOW 数据集更新周期较长（年级别），如需更新，使用 `seed.py` 重新导入即可；
4. **应急恢复**：通过 `pg_restore` 从最近备份恢复。

未来可考虑实现简单的 Web 管理后台用于数据 CRUD，降低数据维护对技术人员的依赖。
