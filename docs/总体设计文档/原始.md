## 1. 项目愿景与核心隐喻

### 1.1 我们要解决什么问题？
传统美食地图只告诉你“这里有什么”，而我们追求**“味道为什么如此”——揭示地理环境如何塑造风味演化**。

### 1.2 核心隐喻：风味是基因，大地是编辑者
我们将食物风味视为一种**可遗传、可变异的生物基因**，将地理环境视为**自然选择的压力**。

- **同一种食物**（如“火锅”）视为**一个属**
- **不同地域的版本**（如川渝红汤锅、北京涮羊肉）视为**同一属下的不同基因型**
- **风味向量**（麻、辣、鲜、咸、甜、脂香、酸）是基因型的数字化编码

### 1.3 同源异构
同一本源的味道，在不同的水土中发生**基因突变**，最终形成迥异的风味形态。我们要在地球上**可视化这场进化**。

## 2. 数据架构：实体与关系

### 2.1 四核心实体

#### 生态地理单元（EcoGeoUnit）
- **来源**：WWF TEOW 全球陆地生态区划（唯一地理分区数据，无国界、无行政边界）
- **含义**：具有均质气候与物产特征的自然空间，是演化发生的“容器”
- **关键属性**：
  - eco_id (PK)：WWF TEOW 唯一编码
  - name：生态区名称
  - climate：温湿度、海拔等气候数据（JSONB）
  - adjacent_ecos：相邻生态单元 ID 列表
  - dominant_ingredients：本地优势原材料列表（关联 Ingredient）

#### 原材料（Ingredient）
- **含义**：构成风味的物质基础（花椒、牛油、椰青…）
- **关键属性**：
  - ingredient_id (PK)
  - name：标准名称
  - type：类别（香辛料 / 油脂 / 谷物 / 海鲜 / 禽畜 / 蔬菜 / 发酵品）
  - origin_ecos：原产地（通过关联表 IngredientOrigin 建立与 EcoGeoUnit 的 M:N 关系）
  - flavor_tags：风味标签数组
  - growth_requirements：适生条件（JSONB，可选）

#### 成品风味基因型（FlavorGenotype）
- **含义**：在某个特定生态单元下诞生的具体地方食物，携带完整风味向量
- **关键属性**：
  - genotype_id (PK)
  - dish_name：成品名（如“川渝红汤火锅”）
  - category：食物属（如“火锅”、“烤鸭”）
  - eco_id (FK → EcoGeoUnit)：所在生态单元
  - genome_vector：七维风味向量（JSONB）
  - coordinates：经纬度坐标（POINT）
  - formation_narrative：双层因果推演文案（JSONB）

> *注：同一种食物（如烤鸭）若存在于多个生态单元，每个版本视为独立基因型，故“所在”关系为 1：N。*

#### 传播事件（DispersalEvent）
- **含义**：某原材料在时空中的一次位移（如辣椒从美洲传入四川）
- **关键属性**：
  - event_id (PK)
  - ingredient_id (FK → Ingredient)：传播的原材料
  - from_eco_id (FK → EcoGeoUnit)：起点生态单元
  - to_eco_id (FK → EcoGeoUnit)：终点生态单元
  - route_geom：路径几何（GEOMETRY）
  - time_label：时期标签
  - time_order：时序排序值
  - vehicle：传播媒介（海运 / 内河 / 陆路 / 人为引种）
  - literature_excerpt：古籍或研究文献摘录

---

### 2.2 三关联实体（解决多对多关系）

#### 原材料原产地（IngredientOrigin）
- **用于**：连接原材料与生态地理单元，实现“原产于”的 M:N 关系
- **关键属性**：
  - ingredient_id (PFK → Ingredient)
  - eco_id (PFK → EcoGeoUnit)

#### 配方连结（RecipeLink）
- **用于**：连接成品基因型与原材料（M:N），并携带额外关系属性
- **关键属性**：
  - genotype_id (PFK → FlavorGenotype)
  - ingredient_id (PFK → Ingredient)
  - role：角色（锅底 / 蘸料 / 主涮 / 配菜）
  - importance：该材料对风味向量的贡献度（0~1）
  - is_native：是否为该生态单元的本地物产

#### 基因溯源链（GenotypeLineage）
- **用于**：记录成品之间的演化路径（M:N 自引用）
- **关键属性**：
  - ancestor_id (PFK → FlavorGenotype)：父基因型
  - descendant_id (PFK → FlavorGenotype)：子基因型
  - mutation_desc：基因变异描述
  - trigger_eco_ids：触发变异的沿途生态单元序列 (FK → EcoGeoUnit)

---

### 2.3 关系总结

| 关系 | 左实体 | 基数 | 右实体 | 实现方式 |
|:---|:---|:---|:---|:---|
| **所在** | 生态单元 | 1 : N | 成品基因型 | FK in FlavorGenotype |
| **原产于** | 原材料 | M : N | 生态单元 | 关联表 IngredientOrigin |
| **构成** | 成品 | M : N | 原材料 | 关联表 RecipeLink |
| **传播** | 传播事件 | N : 1 | 原材料 | FK in DispersalEvent |
| **起点/终点** | 传播事件 | N : 1 | 生态单元 | FK in DispersalEvent |
| **溯源** | 成品(祖先) | M : N | 成品(后代) | 关联表 GenotypeLineage |
| **触发生态** | 溯源链 | N : M | 生态单元 | trigger_eco_ids in GenotypeLineage |

> *注：同一种食物（如烤鸭）若出现在多个生态单元，每个版本视为独立基因型，故“所在”关系为 1：N。*

## 3. 视觉与交互设计核心原则

### 3.1 视觉基调：极致纯净博物志风
- **底色**：明亮、通透的米白与浅灰，彻底剔除冗杂 UI 控件，让地球本身成为视觉主干
- **UI 材质**：高透明度毛玻璃（80% 磨砂透白），边缘无感渐变融入背景，实现“悬浮且不可见边框”
- **数据高亮色**：胭脂红、松石绿、琥珀黄等极高明度自然提取色，在素色底图上形成清晰视觉刺透力
- **字体**：标题使用优雅衬线体，正文使用人文主义无衬线体，字重轻盈、字距疏朗，营造科学图鉴的呼吸感

### 3.2 图层叙事逻辑（四层空间）
- **L0 纯净物理底图**：Natural Earth 浅色地形阴影 + 自然海岸线、主干水系，绝对无国界线
- **L1 自然生态单元层**：WWF TEOW 生态区精确矢量多边形，内部完全透明（Fill Opacity = 0），仅用极细浅褐色实线（1px）勾勒边界。所有环境数据隐藏于 GeoJSON 属性中，静待检索触发
- **L2 脉冲流光传播层**：  
  - 基础轨道：低透明度、低饱和度暗灰色细线（虚线表越洋，实线表陆路/内河）  
  - 流动光效：高亮光斑粒子沿轨道滑动，光斑移动方向即基因传播方向  
  - 时序接力：年代早的事件率先点亮滑动，年代晚的事件随后接力，通过视觉上的先后发车直观表达历史编年史  
  - 非匀速运动：平原地带匀速，跨越 L1 生态边界时短暂减速加亮，暗示环境阻力
- **L3 实体锚点层**：成品节点悬浮于坐标正上方，呈现微弱的呼吸光晕，并在地表投射阴影

### 3.3 无界 HUD 交互架构
- **左侧（约 20% 宽）**：极简搜索输入框（纯白无框、微弱阴影）。默认状态下地图全空，只有山河。用户搜索后，引擎智能计算并显现相关 L1 生态边界、L2 暗线轨道及光效
- **右侧（约 25% 宽）**：动态对象检视器，根据选中对象类型平滑切换内容：
  - 检视 L1 生态多边形 → 展示温湿度双轴曲线图 + 环境筛选推演文案
  - 检视 L3 成品节点 → 展示风味雷达图 + 底层食材积木
  - 检视 L2 流光连线 → 展示古籍文献摘录
- **底部**：完全留空，无时间轴、无过滤器，把最大视野留给地球轮廓和跨洋流光
- **导航**：全部收拢为左下角一枚呼吸光点（琥珀黄），悬停时优雅展开菜单

## 4. 风味形成的双层因果引擎

### 4.1 路径一：直接环境压力（生理筛选）
> 气候 → 人体生理需求 → 风味偏好被筛选 → 编码进基因型

**例**：四川盆地高湿（年均湿度 79%）→ 汗液蒸发受阻 → 需内部“加热排湿” → 花椒碱（激活面部汗腺）与辣椒素（促进循环）被筛选 → 麻辣风味成为进化适应

### 4.2 路径二：间接物产约束（生态供应）
> 气候/土壤 → 当地可产食材 → 锅底/蘸料/配菜锁定 → 风味被限定

**例**：海南岛热带产椰子 → 椰青唾手可得 → 四川的牛油在此无生态优势 → 椰青替代牛油成为锅底 → 辣度由 0.95 骤降至 0.0，甜度由 0.10 直升至 0.90

### 4.3 双因交织——顺德粥底火锅的复合案例
- **物产约束**：珠三角鱼米之乡，香米与河海鲜随手可得 → 米熬成粥底、海鲜涮入粥，是“有什么就用什么”的直接结果
- **生理筛选**：岭南长夏湿热，人体消化系统被抑制 → 粥底温和不刺激肠胃 → **这与川渝的“用麻辣强行激活”形成极端对比——同样应对湿热，顺德选择了安抚，四川选择了刺激**
- **结果**：两条路径共同写就了“鲜甜温润”的对立基因型

## 5. 前端技术栈建议
- **地图渲染引擎**：MapLibre GL JS
- **底图数据**：Natural Earth Cross Blended Hypso（栅格）+ 10m 物理矢量（海岸线、陆地、河流、湖泊）
- **WebGIS 数据驱动层**：Deck.gl（PathLayer 渲染暗线轨道，TripsLayer 渲染光流粒子）
- **图表库**：ECharts（雷达图、温湿度曲线，去除所有装饰，仅保留形状）
- **前端框架**：React 或 Vue 3 + 轻量状态管理（保持上下文连贯）
- **样式管理**：CSS 自定义属性（Design Tokens）统一管理色彩、模糊值、动效时长

## 6. 用户核心体验流

1. **进入首页** → 看到纯净大地与缓慢移动的光点粒子，仅有一句文案“探寻同一种食物，在不同水土中写下的味道基因”和一个“开启地图”按钮
2. **进入地图页** → 全空视角，只见山河，不见数据。左下角一枚呼吸光点暗示交互
3. **搜索“火锅”** → 相关 L1 生态区边界瞬时显现，L2 暗线轨道及光流激活，L3 节点浮现
4. **点击成都节点** → 右侧面板弹出麻辣风味的尖锐雷达图，下方展示推演文案：“该生态区年均湿度 79%，环境筛选了高浓度花椒碱以应对‘外湿内燥’的体征需求”
5. **点击从成都到顺德的暗线** → 光流粒子沿长江水系滑行，跨越南岭生态边界时短暂减速加亮，右侧面板浮现历史文献记载
6. **继续自由拖拽地图** → 用户开始在探索中自发生成因果理解，培养“如何用地理环境解码味道”的思维习惯

## 7. 数据治理与 LLM 辅助(暂时不做)
- **风味向量生成**：爬取开源菜谱/美食文献库 → 通过 LLM 提示工程提取标准化 JSON 风味向量
- **传播事件录入**：基于历史研究与古籍，结构化录入传播事件的起点、终点、路径与时期，附文献摘录
- **生态属性填充**：从 WWF TEOW 及气候数据集中预计算每个生态单元的温湿度、海拔等属性
