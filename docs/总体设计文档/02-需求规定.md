# 需求规定

本章根据前期对饮食文化地理信息系统领域的调研以及"寻味地理"风味博物志项目的原始设计构想，从数据现状、现有数据特征分析、现存同类项目对比以及运行环境等方面出发，具体分析了本系统的数据需求、功能需求及非功能性需求。

## 现状

### 数据现状

#### Natural Earth 物理底图数据

Natural Earth 是一个公共领域的全球地图数据集，由制图师 Tom Patterson 及志愿者社区维护，被全球 GIS 应用广泛采用。本系统使用其中两组核心数据：

1. **Cross Blended Hypsometric Tints（HYP_HR_SR_W_DR）**：综合高程渲染栅格底图，以 Cloud Optimized GeoTIFF（COG）格式提供，分辨率约 10,800×5,400 像素。该数据以柔和的地形阴影和高度着色呈现全球陆地起伏形态，无政治边界、无行政标注，与本项目"以大地为视觉主干"的设计理念高度契合。原始数据以 ZIP 压缩包分发（约 700 MB），解压后为单文件 COG GeoTIFF。

2. **10m Physical Vectors**：全球物理矢量数据，包含海岸线（ne_10m_coastline）、陆地轮廓（ne_10m_land）、河流及湖泊中心线（ne_10m_rivers_lake_centerlines）三个图层。数据采用 WGS84（EPSG:4326）坐标系，以 Shapefile 格式分发。

上述数据均为 Natural Earth v5.1.1 版本，覆盖全球范围，数据质量成熟稳定，被 CartoDB、Mapbox、QGIS 等工具原生支持。

#### WWF 全球陆地生态区划（TEOW）数据

世界自然基金会（WWF）编制的"Terrestrial Ecoregions of the World"是全球陆地生态区划的权威数据集，将全球陆地划分为 827 个具有均质气候、物产和物种特征的生态区。每个生态区包含以下属性：

- 生态区名称及编码（eco_name, eco_code）；
- 所属生物群系（biome，14 类）及生物地理界（realm，8 界）；
- 面积（area_km2）；
- 边界几何（MultiPolygon，WGS84）。

原始数据以 Shapefile 格式分发，包含多个图层版本。本系统选用合并图层 `wwf_terr_ecos_merged.shp`，并在入库时补充中文名称映射（如 biome 名称、生态区中文译名）。注意：TEOW 原始数据不含气候属性字段（如温湿度、海拔），本系统所需的生态区气候特征需从外部气候数据获取后补充。

#### 风味基因型与传播事件数据

本系统的核心业务数据——风味基因型（flavor_genotype）与传播事件（dispersal_event）——当前以手动标注的种子演示数据形式存在：

- **风味基因型**：17 个节点，涵盖中国主要饮食文化区域（成都、重庆、顺德、兰州、北京、杭州、云南、潮汕、海口、广州、贵阳、哈尔滨、楚雄、台州、南京、新疆等），每个节点包含六维风味向量（麻/辣/咸/酸/甜/鲜，0~1 标度）、地理坐标、食材配方、所属食物属等字段；
- **传播事件**：5 条历史传播路径（丝绸之路陆路、海上香料之路、辣椒传播路线、大运河茶叶北行、香料群岛东传），每条路径包含起点生态区、终点生态区、传播媒介、时期标签、路径几何（LineString）等字段；
- **叙事章节**：5 章，对应上述传播事件的历史叙事；
- **食材字典**：约 60 种食材，按类型分为香辛料、油脂、谷物、蛋白、蔬果、发酵品等。

上述数据由领域专家手动标注，数据量受人工产能限制，无法覆盖全国完整的风味地理图景。系统设计目标是通过 LLM 数据管线实现自动化、规模化生产（详见 01-系统概况）。

#### 风味媒体数据

每个风味节点关联一张菜品图片，来源为 Wikimedia Commons，以远程 URL 形式存储。对于未匹配到媒体资源的节点，系统通过后端 SVG 自动生成占位图像。

### 现有数据分析

#### Natural Earth 数据质量评估

Natural Earth 数据为全球公认的公共领域 GIS 基准数据，具备以下特征：

- **完整性**：覆盖全球范围，无缺失区域；
- **精度**：1:10,000,000 比例尺，足以满足全球尺度的交互式地图浏览；
- **一致性**：拓扑关系正确，海岸线与陆地边界吻合，河流入海口衔接良好；
- **适用性**：无国界线的设计与本项目"博物志"视觉定位一致。

#### WWF TEOW 数据质量评估

TEOW 数据为学术领域权威数据集，但存在以下需要处理的问题：

1. **字段语言**：原始属性字段为英文，需建立中英文名称映射表（biome → 生物群系中文名，realm → 生物地理界中文名，生态区名称 → 常用中文译名）；
2. **边界精度**：原始 MultiPolygon 边界在高缩放级别下细节冗余，需在入库时通过 ST_Simplify 进行简化（容差约 0.08°），减少矢量瓦片传输体积；
3. **气候属性缺失**：TEOW 本身不包含气候数据，需从外部数据获取后补充；

#### 风味与传播数据质量评估

当前种子演示数据存在以下局限性：

1. **覆盖范围有限**：仅 17 个风味节点远不足以代表中国丰富的饮食地理多样性，黑龙江、内蒙古、西藏、广西、福建等大量具有独特风味的区域尚未录入；
2. **向量标注主观性**：六维风味向量由人工标注，不同标注者的感官基准可能存在差异，缺少跨标注者的一致性校验；
3. **传播路径不完整**：仅覆盖了最著名的 5 条传播路线，大量区域性的饮食交流路径（如滇藏茶马古道对云南风味的影响、闽粤华侨饮食对东南亚的反哺等）尚未收录；
4. **文献引用缺失**：当前传播事件数据中缺少规范的古籍文献摘录字段（`literature_excerpt`），降低了数据的学术可信度。

上述问题均应由 LLM 数据管线在规模化生产中系统性解决：风味向量提取管线可消除人工主观偏差，传播事件录入管线可从古籍中自动补全文献引用。

### 现存项目分析

饮食文化的地理可视化并非全新的领域。以下分析现有相关项目与本系统的定位差异。

#### 传统美食地图类应用

以 TasteAtlas（tasteatlas.com）为代表的美食地图应用，主要呈现"某地有什么食物"的百科全书式信息，缺乏以下几个维度的深度表达：

1. **无生态因果解释**：仅罗列菜品与地理位置，不解释"为什么这个地方会产生这种味道"；
2. **无传播路径可视化**：不展示食材与烹饪技法在时空中的传播与变异过程；
3. **无定量风味表达**：使用文字描述而非数字化风味向量，无法进行跨地域的定量对比。

"寻味地理"的核心差异在于以**生态地理单元为分析框架**、以**六维风味向量为定量表达**、以**传播路径为动态叙事**，从三个维度超越了传统美食地图。

#### 学术数据库类项目

以 FlavorDB（印度理工学院开发）为代表的食品分子数据库，从化合物层面（香气分子、味觉受体）研究食物风味，但面向的是食品科学研究人员而非普通大众，缺少以下要素：

1. **无空间可视化**：数据以表格和分子结构图呈现，不涉及地理信息；
2. **无叙事层**：不关注风味形成的地理历史叙事。

"寻味地理"定位为面向大众的科普型 WebGIS 应用，以可视化交互而非数据表格作为主要输出形式。

#### 数字人文 GIS 项目

以 CHGIS（中国历史地理信息系统，哈佛大学）为代表的历史地理项目，提供了丰富的历史地名和行政边界时空数据，但与饮食文化的交叉应用较少。"寻味地理"可以被视为数字人文 GIS 在饮食人类学领域的一次应用尝试。

### 运行环境现状

系统目标运行环境为现代 Web 浏览器，用户通过互联网访问。开发环境与生产环境架构如下：

**开发环境**：

- 操作系统：Windows 11（开发机）/ macOS（可选）
- 前端开发服务器：Vite (:5173)，支持热更新
- 后端 API 服务器：FastAPI + Uvicorn (:8001)
- 数据库服务器：PostgreSQL 16 + PostGIS（本地或远程）
- 开发代理：Express (:3001)，转发 API 请求

**生产环境**：

- 应用托管：Express (Node.js) 托管前端静态资源 + FastAPI 后端服务
- 数据库：PostgreSQL 16 + PostGIS
- 反向代理：Nginx（生产环境完整 WebGIS 功能需要代理 `/tiles/*` 至 FastAPI）

**浏览器兼容性要求**：系统依赖 WebGL（MapLibre GL JS + Deck.gl），需支持 WebGL 2.0 的现代浏览器（Chrome 90+, Edge 90+, Firefox 88+, Safari 15+）。

## 需求

### 项目建设总体要求

第一、以"地理驱动风味进化"为核心设计思想贯穿全局。

1. 从数据模型设计上，以生态地理单元（EcoGeoUnit）作为核心空间分析框架，所有风味数据、传播事件均锚定于生态区而非行政边界；
2. 从可视化设计上，以物理底图（无国界线）为视觉基底，生态区边界为唯一显式划分，强化"自然而非人为"的视觉叙事；
3. 从交互设计上，节点-路线-生态区三者联动选择，使用户在操作中自发建立"环境→风味"的因果联想。

第二、以 LLM 数据管线解决规模化内容生产瓶颈。

1. 风味向量提取管线：从非结构化菜谱文本中自动提取标准化六维向量及食材配方，消除人工标注的主观偏差与产能瓶颈；
2. 传播事件录入管线：从古籍文献中自动抽取结构化传播事件，补全文献引用，提升数据学术可信度。

第三、其它总体需求。

1. 前后端分离架构，前端为 SPA（单页应用），后端为 RESTful API + 瓦片服务；
2. 所有空间数据采用统一的 WGS84（EPSG:4326）坐标系；
3. 系统质量和技术要求符合 GB-T 8567-2006 计算机软件文档编制规范的相关要求；
4. 前端界面设计遵循博物志风格视觉基线（纯净底色、毛玻璃材质、极致留白），区别于传统 GIS 应用的工具栏密集风格。

### 数据需求

#### Natural Earth 数据需求

1. HYP_HR_SR_W_DR 栅格数据需转换/确认为 COG 格式（Cloud Optimized GeoTIFF），以支持高效 HTTP 范围请求和按需瓦片读取；
2. 10m 物理矢量需完成 Shapefile 加载与内存常驻（数据量小，约数 MB），在启动时完成几何简化以降低前端渲染负载；
3. 矢量数据需做拓扑验证：海岸线与陆地面应无缝隙，河流入海口应与海岸线相连。

#### WWF TEOW 生态区数据需求

1. Shapefile 导入 PostgreSQL/PostGIS，使用 MultiPolygon 几何类型存储；
2. 建立中英文名称映射表：biome 英文名 → 中文名，生态区英文名 → 常用中文译名；
3. 对边界几何进行简化（ST_Simplify，容差 0.08°），降低矢量瓦片的数据传输量；

#### 风味基因型数据需求

1. 每条风味基因型记录（对应一个特定生态区下的地方食物）需包含以下核心字段：
   - 节点唯一标识（node_key）；
   - 城市名（city_name）、菜品名（dish_name）、食物属（dish_family）；
   - 六维风味向量（genome_vector JSONB）：麻、辣、咸、酸、甜、鲜，各维 0~1；
   - 地理坐标（coordinates POINT, EPSG:4326）；
   - 所属生态区引用（eco_name FK）；
   - 风味主标签（primary_labels）及对应数值（primary_values）；
   - 可视化配色（color_hex）。
2. 每条记录需通过 recipe_link 关联表关联 3~10 种食材，注明各食材在菜品中的角色（锅底/主料/蘸料/配菜）、重要性（0~1）及是否本地物产；
3. 数据生产由 LLM 风味向量提取管线自动化完成，人工仅做异常值审核。

#### 传播事件数据需求

1. 每条传播事件需包含以下核心字段：
   - 路线名称（route_name，唯一）；
   - 路线类型（route_type：land/sea）；
   - 路线颜色（route_color）；
   - 路径几何（route_geom LineString, EPSG:4326），至少包含起点、终点及 2~3 个中途关键节点；
   - 传播媒介（vehicle：海运/内河/陆路/人为引种）；
   - 时期标签（time_label）；
   - 时序排序值（time_order）；
   - 文献摘录（literature_excerpt）。
2. 每条传播事件需引用其所传播的关键食材（ingredient_id FK）；
3. 数据生产由 LLM 传播事件录入管线自动化完成。

#### 叙事章节数据需求

1. 每章需包含标题、年代标签、正文、引用（cite）、来源（source）；
2. 每章可关联一条传播路线（route_name FK），用于在叙事页面生成路径缩略动画。

#### 原材料字典数据需求

1. 每种食材需包含标准名称、类型（香辛料/油脂/谷物/蛋白/蔬果/发酵/其他）、风味标签数组（flavor_tags TEXT[]）；
2. 每种食材需通过 ingredient_origin 关联表与其原产地生态区建立 M:N 关系。

### 系统功能需求

#### 系统体系结构需求

本系统为面向公众的 WebGIS 科普应用，采用纯 B/S 架构：

1. 前端为 Vue 3 单页应用（SPA），通过 Vue Router 管理五个功能页面的路由切换，无需页面刷新；
2. 所有 GIS 渲染在前端浏览器中完成（MapLibre 渲染底图与矢量图层，Deck.gl 渲染 WebGL 数据图层）；
3. 后端 FastAPI 提供 RESTful 数据接口（业务数据查询 + 全局搜索）和栅格/矢量瓦片服务；
4. 栅格瓦片由 rio-tiler 从 COG 文件实时渲染，通过 LRU 缓存提升重复访问性能；
5. 矢量图层（海岸线、河流、陆地）在服务启动时加载至内存，生态区数据从 PostGIS 查询；
6. 前端与后端的 API 通信在开发环境中通过 Vite 代理完成，在生产环境中通过 Nginx 反向代理完成。

#### 数据库需求

##### 数据库体系划分

根据数据来源和用途的不同，本系统数据库体系划分为：

1. **基础地理数据库**：Natural Earth 物理矢量（海岸线、河流、陆地）——以内存 GeoJSON 形式常驻，不入库；
2. **生态区空间数据库**：WWF TEOW 全球生态区划——PostGIS MultiPolygon，含气候属性（JSONB）；
3. **风味业务数据库**：风味基因型、传播事件、食材字典、配方连结、叙事章节——PostgreSQL 关系表；
4. **栅格数据**：HYP COG GeoTIFF 文件——硬盘存储，按需读取渲染。

##### 数据库组织

本系统为单版本数据库，不区分现势库/历史库/工作库。原因：

1. 数据更新频率低（风味数据以批次导入为主，非实时增量）；
2. 数据量小（当前种子数据为百条级别，LLM 管线规模化后预计千条级别），全量备份成本低；
3. 系统定位于科普展示，非生产型 GIS 应用，无需支持复杂的数据版本管理。

数据备份采用 PostgreSQL pg_dump 定期全量备份策略。

#### GIS 通用功能需求

本系统作为 WebGIS 应用，需提供以下通用 GIS 功能：

##### 地图浏览

| 操作 | 说明 |
| --- | --- |
| 缩放 | 支持鼠标滚轮缩放、双击放大、捏合手势（触屏）。缩放范围：0~10 级（栅格瓦片可用范围） |
| 漫游 | 支持鼠标拖拽平移、触屏滑动。要求在各缩放级别下所有图层无缝跟随 |
| 投影切换 | 缩放级别 < 2.2 自动切换为 globe（球面）投影，2.2~2.8 为 morph（过渡）投影，> 2.8 为 mercator（墨卡托）投影。球面模式下自动添加极地遮罩，倾斜角归零 |
| 全图复位 | 提供"回到全球视角"操作，恢复至 zoom=1 的 globe 投影状态 |

##### 图层控制

| 操作 | 说明 |
| --- | --- |
| 图层显示/隐藏 | 用户可通过 Dev Tools 面板独立切换 L0~L3 四层图层的可见性 |
| 图层上下文联动 | 选择节点时自动显示 L2（传播路径）图层，选择生态区时自动强化 L1（生态区边界）线宽，清除选择时 L2 自动隐藏 |
| 图层渲染顺序 | L0（底图）→ L1（生态区边界，MapLibre）→ L2（传播路径，Deck.gl）→ L3（风味节点，Deck.gl + HTML 气泡），保证交互优先级从底到顶递增 |

##### 空间查询

| 功能 | 说明 |
| --- | --- |
| 点击查询 | 点击 L3 风味节点（ScatterplotLayer + HTML 气泡）→ 选中节点，flyTo 飞行至节点坐标；点击 L2 传播路径（PathLayer）→ 选中路线；点击 L1 生态区边界（MapLibre geojson click）→ 选中生态区 |
| 悬停提示 | 悬停 L2 传播路径时，显示路线名称 tooltip |
| 全局搜索 | 顶部搜索栏输入关键词，模糊匹配风味节点（dish_name/city_name/region_label）和传播路线（route_name），搜索结果以下拉列表展示，点击结果即跳转至地图对应位置并选中 |

#### 页面功能需求

##### 首页（HomeView）

1. Canvas 粒子动画：以 Catmull-Rom 曲线为路径，展示缓慢流动的光点粒子，营造"风味在大地上流动"的视觉氛围；
2. 品牌展示：居中显示项目名称"寻味地理"及副标题文案；
3. 入口按钮："开启地图"按钮，点击后导航至 `/map`；
4. 底部预览卡片：展示 3 张风味节点样本卡片（如川渝火锅、广式点心、兰州牛肉面），作为数据内容预告。

##### 探索地图页（MapView）

地图页为本系统的核心交互界面，需满足以下功能需求：

1. **四层地图叠加**：L0 物理底图（HYP 栅格 + 海岸线/河流）→ L1 生态区边界（TEOW 多边形边框）→ L2 传播路径（Deck.gl PathLayer 暗线 + TripsLayer 流光粒子）→ L3 风味节点（Deck.gl ScatterplotLayer 光晕 + HTML 气泡卡片）；
2. **投影自动切换**：根据当前缩放级别自动在 globe/morph/mercator 三种投影间平滑切换；
3. **气泡节点系统**：
   - 高缩放级别（zoom > 4）：节点以独立 HTML 气泡卡片形式展示，显示城市名、菜品缩略图、风味主标签；
   - 低缩放级别（zoom < 4）：节点自动聚类为气泡簇，显示聚合数量；
   - 点击气泡卡片/气泡簇 → 选中节点，地图 flyTo 飞行至节点位置；
4. **粒子流光动画**：每个传播路线上 3 个错峰粒子沿路径循环滑动，模拟"基因沿地理通道传播"的视觉效果。粒子在跨越 L1 生态区边界时短暂减速加亮；
5. **联动选择与互斥**（Pinia 状态管理）：
   - 选中节点 → 清空路线和生态区选择，显示 L2 图层，右侧面板显示风味雷达图 + 食材列表；
   - 选中路线 → 清空节点和生态区选择，显示 L2 图层，右侧面板显示路线信息；
   - 选中生态区 → 清空节点和路线选择，隐藏 L2 图层，强化 L1 线宽，右侧面板显示生态区信息；
   - 点击地图空白区域 → 清除所有选择，隐藏 L2；
6. **右侧详情面板（MapInspector）**：可折叠面板，根据选中对象类型动态切换显示内容：
   - 节点模式：风味雷达图（ECharts，六维）+ 菜品描述 + 食材标签列表 + 同属其他节点链接；
   - 路线模式：路线名称、类型（陆路/海路）、传播媒介、途经节点列表；
   - 生态区模式：生态区名称、生物群系中文名、气候双轴曲线图（ECharts，温度 + 降水量）；
7. **全局搜索**：顶部搜索栏，支持从任意页面搜索并跳转至地图页；
8. **图层图例**：左上角固定图例面板，标注当前各图层对应的视觉元素；
9. **开发工具面板**：右上角可折叠面板，包含图层独立开关（L0~L3）、当前缩放级别/投影模式/globe 状态等调试信息；
10. **键盘快捷键**：按 Escape 键清除当前选择。

##### 风味基因库页（LibraryView）

1. 分类筛选栏：按风味类型（全部/辛辣/鲜甜/咸香/酸鲜）筛选风味节点；
2. 卡片网格：以响应式网格布局展示风味节点卡片，每张卡片包含：城市名、菜品名、所属生态区、六维雷达图缩略图、主风味标签；
3. 卡片点击：点击卡片 → 跳转至 `/map` 并自动选中对应节点（通过 Pinia store 的 selectNode action）。

##### 时空叙事馆页（NarrativeView）

1. 章节导航：左侧纵向时间线，以圆点标记各章节位置，点击圆点切换章节，当前章节高亮；
2. 章节正文：右侧内容区，显示章节标题、年代标签、正文、引用出处；
3. 路径缩略动画：每章内容区顶部以小型 Canvas 展示对应传播路线的暗线粒子动画；
4. 探索入口：每章末尾提供"打开全地图自由探索"按钮，点击后导航至 `/map`。

##### 关于页（AboutView）

1. 数据来源列表：从 `/api/data-sources` 获取数据，展示 Natural Earth、WWF TEOW、Wikimedia Commons 等数据来源的名称、描述、链接；
2. LLM 辅助数据清洗说明：以折叠手风琴组件展示 LLM 管线的工作原理与数据质量控制策略。

#### 数据智能化生产管线功能需求

本系统设计两条 LLM 数据管线，用于解决手工标注的产能瓶颈。

##### 管线一：风味向量自动提取

1. 语料爬取：从开源菜谱网站、饮食人类学文献、地方美食志等来源爬取非结构化菜品描述文本；
2. LLM 提取：通过提示工程，将自然语言描述转化为以下结构化 JSON 输出：
   - 六维风味向量（麻/辣/咸/酸/甜/鲜，各维 0~1）；
   - 食材列表（含食材名称、角色、是否为本地物产）；
   - 食物属分类（如"火锅""烤鸭""面食"）；
3. 领域知识约束：LLM 提示中嵌入地理-风味因果链条（如"高湿环境 → 花椒/辣椒偏好"），确保提取结果具有地理可解释性；
4. 入库目标：`flavor_genotype` 表（风味向量→genome_vector JSONB）、`recipe_link` 表（食材关联）。

##### 管线二：传播事件结构化录入

1. 文献准备：收集历史古籍（《齐民要术》《本草纲目》等）、饮食交流史研究文献、地方志等文本语料；
2. LLM 抽取：从文献段落中自动识别并输出以下结构化字段：
   - 传播物（食材名称）→ 匹配 ingredient 表；
   - 起点/终点 → 映射至 TEOW 生态区编码；
   - 传播媒介（海运/内河/陆路/人为引种）；
   - 时期标签（朝代/年代区间）；
   - 原文摘录（文献原文引用）；
3. 地理多跳推理：LLM 需配合 TEOW 生态区知识库，将含蓄的古文地理参照（如"自天竺经南海至交广"）精确映射至生态区编码；
4. 入库目标：`dispersal_event` 表。

### 系统非功能性需求

#### 用户界面需求

本系统的界面设计遵循博物志风格的视觉基线，区别于传统 GIS 应用：

1. **底色体系**：明亮通透的米白与浅灰底色，让地球本身成为视觉主干，剔除冗杂 UI 控件；
2. **材质语言**：高透明度毛玻璃面板（约 80% 磨砂透白），边缘无感渐变融入背景，实现"悬浮且不可见边框"的视觉效果；
3. **色彩系统**：数据高亮色采用胭脂红、松石绿、琥珀黄等极高明度自然提取色，在素色底图上形成清晰视觉穿透力；
4. **字体系统**：标题使用衬线体（Noto Serif SC），正文使用人文主义无衬线体（Noto Sans SC），字重轻盈、字距疏朗；
5. **导航设计**：全部收拢为左下角一枚呼吸光点（NavDot），悬停时展开菜单，默认状态下不占用屏幕空间；
6. **响应式布局**：地图页采用左侧搜索 + 右侧详情面板 + 中央地图的三栏布局，详情面板可折叠。

#### 系统质量需求

1. **健壮性**：瓦片请求超出 COG 范围时返回透明占位瓦片，不抛出错误；数据库查询失败时返回明确的错误 JSON 响应而非服务崩溃；
2. **效率**：
   - 栅格瓦片响应时间：首次渲染 < 2 秒（COG 冷读取），LRU 缓存命中 < 50 毫秒；
   - API 响应时间：`/api/flavors` 和 `/api/routes` 等列表接口 < 500 毫秒；
   - 前端页面加载时间：首屏 < 3 秒（含 WebGL 初始化）；
3. **易用性**：页面间导航切换有平滑过渡动画（fade + slide，300 ms）；所有可点击元素在悬停时有视觉反馈；
4. **可扩展性**：风味节点和传播路径数量增长至千条级别时，前端渲染性能不应有明显退化（Deck.gl 自动做视锥体裁剪）；数据库查询应依赖索引而非全表扫描；
5. **跨平台兼容性**：前端在 Chrome、Edge、Firefox、Safari 四大主流浏览器上表现一致，在 Windows、macOS、Linux 操作系统上无功能差异。

#### 数据质量需求

1. 所有空间数据统一使用 WGS84（EPSG:4326）坐标系；
2. TEOW 生态区边界在入库时需通过 ST_MakeValid 修正自相交等拓扑错误；
3. 风味向量的各维度值需在 0~1 区间内校验，超出范围的数据由 LLM 管线重新生成或人工修正后入库；
4. 传播路径的 LineString 几何需与生态区 MultiPolygon 在空间上可对应（起点/终点坐标落在对应生态区多边形内）。

#### 性能需求

1. **并发支持**：系统面向公众用户，需支持 50~100 并发访问；
2. **瓦片缓存**：LRU 缓存容量 512 条目，满足常规浏览场景下的缓存命中率 ≥ 60%；
3. **栅格瓦片渲染**：使用 ThreadPoolExecutor（6 线程）并行处理瓦片请求，避免阻塞 asyncio 事件循环；
4. **数据库连接池**：1~4 连接的 ThreadedConnectionPool，应对并发 API 请求；
5. **前端帧率**：Deck.gl 动画渲染（TripsLayer 粒子）维持 ≥ 30 FPS。

#### 安全需求

1. **CORS 限制**：后端仅允许开发域名（localhost:5173, localhost:4173）的跨域请求，生产环境通过反向代理消除跨域；
2. **数据库安全**：DATABASE_URL 以环境变量形式配置（.env 文件），不入库、不提交至版本控制；
3. **静态资源保护**：COG 栅格文件和 Shapefile 源文件仅通过 `/tiles/*` 接口暴露渲染后的瓦片，不提供原始文件下载；
4. **API 限流**（后续版本规划）：对 `/tiles/raster/*` 接口添加速率限制，防止恶意爬取。
