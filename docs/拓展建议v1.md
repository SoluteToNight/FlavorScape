# 拓展建议v1：寻味地理 GIS 的产品化、市场化与服务化方向

## 1. 当前项目适合往哪里扩展

“寻味地理”已经不只是一个课程展示页。它具备三个可以继续产品化的底层能力：

1. **风味空间表达能力**：把菜品、食材、生态区、贸易路线放到同一张地图里，适合做文化地理、文旅展示和地方品牌叙事。
2. **专题数据组织能力**：项目已经有食材传播 JSON、风味节点、生态区、路线、图层服务和 PostGIS 数据库雏形，后续可以沉淀成可维护的数据产品。
3. **叙事型 WebGIS 能力**：主页、地图页、食材传播页、叙事页可以组合成“可浏览、可讲解、可嵌入”的专题地图产品。

因此，后续不建议只把它做成“更多页面的 GIS 作业”。更合理的方向是：把它拆成若干可交付的专题模块，面向文旅、教育、地方食材品牌、餐饮品牌和数据服务场景，逐步变成“风味地理知识地图平台”。

## 2. 可落地的拓展项目

### 2.1 地方风味数字展陈系统

**面向对象**：地方文旅局、博物馆、非遗展馆、城市展厅、农产品展销馆。

**核心价值**：很多地方介绍美食时仍停留在图文展板和宣传片层面，缺少“为什么这个地方会长出这种味道”的空间解释。寻味地理可以把食材来源、水系、气候、交通、移民和菜品谱系组织成一张可讲解的地图。

**产品形态**：

- 一个地方专题地图，例如“成都麻辣风味地图”“潮汕牛肉火锅地理图谱”“北京烤鸭香料与城市消费图谱”。
- 展厅大屏版本：默认自动播放，但观众可以点击食材、路线、生态区查看解释。
- 讲解员版本：提供章节按钮，按“生态基础—食材输入—城市消费—代表菜品”逐步讲解。

**近期 MVP**：

- 先做 1 个城市样板，例如成都或重庆。
- 数据只需要 20-30 个点：核心菜品、关键食材、产地、港口/市场、历史传播节点。
- 输出一个可部署的专题页面和一份讲解脚本。

**需要补的能力**：

- 后台可编辑专题文本、图片和节点数据。
- 地图状态可保存为“讲解章节”。
- 支持导出静态截图，方便写报告、申报材料和宣传图。

### 2.2 地方食材品牌故事地图

**面向对象**：地标农产品品牌、区域公用品牌、地方农业公司、预制菜和调味品企业。

**核心价值**：许多食材品牌有产地优势，但表达方式常常只有“生态好、历史久、品质优”。GIS 可以把“产地可信度”具体化：土壤、水系、海拔、气候、运输路径、加工工艺和消费城市都能放到地图上。

**产品形态**：

- “一物一图”：例如郫县豆瓣、汉源花椒、宣威火腿、阳澄湖大闸蟹、普洱茶。
- 品牌官网嵌入式地图组件：可以嵌入企业官网、公众号 H5 或展会屏幕。
- 产地溯源轻服务：不是做完整供应链系统，而是展示可信的产地、加工和流通节点。

**近期 MVP**：

- 选择 1 个已有数据基础较强的食材，例如花椒、辣椒或八角。
- 做“产地环境 + 历史传播 + 现代消费城市”三层图。
- 输出一个可嵌入网页的地图组件，允许品牌方替换 Logo、主色、文案和图片。

**商业化方式**：

- 单个品牌专题制作费。
- 地方政府或协会的年度维护费。
- 多品牌地图组件授权费。

### 2.3 风味地理研学与数字教材

**面向对象**：中学地理课程、高校通识课、研学机构、博物馆教育部门。

**核心价值**：食物是地理教育的低门槛入口。学生更容易从“火锅为什么在四川重庆流行”“稻米为什么改变南北饮食结构”理解气候、水系、交通、农业和城市化。

**产品形态**：

- 课程包：每个专题包含一张交互地图、一组问题、一份教师讲义和一份学生任务单。
- 研学路线：把地方市场、博物馆、餐馆、产地和历史遗址串联成线下路线。
- 课堂互动：学生选择食材，观察传播路线和生态区变化，再回答地理解释问题。

**近期 MVP**：

- 做 3 个主题课件：辣椒入华、南稻北运、香料进入北方城市。
- 每个主题只保留 8-12 个关键节点，避免课堂使用时信息过载。
- 增加“问题卡片”：例如“为什么辣椒在西南扩散后影响更大？”“为什么稻米能通过漕运改变北京消费？”

**需要补的能力**：

- 教学模式开关：隐藏复杂图层，只保留关键点和问题。
- 章节进度条和讲义导出。
- 学生端轻量答题或记录功能。

### 2.4 餐饮品牌菜单叙事工具

**面向对象**：连锁餐饮、地方菜餐厅、火锅品牌、新中式餐饮品牌。

**核心价值**：餐饮品牌需要讲菜品故事，但传统菜单故事常常空泛。GIS 可以把“这道菜从哪里来、为什么在这里成型、它用了哪些地方食材”转成可视化体验。

**产品形态**：

- 店内屏幕：显示招牌菜的食材来源和传播路径。
- 点餐页组件：用户点开某道菜，看到核心食材的地理来源和历史卡片。
- 品牌故事页：把品牌主打菜系与气候、交通、城市消费连接起来。

**近期 MVP**：

- 选 1 个火锅品牌样板，围绕辣椒、花椒、牛油、豆瓣、盐做五条食材路径。
- 地图不追求学术级完整，重点是“可信、好看、能讲清楚”。
- 提供移动端竖屏版本，适配扫码点餐或公众号文章。

**商业化方式**：

- 按品牌定制收取设计开发费。
- 按门店数量收取屏幕展示授权费。
- 按菜品数量收取内容维护费。

### 2.5 风味知识库与 API 服务

**面向对象**：内容平台、餐饮数据公司、文旅平台、研究团队。

**核心价值**：如果项目只停留在前端页面，价值很难复用。把食材、菜品、地点、生态区和传播事件沉淀为结构化知识库后，可以对外提供查询、嵌入和专题生成能力。

**服务形态**：

- 食材传播 API：输入食材名，返回原产地、传播节点、事件文本、路线 GeoJSON。
- 风味节点 API：输入城市或菜品，返回风味维度、代表食材、生态区和相似菜品。
- 专题地图生成 API：输入主题配置，生成一个可嵌入的 story map。

**近期 MVP**：

- 先规范 `data/ingredient/*.json` 的 schema。
- 给每条传播事件增加 `event_id`、`region_key`、`confidence`、`source_type`、`citation` 字段。
- 建立一个轻量后台，支持新增、审核、发布数据。

**长期价值**：

- 内容可以授权给第三方平台。
- 数据可以支撑后续推荐、搜索和知识问答。
- 项目可以从“一个网站”升级为“风味地理数据服务”。

### 2.6 区域产业辅助分析工具

**面向对象**：地方农业部门、产业园区、食材加工企业、文旅规划团队。

**核心价值**：产业化不一定要一开始就做复杂预测。先做“产业展示 + 基础空间分析”更稳妥：展示某类食材的产地、加工点、消费城市、物流距离、品牌故事和文旅资源。

**产品形态**：

- 食材产业一张图：产区、加工、市场、品牌、文旅点位。
- 风味产业招商页：说明某地为什么适合发展某类风味产业。
- 产地与消费地连接图：帮助解释“地方食材如何进入城市餐桌”。

**近期 MVP**：

- 以一个省或一个城市群为单位。
- 加入基础统计：节点数量、路线长度、主要消费城市、关键产业环节。
- 输出政府汇报版和网页展示版两套视图。

## 3. 页面功能的下一步产品化改造

### 3.1 食材传播页

当前最应该继续增强，因为它最能体现项目差异化。

建议方向：

- 默认显示全部食材传播路径，但线宽、透明度和节点大小保持克制。
- 用户选择单个食材后，强化该食材路径，右侧展示完整历史文本。
- 节点使用真实食材图标，不再使用数字圆点；数字只作为时间线里的辅助信息。
- 路线使用曲线、柔和光晕、方向粒子和首传节点高亮，避免粗糙折线。
- 对同一地区重复传播只绘制第一次传入，后续记录保留在文本卡片中。
- 后续数据多了以后，可以增加“按时间段筛选”“按传播类型筛选”“只看中国境内/全球路径”等控制。

### 3.2 地图主页

主页地图可以从“展示全部风味节点”升级为“风味检索入口”。

建议方向：

- 搜索城市、菜品、食材后，地图自动聚焦相关节点。
- 支持按菜系、味型、生态区、火锅类型筛选。
- 节点卡片中增加“查看相关食材传播”的入口，直接跳转到传播页并选中对应食材。

### 3.3 数据后台

如果继续手写 JSON，项目很难扩展到真实服务。

建议优先做一个轻后台：

- 食材管理：名称、图片、原产地、颜色、简介。
- 传播事件管理：年份、地点、坐标、路线、来源、可信度。
- 专题管理：选择若干食材和节点，生成一个专题地图。
- 发布机制：草稿、审核、发布三个状态，避免错误数据直接出现在前台。

## 4. 数据规范建议

后续每条传播事件建议补充这些字段：

```json
{
  "event_id": "pepper-zhejiang-1591",
  "region_key": "浙江",
  "confidence": "medium",
  "source_type": "地方志",
  "citation": "《遵生八笺》相关条目",
  "display_level": "core"
}
```

字段含义：

- `event_id`：避免前端用年份和地名拼接 key。
- `region_key`：用于识别同一省份或同一地区，减少重复绘制。
- `confidence`：区分确证记录、推测路径和展示性节点。
- `source_type`：区分考古、地方志、文献、学术研究、现代产业数据。
- `display_level`：决定默认总览是否显示，数据变多后非常重要。

## 5. 推荐的阶段计划

### 第一阶段：做出可展示的多食材传播图谱

周期：1-2 周。

目标：

- 至少 3 种食材：辣椒、八角、稻米。
- 支持全部总览和单食材筛选。
- 地图节点使用食材图标，路径具备方向感和首传节点说明。
- 每个节点悬停显示历史卡片。

验收标准：

- 缩小地图时节点不会堆成一团。
- 选择单个食材后，右侧能完整显示传播历史。
- 路线不是简单折线，而是有地图美学的曲线与流动效果。

### 第二阶段：做一个可交付专题样板

周期：2-4 周。

建议主题：`成都/重庆麻辣风味地理`。

内容包括：

- 地图页：展示川渝火锅相关城市、生态区和食材路线。
- 食材页：辣椒、花椒、豆瓣、盐、牛油等传播或供应路径。
- 叙事页：解释湿热盆地、盐业网络、移民和城市消费如何共同塑造麻辣风味。
- 导出材料：一份专题说明 PDF 或演示稿。

### 第三阶段：轻后台和嵌入式组件

周期：4-8 周。

目标：

- 数据不再完全依赖手写文件。
- 可以为不同客户生成不同专题地图。
- 可以把某一条食材传播图嵌入外部网页。

这一阶段完成后，项目就不只是课程作品，而是具备小型商业交付能力的 WebGIS 服务原型。

## 6. 风险与边界

1. **历史数据不能过度确定化**：很多食材传播路径是推测性的，页面上应区分“确证记录”和“解释性路径”。
2. **不要过早做大而全平台**：先做 1-2 个高质量专题样板，比空泛地做账号系统、复杂后台更有说服力。
3. **视觉效果服务于解释**：路径粒子、光晕、图标都应该帮助用户理解方向和层级，而不是单纯炫技。
4. **商业化先从定制交付开始**：早期不适合直接做通用 SaaS，先用文旅、品牌、课程样板验证需求。

## 7. 可视化参考

- Jenny 等人的 OD flow map 设计原则强调减少路径交叉、控制视觉层级、避免流线遮挡关键信息，这与本项目的“首传节点去重、总览降噪、单食材强化”方向一致：<https://berniejenny.info/pdf/2017_Jenny_etal_DesignPrinciplesForODFlowMaps.pdf>
- MapLibre 的 symbol 图层支持图标碰撞避让、缩放表达和数据驱动样式，适合替代 DOM marker，避免地图缩小时节点堆叠：<https://maplibre.org/maplibre-style-spec/layers/>
