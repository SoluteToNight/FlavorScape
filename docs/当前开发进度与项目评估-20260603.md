# 当前开发进度与项目评估

更新日期：2026-06-03

本文档用于概括 FlavorScape / 寻味地理当前开发进度、整体技术架构，以及从用户和开发者两个角度评估现阶段的网站局限、市场适应性、产业化程度、项目深度和审美问题。

## 1. 最近更新内容

### 1.1 产品结构调整

- 顶部导航已从早期的泛产品入口，调整为更明确的产品模块：`产品入口`、`智慧大屏`、`营销海报`、`实证白皮书`、`传播图谱`、`风味底图`。
- `/brand` 当前定位为智慧大屏 / 地理志页面，不再承载旧的品牌公示、产业链、风物档案多模板切换。
- `/marketing` 新增为商业营销海报生成器入口，承接品牌物料和营销展示类需求。
- `/archive` 新增为实证白皮书页面入口，承接资料证明、科研表达、产品可信度包装等内容。
- `/spread` 保留食材传播图谱，继续承担食材历史传播与节点关系表达。
- `/map` 保留 WebGIS 风味底图能力，是完整地图数据和空间交互的基础页面。

### 1.2 智慧大屏页面更新

- `src/views/GeoAtlasView.vue` 已聚焦为智慧大屏 / Geo-Atlas 风格页面。
- 页面采用全屏沉浸式地图作为主视觉，叠加 HUD 信息面板、产品切换、节点时间轴、底部电影字幕式说明。
- 当前支持两个样例产品：`汉源花椒` 和 `五常大米`。
- 每个产品包含核心产地、加工、仓储、市场、餐桌等节点，并通过地图飞行动画与自动巡航串联。
- 旧的品牌公示、产业链、风物档案模板逻辑已从 `/brand` 方向中剥离，当前产品路线更聚焦。

### 1.3 地图底图处理

- 智慧大屏曾尝试接入天地图卫星影像和 Esri 影像，但在当前开发环境中存在加载失败或显示不稳定问题。
- 当前 `/brand` 已回退到项目原本的本地 FastAPI 瓦片：`/tiles/raster/{z}/{x}/{y}.png`。
- 这意味着智慧大屏依赖 FastAPI 后端和本地 GIS 数据；如果只启动 Vite 前端，地图底图可能一直处于加载状态。

### 1.4 文件与文档整理

- 新增 `docs/项目文件架构.md`，用于说明核心运行文件、前端结构、开发资料和启动关系。
- 非核心素材已整理：设计插图进入 `docs/assets/illustrations/`，参考图进入 `docs/assets/reference-pictures/`，日志和临时大包进入 `dev/` 相关目录。
- 当前文档体系开始从“交接记录”转向“架构说明 + 进度评估 + 设计资料”的组合，但仍需要进一步收敛和去重。

## 2. 整体开发架构

### 2.1 前端架构

前端基于 Vue 3、Vite、Vue Router 和 Pinia。

- `src/main.js`：Vue 应用入口，挂载路由、状态和全局样式。
- `src/App.vue`：应用外壳，负责承载导航和路由页面。
- `src/router/index.js`：前端路由表，定义首页、智慧大屏、营销海报、白皮书、传播图谱、风味底图等页面。
- `src/components/Navbar.vue`：全站导航和搜索入口。
- `src/stores/app.js`：全局选择状态，主要服务地图节点、路线、生态区等互斥选择。
- `src/styles/global.css`：全局变量、字体、基础布局和视觉基调。

主要页面：

- `HomeView.vue`：产品入口和项目总览。
- `GeoAtlasView.vue`：智慧大屏 / 地理志，当前主打沉浸式地图叙事。
- `MarketingView.vue`：营销海报与商业物料生成方向。
- `ArchiveView.vue`：实证白皮书方向。
- `IngredientSpreadView.vue`：食材传播图谱。
- `MapView.vue`：完整 WebGIS 风味底图。
- `LibraryView.vue`、`NarrativeView.vue`、`AboutView.vue`：资料、叙事和项目说明页面。

### 2.2 后端架构

后端基于 FastAPI。

- `backend/main.py`：FastAPI 服务入口。
- `backend/routers/api.py`：业务 API 路由。
- `backend/routers/tiles.py`：地图瓦片和矢量数据服务。
- `backend/routers/ingredient_spread.py`：食材传播相关接口。
- `backend/db/`：PostgreSQL / PostGIS 连接、迁移、查询和导入逻辑。
- `backend/data/app_data.py`：展示层元数据。
- `backend/startup.py`：启动时加载 GIS 数据、栅格、矢量和数据库连接。

开发时完整运行通常依赖三部分：

1. FastAPI 后端：`127.0.0.1:8001`。
2. Express：`3001`，由 `npm run dev` 一起启动。
3. Vite：`5173`，浏览器访问入口，代理 `/api/*` 和 `/tiles/*` 到 FastAPI。

### 2.3 数据架构

- `data/ingredient/*.json`：食材传播图谱页面的数据来源。
- PostGIS 数据库：承载更正式的风味节点、路线、生态区、空间查询能力。
- GIS 栅格与矢量数据：支撑 `/tiles/raster/*` 和 `/tiles/vector/*`。
- `api/data.js`：Express 生产模式下的轻量硬编码数据源，但不是完整 WebGIS 数据源。

### 2.4 产品模块关系

当前网站更像一个多原型集合，而不是已经完全收敛的单一产品。

- 智慧大屏负责“展示冲击力”和空间叙事。
- 营销海报负责“商业物料生成”。
- 实证白皮书负责“可信证明和报告包装”。
- 传播图谱负责“知识图谱和历史传播”。
- 风味底图负责“真实 WebGIS 能力”。

这些模块方向明确，但数据、交互和商业闭环尚未完全打通。

## 3. 当前局限、痛点与项目评估

### 3.1 用户视角

#### 开发局限

- 用户需要完整启动前端、后端和 GIS 数据服务，才能看到地图型页面的完整效果。
- 外部底图服务不稳定时，用户只能依赖本地瓦片；如果后端未启动，智慧大屏会出现持续加载。
- 页面模块多，但用户不一定能理解每个模块之间的关系：智慧大屏、营销海报、白皮书、传播图谱、风味底图目前仍像多个独立体验。
- 当前样例产品较少，主要围绕汉源花椒、五常大米，用户难以判断系统对其他品类的泛化能力。

#### 市场适应性

- 当前视觉表达有吸引力，适合汇报、路演、竞赛、课程设计和概念展示。
- 面向真实地方品牌客户时，还缺少明确的交付形态：是 SaaS 工具、定制可视化服务、品牌内容生成器，还是农产品溯源平台，需要继续收敛。
- 中小农产品品牌更关心低成本、快速生成、可直接投放；当前系统的 GIS 和大屏表达较重，可能超出普通商家的使用门槛。
- 政府、园区、展厅、文旅项目可能更容易接受智慧大屏和白皮书方向，但这类市场更依赖项目制交付和数据权威性。

#### 产业化程度

- 当前具备产品演示雏形，但还不是可直接商业部署的成熟产品。
- 缺少后台管理、数据录入、模板配置、用户权限、项目保存、导出管理等产业化基础能力。
- 溯源、检测、地理标志等内容目前更多是展示型文本，尚未形成可审计、可追溯、可更新的数据链路。
- 如果要进入产业场景，需要明确数据来源、版权、地图服务授权、品牌素材授权和客户交付流程。

#### 项目深度

- 项目概念有深度：把地方食材、地理空间、风味传播、品牌叙事结合在一起。
- 但当前很多内容仍是“样例叙事”，真实数据密度不足，尤其是检测报告、产区边界、生产批次、物流节点、市场销量、用户画像等数据尚未形成体系。
- 用户能看到漂亮的故事，但暂时难以验证背后的数据可信度。

#### 审美缺陷

- 智慧大屏风格偏强视觉、偏展示屏，对普通网站用户来说可能信息压力较大。
- 暗色 HUD、科技感地图、电影字幕式叙事适合展厅，但不一定适合电商详情页或普通品牌官网。
- 当前不同页面的视觉语言还不完全统一：有的偏地理信息系统，有的偏品牌海报，有的偏科研报告，整体品牌识别需要继续统一。

### 3.2 开发者视角

#### 开发局限

- 前端页面迭代较快，部分页面仍带有原型阶段痕迹，组件抽象不足。
- `GeoAtlasView.vue` 这类页面承载大量数据、地图逻辑、动画逻辑和样式，后续维护成本偏高。
- 地图能力依赖 FastAPI、GIS 数据、PostGIS、Vite 代理和本地瓦片，链路较长，排错成本高。
- 开发环境和生产环境能力不一致：开发时 Vite 代理 `/tiles/*`，生产 Express 当前不代理 FastAPI 瓦片。
- 外部底图尝试暴露了风险：第三方地图服务可能存在跨域、密钥、服务稳定性、授权和坐标系适配问题。

#### 市场适应性

- 从开发角度看，当前技术路线更适合“定制化可视化项目”，而不是立即做标准 SaaS。
- 如果要做 SaaS，需要把产品、模板、数据、导出、账号、项目管理全部配置化。
- 如果做项目制交付，则需要沉淀可复用模板、数据录入规范和交付检查清单，减少每个客户都从代码层改页面。

#### 产业化程度

- 当前最大短板不是单页视觉，而是工程化产品能力。
- 缺少稳定的数据后台、内容审核流、模板管理、客户素材管理、导出队列、部署脚本、监控和错误日志。
- 地图服务还没有生产级部署方案；如果依赖 FastAPI 瓦片，生产环境需要反向代理或独立部署策略。
- 真实产业场景会要求数据可验证、可维护、可更新，而不仅是一次性可视化。

#### 项目深度

- 代码层已经具备地图、图谱、营销、报告多个方向的探索，但模块之间共享的数据模型还不够清晰。
- `data/ingredient/*.json`、`backend/data/app_data.py`、PostGIS 查询、Express `api/data.js` 存在多套数据来源，长期会产生口径不一致。
- 如果要提升深度，应建立统一的食材实体模型：产区、品类、风味、工艺、证据、路线、检测、品牌素材、输出模板都应围绕同一数据结构组织。

#### 审美与前端实现痛点

- 当前审美方向有冲击力，但存在“过度舞台化”的风险；如果所有页面都追求强视觉，会降低信息阅读效率。
- 不同页面的卡片、按钮、字体、颜色、空间密度还需要统一设计系统。
- 部分页面更像静态视觉稿，交互状态、错误状态、空数据状态、移动端体验还不完整。
- 大屏页面依赖复杂绝对定位和动画，适合固定展示，但响应式和可维护性需要继续加强。

## 4. 下一步建议

- 明确产品主线：优先选择“地方食材品牌可视化生成器”或“展厅级地理志大屏”作为第一阶段主产品。
- 统一数据模型：把食材、产区、节点、路线、证据、检测、图片和模板参数整理成一套可复用结构。
- 收敛页面职责：智慧大屏负责展示，营销海报负责传播物料，白皮书负责证明材料，不要让单页承担所有功能。
- 完善生产部署：明确 FastAPI、Express、Vite 静态资源、PostGIS 和瓦片服务的生产部署关系。
- 建立设计系统：统一颜色、字体、按钮、面板、卡片、图表和地图标注风格。
- 增加真实数据：优先补充 3-5 个真实地方食材样例，用真实产区、检测、工艺、图像和销售场景验证系统泛化能力。
