# 数据库设计

数据库是“寻味地理”风味博物志 WebGIS 系统的数据基础。本章对系统数据库的数据内容、设计原则、总体结构、生态区空间数据库、风味业务数据库、物理存储和维护方式进行说明。系统数据库以 PostgreSQL/PostGIS 为核心，统一管理空间几何、菜肴成品、食材、传播叙事内容和数据来源信息。

## 数据库内容

### 数据概况

系统数据库涉及的主要数据如下：

| 序号 | 数据名称 | 数据类型 | 主要用途 |
| --- | --- | --- | --- |
| 1 | Natural Earth 栅格底图 | COG GeoTIFF | 作为地图自然地形背景，按需生成栅格瓦片 |
| 2 | Natural Earth 物理矢量 | 矢量图层 | 提供海岸线、河流、陆地轮廓等基础地理参照 |
| 3 | WWF TEOW 生态区 | PostGIS MultiPolygon | 作为风味形成和食材来源解释的自然空间单元 |
| 4 | 菜肴成品 | PostgreSQL 关系表 + Point | 记录地方食物版本、坐标、风味基因型属性和展示属性 |
| 5 | 食材 | PostgreSQL 关系表 | 记录食材名称、类型和风味标签 |
| 6 | 传播事件 | PostgreSQL 关系表 + LineString | 记录食材、技法和味型的历史传播路径 |
| 7 | 叙事章节 | PostgreSQL 关系表 | 记录时空叙事馆的章节内容和引用来源 |
| 8 | 数据来源 | PostgreSQL 关系表 | 记录地图、生态区、图片和文本材料来源 |

### 数据入库前处理

生态区空间数据入库前应进行坐标系检查、几何有效性检查、属性字段整理和必要的边界简化。风味业务数据入库前应进行字段完整性检查、风味基因型属性取值范围检查、食材名称规范化和生态区映射检查。传播事件数据入库前应检查路线几何、起止生态区、传播对象和来源说明。

基础栅格底图不作为关系表入库，而是以文件形式保存，由后端服务按需读取并生成地图瓦片。海岸线、河流和陆地轮廓等物理矢量可在服务启动时加载，用于地图基础图层输出。

## 数据库设计原则

1. **空间与属性一体化**：生态区边界、菜肴成品坐标和传播路径几何均在统一空间参考下组织，保证地图叠加和空间查询一致。
2. **核心实体清晰**：生态地理单元、菜肴成品、食材、传播事件、叙事章节和数据来源作为核心实体，分别承担明确业务职责。
3. **属性语义清晰**：风味基因型是菜肴成品的六维风味向量属性，不作为独立成品实体。
4. **关联实体规范**：食材原产地、配方连结和菜肴溯源链作为关联实体，用于表达多对多关系和自引用关系。
5. **减少数据冗余**：可独立成为实体的数据应通过外键和关联表组织，避免在多个表中重复维护。
6. **索引驱动查询**：空间几何、关联字段、标签数组和半结构化字段应建立相应索引，保证地图浏览和搜索查询效率。

## 数据库总体设计

### 数据库体系构成

系统数据库采用单数据库、单 Schema 的组织方式，主要包含以下数据组：

| 数据组 | 数据表 | 说明 |
| --- | --- | --- |
| 生态区空间数据组 | `eco_geo_unit` | 保存 WWF TEOW 生态区边界和属性 |
| 风味业务数据组 | `dish`、`ingredient`、`ingredient_origin`、`recipe_link` | 保存菜肴成品、食材及其关联关系 |
| 传播叙事数据组 | `dispersal_event`、`chapter`、`dish_lineage` | 保存传播路径、叙事章节和菜肴谱系关系 |
| 系统元数据组 | `data_source` | 保存数据来源说明 |
| 枚举类型 | `ingredient_type` | 约束食材类型取值 |

系统当前采用单版本数据库组织方式，不划分现势库、历史库和工作库。该方式适合当前展示型 WebGIS 和批次导入数据模式。后续若出现频繁更新、多人编辑或历史版本追踪需求，可扩展版本字段或建立历史表。

### 数学基础设计

| 属性 | 取值 |
| --- | --- |
| 坐标系 | WGS84 大地坐标系 |
| 空间参考标识符 | EPSG:4326 |
| 坐标单位 | 十进制度 |
| 几何类型 | Point、LineString、MultiPolygon |
| 地图渲染 | 浏览器端根据地图引擎进行显示转换 |

### 数据库对象命名规则

| 对象类型 | 命名规则 | 示例 |
| --- | --- | --- |
| 表名 | 小写蛇形命名 | `eco_geo_unit`、`dish` |
| 主键 | 能表达业务对象身份的 ID 或名称 | `dish_id`、`eco_name` |
| 外键 | 与引用对象保持语义一致 | `ingredient_id`、`eco_name` |
| 空间字段 | 按几何含义命名 | `boundary`、`coordinates`、`route_geom` |
| 索引 | 使用表名和字段含义组合 | `idx_dish_coord` |

## 生态区空间数据库设计

### 数据内容

生态区空间数据库以 WWF TEOW 全球陆地生态区为核心。每个生态区记录生态区名称、中文名称、生态区编码、生物地理界、生物群系、面积、气候属性、相邻生态区、本地优势食材、环境描述和边界几何。

生态区边界是系统进行地图展示和空间解释的主要面状数据。生态区属性用于支撑地图详情面板、菜肴成品生态解释和食材原产地关联。

### 概念模型设计

生态地理单元是系统的核心空间容器。一个生态区可包含多个菜肴成品，也可作为多种食材的可能来源地。传播事件可以从一个生态区指向另一个生态区，表达食材、技法或味型在自然空间中的迁移过程。

### 逻辑模型设计

**表名：`eco_geo_unit`**

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `eco_name` | VARCHAR(150) | 主键 | 生态区英文名称 |
| `eco_name_cn` | VARCHAR(100) | 可空 | 生态区中文名称 |
| `eco_code` | VARCHAR(20) | 可空 | 生态区编码 |
| `realm` | VARCHAR(20) | 可空 | 生物地理界 |
| `biome` | VARCHAR(80) | 可空 | 生物群系英文名 |
| `biome_cn` | VARCHAR(30) | 可空 | 生物群系中文名 |
| `area_km2` | NUMERIC | 可空 | 面积，单位平方千米 |
| `climate` | JSONB | 可空 | 气候属性扩展 |
| `adjacent_ecos` | TEXT[] | 可空 | 相邻生态区 |
| `dominant_ingredients` | TEXT[] | 可空 | 本地优势食材 |
| `description` | TEXT | 可空 | 环境描述 |
| `boundary` | GEOMETRY(MultiPolygon, 4326) | 可空 | 生态区边界 |

### 物理模型设计

生态区边界字段建立 GiST 空间索引，用于生态区图层输出和空间过滤。生态区编码字段建立普通索引，用于按编码检索。气候属性字段建立 GIN 索引，为后续 JSONB 属性查询预留能力。

生态区边界在入库或输出前应进行几何有效性检查和必要简化，避免复杂边界造成地图传输和前端渲染压力过大。

## 系统业务数据库设计

### 数据内容

系统业务数据库包括菜肴成品、食材、食材原产地、配方连结、传播事件、叙事章节、菜肴溯源链和数据来源。其核心目标是将地方食物、生态环境、食材来源、传播路径和叙事材料组织为可查询、可展示、可扩展的数据体系。

### 概念模型设计

核心关系如下：

| 关系 | 类型 | 说明 |
| --- | --- | --- |
| 生态地理单元与菜肴成品 | 一对多 | 一个生态区可关联多个地方菜肴 |
| 食材与生态地理单元 | 多对多 | 一种食材可对应多个可能原产生态区 |
| 菜肴成品与食材 | 多对多 | 一种地方菜肴由多种食材构成 |
| 传播事件与食材 | 多对一 | 一条传播路径可指向一个关键传播食材 |
| 传播事件与生态地理单元 | 多对一 | 路线起点和终点分别关联生态区 |
| 传播事件与叙事章节 | 一对多 | 一条路线可关联多个叙事章节 |
| 菜肴成品与菜肴成品 | 自引用多对多 | 用于表达菜肴谱系或变异关系 |

食材原产地、配方连结和菜肴溯源链属于关联实体。它们不仅用于连接核心实体，还可记录关系自身的业务属性。

### 物理模型设计

#### 空间索引

| 表 | 字段 | 索引类型 | 用途 |
| --- | --- | --- | --- |
| `eco_geo_unit` | `boundary` | GiST | 生态区空间过滤和图层输出 |

| `dispersal_event` | `route_geom` | GiST | 传播路线空间查询 |

#### 半结构化与数组索引

| 表 | 字段 | 索引类型 | 用途 |
| --- | --- | --- | --- |
| `eco_geo_unit` | `climate` | GIN | 气候扩展属性查询 |

| `ingredient` | `flavor_tags` | GIN | 食材标签查询 |

#### 普通关系索引

数据库应在生态区编码、菜肴成品生态区、城市名、菜品名、传播事件名称、传播事件顺序、起止生态区、配方食材和章节路线等字段上建立普通索引，以提升列表查询、搜索和多表关联查询效率。

## 数据库建库与维护更新

### 建库流程

数据库建库应完成空间扩展启用、核心表创建、枚举类型创建、索引创建、生态区数据导入、风味业务数据导入和数据质量检查等步骤。建库完成后，应检查各表记录数量、空间字段有效性、外键关联完整性和主要接口输出结果。

### 数据维护更新

数据维护以批次导入为主。菜肴成品、食材、传播事件和叙事章节更新后，应同步检查关联实体是否完整、风味基因型属性取值是否符合范围、传播路径是否具有有效几何、叙事章节是否能正确关联路线。

数据库应定期进行逻辑备份。后续如系统进入多人协作或高频更新阶段，可进一步引入数据版本、审核状态、历史记录和回滚机制。
