# 需求规定

本章主要根据“寻味地理”风味博物志 WebGIS 系统的建设目标、数据基础和应用场景，从数据现状、现有数据分析、相关项目分析和运行环境等方面出发，规定系统的数据需求、功能需求和非功能性需求。系统需求以饮食文化空间化表达为核心，围绕生态地理单元、菜肴成品、食材构成、传播路径和叙事内容等对象展开。

## 现状

### 数据现状

#### 基础地理数据

基础地理数据是系统进行空间表达和地图浏览的底层支撑。系统采用 Natural Earth 物理底图和物理矢量数据作为自然地理背景，突出地形、水系、海岸线等要素，弱化现代行政边界对饮食文化解释的干扰。

Natural Earth 高程分层设色栅格底图用于提供全球尺度的自然地形背景。该数据以高程、地貌和自然色彩为主要表现内容，不包含国界线和行政区划标注，能够满足本系统“以自然地理为视觉基底”的展示要求。

Natural Earth 物理矢量数据用于提供海岸线、河流和陆地轮廓等基础地理骨架。上述矢量要素在地图中主要承担空间参照作用，帮助用户判断风味节点和传播路径与山地、水系、海岸和大陆轮廓之间的关系。

#### 生态地理数据

系统采用 WWF 全球陆地生态区划作为风味形成的自然空间框架。生态区划不是按行政边界划分，而是依据气候、植被、物种分布和生态系统特征组织空间单元，更适合用于解释食材供给、环境适应和味型形成之间的关系。

当前生态地理数据包括 827 个全球陆地生态区。每个生态区应具有生态区名称、生态区编码、生物地理界、生物群系、面积和边界几何等基本属性。系统在展示时需要能够输出生态区边界，并在用户点击生态区时返回其名称、类型和相关解释信息。

#### 风味业务数据

风味业务数据是本系统区别于普通地图应用的核心数据。当前系统以人工整理的种子数据为基础，已形成菜肴成品、食材、传播事件、叙事章节和数据来源等基本对象。

菜肴成品用于表示特定地域或生态环境下形成的地方食物版本。每个菜肴成品应具备食物名称、所在城市或地域、所属食物类型、风味基因型属性、坐标位置、风味标签、描述文本、关联食材和可视化展示信息。其中，风味基因型特指菜肴成品的六维风味向量属性。

食材数据用于说明风味形成的物质基础。食材应按香辛料、油脂、谷物、蛋白、蔬果、发酵品和其他类型进行分类，并记录其风味标签及可能对应的生态来源。食材与生态地理单元之间可存在多对多关系，应通过食材原产地关联表进行组织。

配方连结用于表示菜肴成品与食材之间的关系。该关联实体不仅连接地方食物和食材，还应记录食材在某一菜肴中的角色、贡献权重和是否为本地物产等关系属性。

传播事件用于表示食材、技法或味型在历史时空中的迁移过程。每条传播事件应包含路线名称、路线类型、关键节点、路径几何、传播媒介、时期说明和关联食材等内容。传播事件是地图流光动画和时空叙事内容的主要数据来源。

菜肴溯源链用于表示菜肴成品之间的演化关系。该关联实体面向后续扩展，用于记录同一食物类型下不同地域版本之间的继承、改造和变异关系。

叙事章节用于承载面向公众阅读的历史材料和解释文本。每个章节可围绕一条传播路径或一个典型饮食文化现象展开，提供年代标签、正文内容、引用来源和与地图对象的关联。

#### 媒体与来源数据

系统需要为风味节点、叙事章节和方法说明提供必要的媒体与来源信息。媒体数据主要包括菜品图片、地图展示图形和占位视觉资源；来源数据主要包括 Natural Earth、WWF TEOW、图片来源和饮食文化材料来源等。

数据来源信息应能够在关于页面或方法说明页面中展示，说明数据名称、来源机构、用途、链接和引用方式。对于未能提供稳定媒体资源的风味节点，系统应提供统一的占位展示策略，避免页面出现内容缺失。

### 现有数据分析

#### 基础地理数据分析

Natural Earth 数据具有覆盖范围广、开放程度高、制图质量稳定等特点，适合作为课程设计、原型系统和公开展示系统的基础地图数据。其高程设色栅格和物理矢量能够支持全球尺度浏览，满足风味传播路径跨区域展示的需要。

该类数据的主要限制在于比例尺较小，不适合表达街区级、餐馆级或精细行政单元级别的信息。因此，本系统不将其用于精确导航和位置测量，而是用于构建自然地理背景和宏观空间叙事。

#### 生态区数据分析

TEOW 生态区数据能够较好地表达自然环境差异，是系统组织风味地理知识的重要空间容器。与行政区划相比，生态区更能反映气候、植被和物产结构差异，对解释食材原产、调味偏好和烹饪方式具有更强的适用性。

生态区数据在系统使用中仍需处理以下问题：第一，原始属性以英文为主，需要补充中文名称；第二，部分边界几何较复杂，需要在保证空间表达合理性的前提下进行简化；第三，生态区本身不等同于气候数据集，若需要展示温度、降水等指标，应通过外部数据或人工整理内容补充。

#### 风味业务数据分析

当前风味业务数据以种子演示数据为主，能够支撑系统核心流程展示，但尚不能覆盖完整的中国饮食文化版图。现有风味节点数量有限，传播路径以典型历史路线为主，适合用于验证数据模型、地图交互和叙事方式。

风味向量具有较强解释力，但其标注需要统一标准。同一食物类型下不同地域版本之间应保持可比标度，不同食物类型之间应保持基本解释一致。若缺少统一标注规范，风味向量容易受到人工主观判断影响。

传播事件数据需要更严格的文献支撑和地理锚定。历史材料中的地名、路线和传播媒介往往存在时代差异，不能直接等同于现代地理坐标或生态区名称。后续扩展时应建立“文献摘录、地名解释、生态区映射、人工审核”的数据质量控制流程。

#### 数据生产现状分析

人工整理方式能够保证初期数据可控，但难以支撑规模化扩展。系统后续需要引入 LLM 辅助数据生产管线，用于从菜谱文本、地方美食志、饮食研究材料、古籍和地方志中提取结构化信息。

LLM 辅助管线不应直接替代人工审核。其输出应经过格式校验、取值校验、地名校验、生态区映射校验和人工复核后，方可进入数据库。对于缺少出处或解释链条不充分的数据，应标记为待核验数据，不进入正式展示范围。

### 现存项目分析

#### 传统美食地图类应用

传统美食地图通常以地点和菜品为主要对象，重点回答“某地有什么食物”。这类应用便于检索和旅游消费决策，但通常缺乏对自然环境、食材结构和历史传播的解释。

本系统与传统美食地图的差异在于：系统不以餐馆检索或消费推荐为目标，而是以饮食文化地理解释为目标；系统不只展示菜品位置，还需要展示风味特征、生态背景、食材来源和传播路径之间的关系。

#### 食品科学数据库类项目

食品科学数据库通常关注食品成分、香气分子、营养指标和味觉机制，适合科研人员进行分子层面的查询与分析。此类项目的数据精度较高，但多以表格、分子结构和检索页面为主要形式，空间叙事和公众可读性相对不足。

本系统并不试图替代食品科学数据库，而是以 WebGIS 和数字人文叙事为主要表达方式，将风味作为可比较、可解释、可视化的文化地理对象。

#### 数字人文 GIS 项目

数字人文 GIS 项目通常关注历史地名、行政区划、人口迁移、交通网络和文化传播。该类项目为饮食文化研究提供了方法参考，但现有项目较少将食材、味型和生态环境放在同一空间框架中组织。

本系统可视为数字人文 GIS 在饮食文化领域的应用探索。其核心价值在于将地方食物从孤立条目转化为空间化、关系化和叙事化的数据对象。

### 运行环境现状

系统采用 B/S 架构，用户通过现代 Web 浏览器访问系统，不需要安装专用客户端。浏览器端承担地图渲染、页面交互、图表展示和 WebGL 可视化任务；服务端承担业务数据接口、栅格瓦片服务、矢量图层服务和数据库访问任务；数据库负责生态区空间数据和风味业务数据的统一管理。

系统运行需要具备 WebGL 能力的现代浏览器。对于不支持 WebGL 或图形性能较弱的设备，应至少保证页面能够给出明确提示，避免用户误认为系统无响应。

开发环境通常由前端开发服务器、Node 服务、后端服务和空间数据库共同组成。生产环境应保证前端静态资源托管、业务接口、瓦片服务和数据库连接稳定可用，并根据实际部署要求配置域名、HTTPS、反向代理和数据备份机制。

## 需求

### 项目建设总体要求

第一，以生态地理解释风味形成为系统建设主线。

1. 系统应以自然地理单元作为主要空间组织框架，避免仅用行政区划解释饮食文化差异；
2. 系统应将地方食物抽象为可比较的菜肴成品，并通过风味基因型属性表达其六维味型特征；
3. 系统应将食材、生态区、风味节点和传播路径建立关联，使用户能够理解“生态环境—食材供给—风味表达—历史传播”的关系；
4. 系统应以地图为主要入口，同时提供结构化浏览和线性叙事页面，满足自由探索、比较浏览和专题阅读三类需求。

第二，以统一数据模型支撑风味知识组织。

1. 系统应建立生态地理单元、菜肴成品、食材、传播事件、叙事章节和数据来源等核心数据对象；
2. 系统应建立食材原产地、配方连结和菜肴溯源链等关联实体，表达多对多关系和自引用演化关系；
3. 系统应统一空间坐标、风味向量、食材分类、路线类型和文献引用口径，保证后续数据扩展时仍能保持一致；
4. 系统应支持种子数据展示和后续规模化数据扩展，数据导入流程应保留格式校验和人工审核环节。

第三，以 WebGIS 可视化提高公众理解能力。

1. 系统应提供清晰、稳定、易操作的地图浏览能力，支持用户在不同尺度下理解风味节点和传播路径；
2. 系统应通过图层叠加、对象选择、详情面板和图表展示实现信息联动；
3. 系统应避免将界面设计成传统专业 GIS 工具台，面向公众用户提供简洁、可读、低学习成本的交互体验；
4. 系统应保证数据来源和方法说明可追溯，提高内容可信度。

### 数据需求

#### 基础地理数据需求

1. 系统应具备稳定的全球自然地形底图，用于展示地形、水系和陆海格局；
2. 系统应具备海岸线、河流和陆地轮廓等基础矢量图层，用于辅助识别风味传播路线和生态空间背景；
3. 基础地理数据应统一使用 WGS84 坐标系，保证与生态区、风味节点和传播路径在空间上可叠加；
4. 基础地理数据的使用应符合其来源许可和引用要求，并在数据来源说明中列明。

#### 生态地理数据需求

1. 系统应保存 WWF TEOW 生态区边界及其基本属性，支持地图显示、点击查询和空间关联；
2. 生态区对象应至少包含生态区名称、生态区编码、生物地理界、生物群系、面积和边界几何；
3. 生态区应补充适合公众阅读的中文名称和类型解释，避免只展示英文原始字段；
4. 生态区边界应经过几何有效性检查和必要简化，保证前端浏览时具有合理的加载速度和视觉效果；
5. 若生态区展示涉及气候指标，应明确其来源和处理方式，不得将 TEOW 生态区划本身误写为气候观测数据。

#### 菜肴成品数据需求

1. 每个菜肴成品应表示一个具体地方食物版本，如特定地域的火锅、点心、面食或其他地方风味；
2. 菜肴成品应记录食物名称、地域名称、所属食物类型、生态区关联、空间坐标、风味描述和主风味标签；
3. 菜肴成品应具备风味基因型属性，维度包括麻、辣、咸、酸、甜、鲜，各维取值范围应保持统一；
4. 菜肴成品应能够关联若干食材，并说明食材在该风味中的作用；
5. 同一食物类型下的不同地域版本应具备可比较性，便于系统展示风味变体和演化差异。

#### 食材与关联实体数据需求

1. 食材对象应记录标准名称、食材类型和风味标签；
2. 食材原产地关联表应表达食材与生态地理单元之间的多对多关系；
3. 配方连结表应表达菜肴成品与食材之间的多对多关系，并记录食材角色、贡献权重和本地性等关系属性；
4. 菜肴溯源链表应表达菜肴成品之间的祖先与后代关系，为后续描述风味变异和谱系演化预留数据结构；
5. 关联实体中的关系属性应采用统一命名和取值口径，避免同一含义在不同数据批次中出现多种表达。

#### 传播事件数据需求

1. 每条传播事件应记录路线名称、路线类型、传播对象、传播媒介、起点、终点、关键节点和路径几何；
2. 传播事件应支持陆路、海路等路线类型，并允许后续扩展其他传播方式；
3. 传播事件应记录时期标签或时间顺序信息，用于支撑时空叙事；
4. 传播事件应尽量提供文献摘录或参考来源，缺少来源的传播事件应在数据状态上进行区分；
5. 传播路径的几何表达应与地图尺度匹配，既能表达大致方向，又不应误导为精确历史路线复原。

#### 叙事章节数据需求

1. 叙事章节应记录标题、年代标签、正文、引用来源和关联传播事件；
2. 章节内容应采用公众可读的叙事语言，避免只堆砌字段或资料摘录；
3. 章节与地图对象之间应建立可跳转或可联动关系，使用户能从阅读进入地图探索；
4. 叙事章节应能随着传播事件和风味节点扩展进行增补。

#### 数据来源数据需求

1. 数据来源应记录名称、说明、发布机构或作者、链接和使用范围；
2. 系统应在关于或方法说明页面展示主要数据来源；
3. 对图片、地图、生态区划和文本材料等不同类型来源，应分别说明其用途；
4. 对后续 LLM 辅助生成的数据，应记录原始材料来源、抽取过程和人工审核状态。

### 系统功能需求

#### 系统体系结构需求

本系统采用前后端分离的 B/S 架构，前端负责页面呈现和交互，后端负责数据服务和瓦片服务，数据库负责空间数据与业务数据管理。

1. 前端应提供单页应用式的页面切换能力，使用户在首页、地图、风味基因库、时空叙事馆和关于页面之间顺畅切换；
2. 后端应提供风味节点、传播路径、叙事章节、数据来源和搜索等业务数据服务；
3. 后端应提供栅格瓦片和矢量图层服务，支撑地图底图、生态区边界和基础物理矢量展示；
4. 前端不得直接访问数据库，所有业务数据和空间数据均应通过服务端统一输出；
5. 系统应支持开发环境和生产环境的部署差异，保证业务接口与瓦片服务在两类环境下均能稳定访问。

#### 数据库需求

系统数据库应同时支持空间数据、关系数据和半结构化数据管理。

1. 数据库应能够存储生态区 MultiPolygon、风味节点 Point、传播路径 LineString 等空间几何；
2. 数据库应能够存储风味向量、食材标签、生态区属性和叙事文本等业务内容；
3. 数据库应通过外键和关联表维护核心实体之间的关系；
4. 数据库应建立必要的空间索引、文本检索索引和关系查询索引，保证地图浏览和搜索查询的响应速度；
5. 数据库应支持批次导入、数据重建和逻辑备份，满足课程设计、演示和后续扩展需要；
6. 系统当前可采用单版本数据库组织方式，后续若出现频繁更新或历史版本保留需求，可扩展版本字段或历史表。

#### GIS 通用功能需求

##### 地图浏览

1. 系统应支持地图缩放、平移、拖拽和视图复位等基本浏览操作；
2. 系统应在全球尺度和区域尺度下均保持图层对齐；
3. 系统应支持自然地形底图、物理矢量、生态区边界、传播路径和风味节点的叠加显示；
4. 系统应支持球面视图与平面视图之间的合理切换或过渡，满足全球尺度叙事和区域尺度浏览需要。

##### 图层控制

1. 系统应提供图层显示和隐藏控制，便于用户根据浏览目标调整地图信息密度；
2. 图层渲染顺序应保证底图位于最底层，生态区边界和物理矢量提供空间背景，传播路径和风味节点作为主要交互对象；
3. 系统应在选中节点、路线或生态区时调整相关图层的视觉状态，突出当前对象及其上下文；
4. 图层状态变化不应造成地图跳动或页面布局错位。

##### 空间查询

1. 用户应能够点击风味节点查看对应地方食物的详情；
2. 用户应能够点击传播路径查看路线类型、传播对象和关键节点信息；
3. 用户应能够点击生态区查看生态区名称、生物群系和相关说明；
4. 用户应能够通过关键词搜索风味节点和传播路径，并从搜索结果跳转至对应地图对象；
5. 系统应保证节点、路线和生态区的选择状态互斥，避免详情面板同时展示多个对象造成理解混乱。

##### 图表展示

1. 系统应以雷达图展示菜肴成品的风味基因型属性；
2. 系统可通过图表展示生态区的气候或类型信息，相关数据应有明确来源；
3. 图表应与地图选择状态联动，用户切换对象时图表内容应同步更新；
4. 图表不应替代文本说明，应与描述、标签和来源信息共同构成解释。

#### 页面功能需求

##### 首页功能

1. 首页应清晰展示系统名称、主题定位和进入地图探索的主要入口；
2. 首页应提供具有风味地理意象的视觉背景或动态效果，建立系统识别；
3. 首页可展示若干代表性风味节点或传播主题，引导用户理解系统内容范围；
4. 首页应保证在桌面和移动端均能快速进入核心功能。

##### 探索地图功能

探索地图是系统核心功能页面，应满足以下要求：

1. 展示自然地形底图、海岸线、河流、生态区边界、传播路径和风味节点；
2. 支持风味节点、传播路径和生态区的点击选择；
3. 支持右侧详情面板随选择对象动态切换内容；
4. 节点详情应展示食物名称、地域信息、风味雷达图、描述文本、食材标签和同类型风味关联；
5. 路线详情应展示路线名称、路线类型、关键节点和传播说明；
6. 生态区详情应展示生态区名称、生物群系、生物地理界和必要的环境解释；
7. 地图应提供搜索入口、图例说明和必要的图层控制；
8. 当用户清除选择时，地图应恢复普通浏览状态。

##### 风味基因库功能

1. 风味基因库应以结构化卡片方式展示风味节点；
2. 用户应能够按风味类型或标签筛选节点；
3. 每个卡片应展示食物名称、地域信息、所属生态区、主风味标签和风味向量概览；
4. 用户点击卡片后，应能够进入地图并定位到对应风味节点；
5. 风味基因库应适合横向比较不同地域风味版本。

##### 时空叙事馆功能

1. 时空叙事馆应以章节方式组织典型传播事件和历史材料；
2. 章节应具有标题、年代标签、正文和引用来源；
3. 页面应提供章节导航，便于用户在线性叙事之间切换；
4. 章节内容应能够关联地图中的传播路径或风味对象；
5. 用户应能够从章节阅读进入地图探索，继续查看相关空间对象。

##### 关于与方法说明功能

1. 关于页面应说明系统主题、数据来源和方法框架；
2. 页面应展示主要数据来源，包括基础地图、生态区划、图片材料和饮食文化材料；
3. 页面应说明风味向量、生态区映射和传播事件整理的基本原则；
4. 页面应说明 LLM 辅助数据生产的定位，即作为后续数据扩展工具，并需经过人工审核。

##### 异常页面功能

1. 当用户访问不存在的页面时，系统应展示友好的异常页面；
2. 异常页面应提供返回首页或返回地图的入口；
3. 异常页面不应暴露服务端错误堆栈或敏感信息。

#### 数据智能化生产管线功能需求

数据智能化生产管线是后续扩展任务，主要用于提升风味节点、传播事件和叙事材料的生产效率。

##### 风味向量提取需求

1. 管线应面向菜谱文本、地方美食志、饮食研究材料等语料；
2. 管线应从文本中提取食物名称、地域信息、食物类型、主要食材、食材角色和风味描述；
3. 管线应将风味描述转换为六维风味向量，并保证同一食物类型下的标度一致；
4. 管线输出应包含来源信息和置信度或审核状态；
5. 管线输出不得直接进入正式展示数据，应经过格式校验和人工复核。

##### 传播事件抽取需求

1. 管线应面向古籍、地方志、饮食交流史文献和研究论文等语料；
2. 管线应抽取传播对象、起点、终点、关键中转地、传播媒介、时期标签和原文摘录；
3. 管线应将历史地名和现代地理对象进行映射，并标记映射依据；
4. 管线应支持将传播事件与食材、生态区和叙事章节建立关联；
5. 对地名含混、来源不足或解释链条不完整的结果，应进入人工复核队列。

### 系统非功能性需求

#### 用户界面需求

系统界面应面向公众用户，兼顾博物志气质和 WebGIS 操作效率。

1. 界面应保持清晰、简洁、稳定，避免专业 GIS 工具式的密集按钮堆叠；
2. 地图应成为页面主要视觉对象，面板、图例和搜索控件不应遮挡核心信息；
3. 文字说明应简明可读，适合非专业用户理解；
4. 交互对象应具有明确的悬停、选中和禁用状态；
5. 详情面板、卡片和导航应在不同屏幕宽度下保持可读和可操作；
6. 系统应避免只依赖颜色表达状态，必要信息应配合文字或图形符号说明。

#### 软硬件环境需求

##### 硬件环境需求

| 设备类型 | 需求说明 |

| 数据库服务器 | 应具备稳定存储、空间索引查询和定期备份能力 |

#### 系统质量需求

1. 健壮性：当基础数据缺失、瓦片读取失败或数据库连接异常时，系统应给出明确反馈，并尽量保持页面可用；
2. 容错性：无效查询、空搜索结果和越界地图请求不应导致系统崩溃；
3. 易用性：核心功能应能在较少学习成本下完成，用户能够快速找到地图入口、搜索入口和详情信息；
4. 一致性：同类对象的展示样式、字段命名和交互反馈应保持一致；
5. 可扩展性：系统应支持风味节点、传播路径、叙事章节和数据来源的持续扩充；
6. 可维护性：数据规范、接口规范和文档说明应保持同步，便于后续团队继续建设。

#### 数据质量需求

1. 空间数据应统一坐标系，并在入库或加载前进行基本几何有效性检查；
2. 风味向量各维度应保持统一取值范围，缺失或异常数据不得直接进入正式展示；
3. 食材名称、食物类型、路线类型和生态区名称应采用统一字典或统一映射口径；
4. 传播事件应记录来源或审核状态，避免无法追溯的历史叙事被当作确定事实展示；
5. 数据导入后应进行数量、字段完整性、空间几何和关联关系检查；
6. 面向公众展示的数据应优先保证解释清楚、来源可查和表达一致。

#### 性能需求

1. 首页、风味基因库、时空叙事馆和关于页面应在常规网络环境下快速加载；
2. 地图页面应在底图、生态区、传播路径和风味节点同时展示时保持顺畅交互；
3. 瓦片服务应具备缓存机制，减少重复读取和重复渲染；
4. 搜索和列表类业务接口应在常规数据规模下快速返回；
5. 当数据规模增长时，系统应通过索引、分页、聚合或图层分级等方式控制加载压力；
6. 动态视觉效果不应明显影响地图浏览和对象选择。

#### 安全需求

1. 数据库连接信息、密钥和部署凭据不得写入公开代码或文档；
2. 服务端应对输入参数进行基本校验，避免异常请求影响服务稳定；
3. 原始栅格文件、源 Shapefile 和内部数据文件不应直接向公众开放下载；
4. 生产环境应通过 HTTPS、反向代理、跨域控制和必要的访问控制保障服务安全；
5. 数据备份应定期执行，并具备基本恢复流程；
6. 面向公众展示的来源链接和媒体资源应避免引入不可信脚本或不受控内容。
