# 总体设计

本章从系统总体组织角度说明“寻味地理”风味博物志 WebGIS 系统的技术路线、建设架构、B/S 架构下的 WebGIS 应用方案、软件平台配置和开发运行环境方案。系统总体设计以浏览器端地图展示、后端数据服务和空间数据库支撑为基本框架，形成面向公众浏览和饮食文化地理解释的 WebGIS 应用。

## 总体技术路线

系统总体技术路线可以概括为“基础地理数据整理—生态区与风味业务数据组织—空间数据库支撑—后端服务输出—前端 WebGIS 展示—数据扩展与审核闭环”的建设流程。该流程既保证空间数据、业务数据和叙事内容能够在统一框架下组织，也保证浏览器端能够以地图、图表和文本叙事的方式进行综合展示。

1. **基础地理数据整理**：整理自然地形底图、海岸线、河流和生态区边界等基础地理数据，统一空间参考和图层用途，为系统地图展示提供自然地理背景。
2. **生态区与风味业务数据组织**：以生态地理单元、菜肴成品、食材、传播事件、叙事章节和数据来源为主要对象，建立风味业务数据体系，并通过食材原产地、配方连结和菜肴溯源链表达对象间关系。
3. **空间数据库支撑**：采用对象关系数据库和空间扩展统一管理面状生态区、点状风味节点、线状传播路径和业务属性数据，使空间查询、属性查询和关系查询能够协同运行。
4. **后端服务输出**：后端统一提供业务数据接口、栅格瓦片服务、矢量图层服务和搜索服务，将数据库和基础地理数据转换为前端可直接使用的接口结果。
5. **前端 WebGIS 展示**：浏览器端完成地图渲染、图层叠加、节点与路线展示、详情信息展示、风味雷达图展示和页面导航，使用户能够在地图场景中理解风味形成和传播关系。
6. **数据扩展与审核闭环**：后续数据扩展采用语料收集、结构化抽取、人工审核、格式校验和批量导入的流程，保证风味节点、传播事件和叙事材料在扩充时保持字段、来源和表达口径一致。

## 系统建设架构

系统采用 B/S 架构下的三层组织方式，整体划分为展示层、服务层和数据层。展示层面向浏览器用户，服务层负责业务逻辑和数据输出，数据层负责空间数据、业务数据和基础地理数据的统一支撑。

### 展示层

展示层由浏览器端单页应用构成，承担地图浏览、图层显示、对象选择、页面导航、图表展示和叙事阅读等任务。用户不需要安装专用客户端，通过浏览器即可访问系统页面并完成地图探索、风味节点浏览和章节阅读。

展示层以地图为核心展示载体，在自然地理底图上叠加生态区边界、传播路径和风味节点。图表、详情面板和文本内容围绕地图选择状态展开，使空间对象和解释信息保持同步。展示层只通过服务层获取数据，不直接访问数据库和基础数据文件。

### 服务层

服务层由后端 Web 服务和开发辅助服务组成，负责业务接口、瓦片服务、矢量图层输出、搜索查询、健康检查和数据库访问。服务层对外提供稳定的 HTTP 接口，对内连接空间数据库并读取基础地理数据，从而屏蔽前端对数据库结构和数据文件组织方式的感知。

业务数据服务主要输出风味节点、传播路径、叙事章节、数据来源和搜索结果。空间服务主要输出自然地形瓦片、海岸线、河流、陆地轮廓和生态区边界等地图图层。服务层应保证接口结构稳定、异常响应明确，并通过缓存、索引和数据预处理降低重复计算压力。

### 数据层

数据层由 PostgreSQL/PostGIS 空间数据库和基础地理数据文件共同构成。空间数据库用于保存生态区边界、菜肴成品、食材、传播事件、叙事章节、数据来源及其关联关系；基础地理数据文件用于支撑自然地形底图、物理矢量和生态区基础图层的加载与输出。

数据层既保存结构化业务数据，也保存空间几何和半结构化风味向量。生态地理单元作为主要空间容器，连接菜肴成品、食材来源和传播事件；菜肴成品通过风味基因型属性表达六维风味向量；传播事件通过路径几何和叙事材料表达历史时空过程。

### 层间协同关系

浏览器端向服务层发送页面数据、地图图层和搜索请求，服务层根据请求访问数据库或基础地理数据，并将结果转换为前端可渲染的数据结构。展示层负责呈现和交互，服务层负责逻辑组织和数据转换，数据层负责持久化存储和空间支撑。

该分层方式能够保持系统边界清晰：前端关注展示和交互，后端关注数据服务和空间服务，数据库关注数据完整性和查询效率。后续扩充风味节点、传播事件或叙事章节时，优先通过数据层和服务层进行扩展，展示层按稳定接口消费数据。

## B/S 架构下的 WebGIS 应用方案

本系统采用浏览器/服务器结构。用户通过浏览器访问系统，前端页面负责地图展示和交互操作，后端服务负责数据查询、空间图层输出和瓦片生成，数据库和基础地理数据为后端服务提供支撑。该结构降低客户端安装和维护成本，也便于系统在不同浏览器环境中进行展示。

### 浏览器端展示方案

浏览器端采用单页应用方式组织页面。地图探索、风味基因库、时空叙事馆和关于页面在同一前端应用中完成切换，页面切换不依赖整页刷新。地图页面作为核心页面，负责综合展示自然地形底图、生态区边界、传播路径和风味节点。

地图渲染应保证底图、矢量图层和 WebGL 数据图层在同一空间参考下叠加。用户选择风味节点、传播路径或生态区后，前端根据选中对象更新详情面板和图表内容。风味雷达图用于表达菜肴成品的风味基因型属性，文本说明用于解释风味形成、生态背景和传播关系。

### 后端服务方案

后端服务统一承担业务数据服务和空间数据服务。业务数据服务面向风味节点、传播路径、叙事章节、数据来源和搜索结果，输出适合前端页面直接消费的数据结构。空间数据服务面向地图图层，输出栅格瓦片和矢量图层数据。

后端服务不应将数据库表结构直接暴露给前端。对于前端而言，风味节点、传播路径、生态区和叙事章节是展示对象；对于数据层而言，菜肴成品、食材、传播事件、叙事章节和数据来源是主要数据对象。服务层负责完成两类口径之间的转换，保证页面展示稳定。

### 空间数据与业务数据协同方案

系统通过生态地理单元将自然环境与风味业务数据建立联系。生态区边界用于表达自然空间背景，菜肴成品通过坐标位置落在地图上，食材通过来源关系与生态区建立关联，传播事件通过路径几何表达跨区域流动，叙事章节则对典型路线和风味现象进行解释。

在地图展示中，生态区提供面状背景，传播事件提供线状过程，菜肴成品提供点状风味节点。三类空间对象相互叠加，使用户能够同时观察“在哪里形成”“沿什么路径传播”“与何种生态条件相关”等问题。业务属性、风味向量和叙事文本通过详情面板和页面内容补充空间图形无法直接表达的信息。

### 数据扩展与维护方案

系统数据扩展以批次整理和审核为主。新增风味节点、食材、传播事件和叙事章节时，应先完成语料收集和结构化整理，再进行来源核对、字段校验、空间映射和人工审核，最后进入数据库和前端展示流程。

风味向量类数据应保证维度和取值范围一致，传播事件类数据应保证起止位置、路径表达和来源说明清楚，叙事章节类数据应保证文本可读且能与地图对象建立关联。数据更新后，应检查接口输出、地图展示和页面说明是否保持一致。

## 软件平台配置

系统软件平台围绕浏览器端展示、后端服务、空间数据库和空间数据处理进行配置。主要软件或技术如下表所示。

| 类型 | 软件或技术 | 用途 |
| --- | --- | --- |
| 前端框架 | Vue 3、Vite | 构建浏览器端单页应用，支持页面组织和开发调试 |
| 地图渲染 | MapLibre GL JS | 展示自然地理底图、矢量图层和地图交互 |
| WebGL 可视化 | Deck.gl | 展示传播路径、动态效果和风味节点等高性能图层 |
| 图表展示 | ECharts | 展示风味雷达图和相关统计图表 |
| 状态管理 | Pinia | 管理风味节点、传播路径、生态区和页面状态 |
| 后端框架 | FastAPI、Uvicorn | 提供业务接口、瓦片服务和后端运行支撑 |
| 静态服务辅助 | Express、Node.js | 支撑开发服务和静态资源链路验证 |
| 数据库 | PostgreSQL/PostGIS | 管理空间几何、业务属性、关联关系和空间索引 |
| 空间数据处理 | rio-tiler、rasterio、Fiona、Shapely | 处理栅格瓦片、矢量数据和几何对象 |

上述软件平台能够满足本系统对 WebGIS 展示、空间数据读取、业务数据查询和前端交互展示的需要。前端平台负责可视化表达，后端平台负责服务输出，数据库平台负责数据管理，空间数据处理工具负责在数据文件、空间数据库和地图服务之间完成必要转换。

## 开发运行环境方案

系统运行环境按开发阶段组织。开发环境需要保证浏览器端页面、前端开发服务、后端服务、Node 辅助服务、PostgreSQL/PostGIS 数据库和基础地理数据均可正常访问。

### 开发环境组成

开发环境由客户端浏览器、前端开发服务、Node 辅助服务、后端 Web 服务和空间数据库组成。客户端浏览器用于访问系统页面；前端开发服务用于页面热更新和前端资源加载；Node 辅助服务用于静态资源链路和服务联调；后端 Web 服务用于输出业务接口、瓦片和矢量图层；空间数据库用于保存生态区和风味业务数据。

### 开发运行流程

开发运行时，用户通过浏览器访问前端页面，前端页面根据用户操作向后端请求业务数据、地图瓦片和矢量图层。后端服务根据请求访问 PostgreSQL/PostGIS 数据库或基础地理数据，完成查询、读取和转换后返回给前端。前端接收结果后完成地图图层渲染、详情内容更新和图表展示。

该流程保证前端、后端和数据层职责分离。前端只处理展示和交互，后端统一处理数据访问和服务输出，数据库和基础地理数据不直接暴露给浏览器端。

### 数据库与基础数据依赖

系统开发运行依赖 PostgreSQL/PostGIS 数据库和基础地理数据。数据库应启用空间扩展，并保存生态地理单元、菜肴成品、食材、传播事件、叙事章节和数据来源等数据。基础地理数据应包括自然地形底图、物理矢量和生态区边界等内容，用于支撑地图底图、空间背景和图层输出。

基础数据在进入服务链路前应完成坐标参考检查、几何有效性检查和必要的数据整理。业务数据在进入数据库前应完成字段完整性检查、来源检查和关联关系检查，避免页面展示中出现对象缺失或关系断裂。

### 环境配置要求

开发环境应保证前端服务、后端服务、Node 辅助服务和数据库可以稳定协同。数据库连接信息应通过环境变量或本地配置文件提供，不应写入公开文档或源代码。基础地理数据应放置在后端服务可访问的位置，并在服务启动时完成必要检查。

开发环境还应具备基本的异常反馈能力。当数据库连接失败、基础地理数据缺失或瓦片读取失败时，后端应返回明确的错误状态，前端应给出可理解的页面反馈，避免用户将数据缺失误认为系统整体不可用。
