第一章 系统概况

1.1 项目背景与问题提出

“寻味地理”的出发点不是再做一个普通美食地图。传统美食地图通常回答“哪里有什么”“哪家店更值得去”，它解决的是位置索引和消费决策问题，却很少解释一道地方风味为什么会在这个地点形成，又为什么在迁移到另一个地点后发生改变。对于饮食文化研究、城市文旅展示和地方品牌传播来说，仅有地点标注是不够的。真正有价值的是把食材、气候、水系、交通、迁徙和历史贸易放到同一张空间框架里，让用户看见风味形成背后的地理逻辑。

项目当前选择“风味”作为核心对象，而不是把餐馆、菜谱或行政区作为唯一入口。以火锅为例，成都和重庆的麻辣牛油锅、海南的椰子鸡火锅、广州的粥底火锅、北京的铜锅涮羊肉、贵阳的酸汤鱼火锅、哈尔滨的酸菜白肉锅，都可以被归入“火锅”这一食物属，但它们的汤底、香辛料、主料、保存方式和味觉重心并不相同。这种差异不是菜单名称上的差异，而是自然条件、物产结构、交通输入和地方饮食习惯长期叠加后的结果。系统要表达的正是这种“同源而异构”的风味演化。

从当前技术条件看，项目已经具备把这一问题可视化的基础。Natural Earth 物理底图能够提供不依赖行政边界的自然地理视觉基底，WWF TEOW 全球陆地生态区划可以作为生态单元框架，PostgreSQL/PostGIS 能够承载空间数据与业务数据的关联查询，MapLibre GL JS 能够在浏览器中完成交互式地图渲染，FastAPI 能够对外提供瓦片、节点、路线、搜索和食材传播接口。项目要解决的问题不是“能不能把点画在地图上”，而是如何把风味知识组织成可积累、可解释、可复用的数据资产。

因此，本系统的建设意义可以概括为三点。第一，把地方风味从零散文本、图片和经验描述中抽象为结构化对象。第二，把食物差异放回生态地理和历史传播的空间语境中解释。第三，为后续面向文旅、教育、展陈、区域品牌和内容服务的产品化应用保留接口、数据模型和内容生产机制。

1.2 系统定位

“寻味地理”定位为一个面向公众展示和行业内容服务的风味地理知识平台。它的前台形态是 WebGIS 交互地图和多页面叙事体验，后台基础是空间数据库、结构化风味数据、食材传播数据和可持续扩展的数据生产流程。系统既不是餐饮点评工具，也不是单纯的电子图册；它更接近一种“可交互的风味博物志”，用地图组织知识，用路线讲述传播，用卡片和叙事页面降低普通用户的理解门槛。

在当前研发阶段，系统重点验证三件事。第一，建立稳定的地图底座，将自然地理、生态区、风味节点和传播路径叠加到同一套坐标系统中。第二，建立可扩展的数据模型，使一个地方风味既能被地图选中，也能进入风味基因库、叙事章节和搜索结果。第三，形成具有审美辨识度的交互体验，让地图不是单纯的工程界面，而是具有文化叙事感的探索入口。

面向产业化时，系统的定位需要进一步从“项目展示”转向“内容与数据服务”。这意味着产品价值不只来自一个前端页面，而来自可持续维护的风味知识库、可复用的地图组件、可输出的专题内容包、可对接外部平台的 API，以及围绕地方饮食文化形成的采集、审核、编辑和发布流程。只有这样，系统才有可能服务于文旅宣传、博物馆展陈、学校课程、地方农产品品牌、城市文化传播和内容出版等多类场景。

1.3 核心设计思想

系统采用“风味基因型”的概念来组织业务数据。这里的“基因型”不是生物学意义上的严格遗传模型，而是一种便于结构化表达的设计隐喻：同一种食物可以在不同地域形成不同变体，每个变体都携带一组可以被记录和比较的特征，包括地点、生态环境、食材结构、六维风味向量、代表性标签和文献或图片来源。例如“重庆麻辣火锅”和“海南椰子鸡火锅”都属于火锅谱系，但前者强调麻、辣、油脂和江湖码头饮食，后者强调甜、鲜、椰青和热带物产，两者应当在同一模型下可比较，而不是被拆成互不相干的条目。

系统的第二个核心思想是用生态地理单元补充行政区叙事。行政区适合管理和检索，但不一定能解释物产、气候和饮食习惯。TEOW 生态区、地形底图、河流海岸线等自然地理数据可以帮助用户理解风味形成的环境条件。系统并不把生态因素绝对化，也不宣称某种味道由地理环境单独决定，而是把生态背景作为解释地方风味的底层语境。

系统的第三个核心思想是把“传播”作为叙事线索。风味不是静止分布的，辣椒、香料、茶叶、稻米、盐、发酵技术和烹饪器具都会沿着海路、陆路、河流、移民路线和商贸网络流动。当前系统已经提供传播路线数据、叙事章节和食材传播页面，后续应把传播事件进一步拆解为证据点、时间段、置信度、交通媒介和文献来源，使传播路线既具有视觉冲击力，也经得起资料追溯。

1.4 建设目标

本系统的近期建设目标，是完成一个可运行、可展示、可继续扩展的风味地理 WebGIS 原型。该原型应能支持用户从首页进入地图，在地图上查看风味节点、传播路线和生态背景；用户可以通过风味基因库浏览地方风味卡片，通过时空叙事馆阅读历史章节，通过食材传播页面查看食材从起源地到中国地方饮食体系中的传播过程；系统还应提供搜索和详情面板，使用户能够在“地图探索”和“内容阅读”之间自然切换。

本系统的中长期建设目标，是形成一套面向产业化的风味知识生产与服务体系。它应当能够持续接收新的菜品、食材、路线、文献和图片资料，通过统一的数据结构入库，通过人工审核和版本管理保证质量，再通过地图服务、专题页面、开放接口和内容包输出到不同业务场景。换言之，未来系统的竞争力不应只停留在一次性页面效果，而应体现在“数据能沉淀、内容能生产、接口能复用、场景能落地”。

目标层面当前研发阶段产业化演进方向
数据基础以风味节点、传播路线、叙事章节和食材传播 JSON 作为种子数据，验证结构是否可用。扩展为可持续维护的风味知识库，覆盖更多菜系、食材、路线、文献和图片来源。
地图表达完成自然地理底图、生态区、风味节点、传播路径和详情面板的组合展示。形成可复用的 WebGIS 展示组件，按城市、主题、展览或课程快速生成专题地图。

1.5 当前实现基础

当前项目已经不是只停留在概念层面的设计稿，而是具备前后端分离、地图渲染、空间数据服务和页面路由的可运行原型。开发环境中，浏览器访问 Vite 前端服务，前端通过代理请求 FastAPI 后端；后端再读取 PostgreSQL/PostGIS 数据、项目内 JSON 数据和本地 GIS 资源，并通过 REST API、GeoJSON、MVT 和栅格瓦片向前端提供数据。

层面当前实现对总体设计的意义
前端应用采用 Vue 3、Vite、Vue Router 和 Pinia，已形成首页、探索地图、风味基因库、时空叙事馆、食材传播、关于页面等路由。证明系统不是单页地图演示，而是可以承载多入口、多叙事形态的前端应用。
地图渲染核心地图使用 MapLibre GL JS，主地图叠加 Natural Earth HYP 栅格、海岸线、河流、生态区、风味节点和传播路径。为“自然地理底图 + 风味知识图层”的视觉策略提供技术基础。
可视化组件使用 ECharts 表达六维风味雷达图，地图交互中包含节点选择、路线选择、详情面板、投影切换和镜头飞行。使抽象的风味向量和传播路径转化为用户可读的视觉对象。
后端服务FastAPI 提供 /api/flavors、/api/routes、/api/chapters、/api/data-sources、/api/search，以及 /api/ingredients/spread 相关接口。把前端展示和数据来源解耦，为后续服务化和数据治理保留接口边界。
瓦片与空间数据后端通过 rio-tiler 读取 Natural Earth 栅格瓦片，通过 GeoJSON/MVT 提供海岸线、河流、生态区等矢量数据，并使用 LRU 缓存提升瓦片响应。说明系统具备真正的 WebGIS 底层能力，而不是静态图片或普通网页动画。
数据库设计PostgreSQL/PostGIS schema 包含生态地理单元、风味基因型、食材、食材产地、传播事件、配方关联、叙事章节和数据来源等表。为风味节点、食材、路线、生态区和文献来源之间建立结构化关系。
食材传播项目内已设计 data/ingredient/*.json 结构，包含食材起源、时间线、传播段、参考文献、置信度和可视化参数。为后续表达辣椒、香料、茶叶等食材的跨区域传播提供统一录入规范。
运行方式Windows 下通过 start.ps1 启动，FastAPI 默认 8001，Vite 默认 5173，Express 开发代理默认 3001。保证项目能被稳定复现，为后续演示、验收和部署整理打基础。

1.6 面向产业化的演进方向

如果把当前系统继续向产业化方向推进，重点不应只放在增加页面数量或堆叠视觉效果上，而应放在“数据资产化”和“内容产品化”两条主线上。数据资产化要求每一个风味节点、食材传播事件、图片、文献和解释性文字都能够被追溯、被复用、被更新；内容产品化要求这些数据能够被组合成不同客户或场景需要的产品形态，例如城市美食地图、非遗饮食展陈、研学课程、地方农产品品牌故事、短视频脚本素材和专题网页。

产业化版本可以拆成四个层次。第一层是数据资产层，负责维护标准化的风味节点、食材、传播事件、生态区、配方关系和来源记录。第二层是知识生产层，负责把地方志、论文、菜谱、展馆资料、图片素材和采访文本转化为可入库的数据，过程中可以使用大语言模型辅助提取，但必须保留人工审核。第三层是服务层，负责提供稳定 API、地图瓦片、专题配置和可嵌入组件。第四层是应用层，面向文旅、教育、展览、媒体和地方品牌输出具体产品。

在这个方向上，系统未来最重要的能力不是“自动生成很多内容”，而是建立可信的数据生产流程。食物历史常常存在地域争议、年代不确定和文献表述差异，系统需要记录证据来源、推断依据、置信度和审核状态。对于老师、研究者或行业客户来说，可信度往往比视觉炫目更重要。只有把资料来源和编辑流程做清楚，地图上的每一条路线、每一个节点、每一句解释才有长期使用价值。

能力维度原型阶段做法产业化阶段要求
数据规模少量种子节点和路线，用于验证模型与视觉效果。建立覆盖多食物属、多食材、多地区的数据集，并支持持续增量更新。
数据可信度人工整理为主，部分字段用于展示，来源粒度仍需继续细化。每条事实具备来源、摘录、置信度、审核人、更新时间和版本记录。
内容生产页面内容与代码、JSON、数据库记录混合维护，适合研发阶段。建立采集、清洗、结构化、审核、发布的工作流，降低后续维护成本。
产品交付以 Web 前端演示为主要交付形态。支持专题地图、展陈大屏、教育课件、城市专题站、API 数据服务等多形态交付。
商业场景以项目展示和功能验证为主。面向文旅部门、博物馆、学校、地方品牌、内容机构和餐饮文化 IP 提供服务。
运营维护开发者直接维护数据和代码。引入后台管理、权限、数据审校、发布状态、日志监控和备份策略。

1.7 本阶段建设边界

为了避免系统目标失焦，本阶段需要明确建设边界。第一，当前系统不是全国权威美食数据库，种子数据用于验证系统结构和表达方式，不能被表述为完整覆盖。第二，系统不提供餐馆推荐、点餐、团购或实时消费信息，也不以“哪家店好吃”作为主要问题。第三，历史传播路线属于解释性可视化，路线形态应服务于历史脉络表达，不能被理解为精确到道路级别的 GPS 轨迹。第四，大语言模型可以参与资料抽取和初步结构化，但不能代替文献核验、人工判断和版权审查。

同时，系统也不应把地理环境解释成单一决定因素。自然地理、生态资源和交通条件确实会影响食材获得、保存方式和口味偏好，但政治、贸易、移民、宗教、经济和技术同样会改变饮食结构。因此，系统在叙事上应坚持“多因素解释”，在数据上保留来源和不确定性，在视觉上避免用过度确定的符号暗示复杂历史只有唯一答案。

1.8 术语说明

术语说明
寻味地理 / FlavorScape本项目名称，指在地理空间中探索地方风味形成、差异和传播的 WebGIS 知识平台。
风味基因型系统核心业务对象，表示某一地域中的具体地方风味变体，包含地点、菜品、食物属、风味向量、食材、生态背景和展示信息。
风味向量用数值方式表达一道地方风味在麻、辣、咸、酸、甜、鲜等维度上的强弱，便于比较和可视化。
食物属对同源或相近烹饪形态的抽象分类，例如火锅、烤鸭、面食等，用于比较不同地域变体。
生态地理单元用于描述自然环境背景的空间单元，当前主要参考 WWF TEOW 生态区数据。
传播事件某种食材、风味、技法或饮食习惯在时间和空间中的一次迁移记录，可包含起点、终点、时间、媒介、路线和证据。
食材传播时间线针对单一食材建立的历史节点序列，用于表达起源、传入、扩散、地方化和文献记录。
证据点支撑某一传播节点或风味解释的文献、图像、地方志、研究资料或人工整理记录。
产业化本文中指系统从研发原型发展为可持续运营的数据、内容、接口和场景服务，而不仅是商业销售行为。

1.9 本章小结

总体来看，“寻味地理”的系统价值不在于把若干美食点位摆上地图，而在于把地方风味组织为可解释、可追溯、可扩展的数据对象，并通过 WebGIS 把这些对象放回自然环境和历史传播的空间关系中。当前项目已经具备前端页面、后端接口、空间瓦片、数据库结构和种子数据基础，下一阶段应围绕数据质量、内容生产流程和行业场景落地继续深化。

后续总体设计应在这一定位下展开需求规定、系统架构、数据库设计、功能设计、接口设计和界面设计，并始终区分“已经实现的原型能力”和“面向产业化的演进目标”。这样写出来的文档既能对应当前项目实际，也不会把尚未完成的部分包装成空泛承诺。
