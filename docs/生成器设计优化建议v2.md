## 🛠️ 核心 Bug 深度解构与药方

### Bug 1：导出 Canvas 崩溃（Unsupported color function "color"）

- **报错信息：** `Error: Attempting to parse an unsupported color function "color" at Object.parse (color.ts:15:23)`
- **病因：** 你的前端样式（可能是由于使用了现代 CSS 特性或某些暗黑模式的主题库）使用了较新的 CSS 颜色语法（例如 `color(display-p3 ...)`、`color-mix()` 或某些未被标准解析的 CSS 变量颜色）。而 `html2canvas` 这个库的原理是**人工去模拟解析 computedStyle 并画到 Canvas 上**，它极其古老，根本不认得这种现代 CSS 颜色函数，于是直接崩溃罢工。
- **解法：** 强迫 Codex 在执行导出的那个容器（`#capture-zone`）内，将所有背景、文字颜色强制转换为最安全的十六进制（Hex，如 `#ffffff`）或标准 `rgba()` 颜色，避开高级颜色函数。

### Bug 2：地图图标加载大小不匹配（mismatched image size）

- **报错信息：** `RangeError: mismatched image size. expected: 0 but got: 36864 ... Image "ingredient-icon-pepper" could not be loaded.`
- **病因：** 在 `IngredientSpreadView.vue` 里面，Codex 尝试用 `map.addImage()` 往 Mapbox/Gaode GL 地图里注册花椒和大米的图标。但在传入图片数据时，它直接塞入了原始的像素数组（ArrayBuffer），却没有显式指定图片的 `{ width, height }` 参数；或者它在图片还没有完全异步加载完成（`onload` 之前）就急着调用了注册函数，导致 WebGL 上下文解析尺寸对不上（预预期是 0，却抓到了 36864 字节的数据）。
- **解法：** 用标准 JavaScript 的 `new Image()` 预加载包装成 Promise，等图片完全 onload 拿到宽高后，再执行 `map.addImage`。

### Bug 3：数据状态未绑定（五常大米显示汉源花椒文本）

- **视觉表现（`QQ_1780300341960.jpg`）：** 你已经切换到了“五常大米”，右侧的图标也换成了大米，但左侧醒目的平面大字依然写着“**汉源花椒**产地公示”。
- **病因：** 典型的数据硬编码（Hardcoded Text）。Codex 在重构布局时，把这一行大字写死了，忘记绑定你全局的 `activeIngredient.name` 变量。
