系统概况

本章主要对“寻味地理”风味博物志 WebGIS 系统的建设背景、目标、任务边界和核心术语进行说明。本系统当前以 WebGIS 展示、空间数据库和前后端服务为建设重点，以 LLM 辅助数据生产管线作为后续规模化数据建设的重要方向。文档在描述已实现能力时以当前项目代码和数据结构为准，在描述后续方向时强调数据资产、内容生产和行业场景的可持续扩展，避免把尚处于规划阶段的能力写成已经完成的功能。

系统背景

饮食文化具有天然的地域性。不同地区的气候、水系、地形、物产和历史交流条件，会长期影响当地人对食材、烹饪方式与味型的选择。传统美食地图通常以“某地有什么菜”为主要表达方式，能够帮助用户定位美食，却难以进一步说明“为什么这种味道会在这里形成”“同一种食物为什么会在不同地区演化出不同版本”。对于本项目而言，地图不只是信息标注工具，而是组织风味知识、解释地域差异和承载文化叙事的基础界面。

例如，同属“火锅”的地方食物，在重庆表现为重油、重麻、重辣的红汤锅，在海南表现为以椰青和文昌鸡为核心的清甜锅底，在北京则表现为铜锅清水涮羊肉。它们并非彼此孤立的食物条目，而是在不同生态条件、物产结构和传播路径影响下形成的风味变体。若仅以行政区划或菜品名进行展示，难以呈现其背后的自然环境和历史传播逻辑。

“寻味地理”风味博物志 WebGIS 系统正是在这一背景下提出。系统将地方风味视为可以被编码、比较和解释的知识对象，将自然地理环境、食材供给和历史传播作为理解风味差异的重要线索。系统以自然地理单元而非行政边界作为主要空间框架，以风味节点、食材、传播路径和叙事章节作为业务对象，通过 WebGIS 可视化方式展示风味在地理空间中的形成、传播与变异。

从技术角度看，当前系统采用前后端分离的 B/S 架构。前端基于 Vue 3、MapLibre GL JS、Deck.gl 和 ECharts 构建交互式地图与图表界面；后端基于 FastAPI 提供业务数据接口、栅格瓦片服务、矢量图层服务和食材传播接口；数据库采用 PostgreSQL/PostGIS 统一管理生态区空间数据和风味业务数据。系统已完成以 Natural Earth 物理底图、WWF TEOW 生态区、17 个种子风味节点、5 条传播路径、5 个叙事章节和食材传播时间线为核心的演示系统建设，为后续扩充完整的风味地理知识库奠定基础。

从后续发展看，本系统不宜只被理解为一次课程展示或单一网页作品。它更适合作为一套风味文化数据底座的原型：前端页面负责把知识讲清楚，后端接口负责把数据稳定输出，数据库负责沉淀可复用的结构化资产。只有把这三层关系建立清楚，项目后续才有可能面向文旅展示、博物馆展陈、研学课程、地方品牌传播和内容出版等场景扩展。

系统目标

“寻味地理”风味博物志 WebGIS 系统的总体建设目标为：利用 WebGIS、空间数据库和前端可视化技术，建设一个面向公众的饮食文化地理探索平台，实现对地方风味、生态环境、食材构成和历史传播路径的综合展示，使用户能够在地图中直观理解“地理如何塑造味道”。在此基础上，系统还应为后续产业化演进预留数据治理、接口复用和专题内容生产能力，使项目从“可展示的原型”逐步转向“可持续维护的风味知识服务”。

围绕上述目标，系统建设需达到以下要求：

  1. 建设以自然物理底图为视觉基底的 WebGIS 展示平台，弱化现代行政边界，突出地形、水系和生态气候区对风味形成的影响；
  2. 建立生态地理单元、菜肴成品、食材、传播事件、叙事章节和数据来源等核心数据对象，并设置食材原产地、配方连结、菜肴溯源链等关联实体，形成统一的数据组织体系；
  3. 使用六维风味向量（麻、辣、咸、酸、甜、鲜）描述地方食物的风味特征，使不同地域的同类食物可以进行定量比较；
  4. 通过传播路径和流光动画展示食材、技法和味型在历史时空中的迁移方向，表达风味演化的动态过程；
  5. 实现节点、路线、生态区之间的联动选择，使用户在交互过程中形成“生态环境—食材供给—风味表达—历史传播”的关联认知；
  6. 提供地图探索、风味基因库、时空叙事馆、关于方法论等多种浏览方式，兼顾自由探索、结构化比较和线性叙事；
  7. 建立可扩展的数据生产、审核与入库机制，使当前手动标注的种子数据能够在后续通过 LLM 辅助管线扩展为规模化内容库，并保留来源、置信度和审核状态；
  8. 形成稳定的前后端服务与空间数据库支撑体系，保证系统在常规浏览、搜索、瓦片读取和图层切换场景下运行稳定。

根据系统建设的总目标，建设任务主要包括数据库建设、应用系统建设、基础地理数据处理、数据生产管线建设、标准规范建设和运行维护保障六个方面。

  1. 数据库建设

数据库建设是本系统的基础工作。系统采用 PostgreSQL/PostGIS 作为统一的数据管理平台，建立生态地理单元表、风味基因型记录表、食材表、传播事件表、叙事章节表和数据来源表等核心实体表，同时建立食材原产地关联表、配方连结表和基因型溯源链表等关联实体表。数据库中既包含 MultiPolygon、Point、LineString 等空间几何，也包含 JSONB 风味向量属性、数组标签和关系型外键约束。文档中的“菜肴成品”主要指业务语义，当前代码实现中对应的物理表为 flavor_genotype。

其中，关联实体表用于表达核心实体之间的多对多关系或自引用关系。食材原产地关联表用于连接食材与生态地理单元，表达食材可能对应多个原产生态区的情况；配方连结表用于连接风味基因型记录与食材，并记录食材角色、贡献权重和是否为本地物产等关系属性；基因型溯源链表用于表达地方风味记录之间的祖先与后代关系，为后续描述风味变异过程预留数据结构。

当前数据库已导入 827 个 WWF TEOW 全球陆地生态区，录入 17 个种子风味节点、5 条种子传播路径、5 个叙事章节及相关食材数据。数据库通过 GiST、GIN 和 B-Tree 索引支持空间查询、半结构化数据查询和多表关联查询，为前端地图展示和全局搜索提供支撑。

  1. 应用系统建设

应用系统建设包括前端展示系统和后端服务系统。前端采用 Vue 3 单页应用结构，包含首页、探索地图、风味基因库、时空叙事馆、食材传播和关于方法论六个主要页面。地图页面是系统核心，通过 MapLibre GL JS 展示物理底图和生态区边界，通过 Deck.gl 展示主地图中的传播路径、流光粒子和风味节点，通过 ECharts 展示风味雷达图和生态区气候示意图。食材传播页面则围绕单一食材的时间线、路径节点和镜头叙事展开，是后续专题化内容产品的重要入口。

后端采用 FastAPI 提供 RESTful 数据服务和瓦片服务。业务数据接口负责返回风味节点、传播路径、叙事章节、数据来源和搜索结果；瓦片接口负责从 Natural Earth HYP COG 栅格中生成 PNG 瓦片，并提供海岸线、河流、生态区等矢量图层。Vite 在开发环境中承担前端热更新和开发代理职责，Express 在生产环境中承担前端静态资源托管，并将 API 与瓦片请求代理至 FastAPI。

  1. 基础地理数据处理

基础地理数据处理主要包括 Natural Earth 栅格底图、Natural Earth 10m 物理矢量和 WWF TEOW 生态区划的整理与加载。系统使用 Natural Earth Cross Blended Hypsometric Tints 作为无国界线的自然地形底图，使用海岸线和河流矢量作为辅助地理骨架，使用 WWF TEOW 全球陆地生态区划作为风味形成的自然空间容器。

所有矢量数据统一采用 WGS84（EPSG:4326）坐标系。生态区边界在入库和服务输出时进行几何校验与简化，以降低前端渲染和网络传输压力。栅格底图以 COG GeoTIFF 形式存储，后端通过 rio-tiler 按需读取并生成瓦片，不需要预先切片。

  1. 数据生产管线建设

当前系统的风味节点和传播路径仍以人工标注的种子演示数据为主，数据覆盖范围和学术引用深度有限。为解决后续规模化内容生产问题，系统设计 LLM 辅助数据生产管线。该管线并非当前 WebGIS 展示链路的运行前置条件，而是后续数据扩展的重要组成部分。产业化阶段尤其需要避免“内容越多越好”的简单扩张，应优先建立语料来源、结构化抽取、人工审核、版本发布和争议标注机制，使扩展后的数据能够被长期维护和对外服务。

风味向量提取管线面向菜谱文本、地方美食志和饮食研究材料，目标是将非结构化文字转化为标准化的风味向量、食材列表、食材角色和食物属分类。传播事件结构化录入管线面向古籍、地方志和饮食交流史文献，目标是抽取传播物、起点、终点、传播媒介、时期标签和文献摘录。两条管线产出的结构化数据经人工审核后，按系统数据库导入。

  1. 标准规范建设

系统建设需要形成统一的数据和接口规范。空间数据方面，应统一坐标系、几何类型、生态区命名和边界简化策略；风味数据方面，应统一风味向量维度、取值范围、食材类型和食物属标注口径；传播事件方面，应统一路线类型、关键节点、时间标签和文献引用方式；接口方面，应统一 API 路径、响应格式和错误处理方式。

上述规范既服务于当前系统开发，也服务于后续数据扩充、团队协作和外部交付，避免在数据规模扩大后出现字段含义不清、风味标度不一致、路线几何不可比较等问题。若系统面向文旅、教育或展陈场景输出专题内容，统一规范还可以降低重复整理成本，使同一批数据能够在地图页面、展陈大屏、课程材料和接口服务中复用。

  1. 运行维护保障

系统运行维护主要包括后端启动检查、数据库连接池初始化、瓦片缓存管理、静态资源托管、环境变量配置和数据备份。当前开发环境由 FastAPI 后端服务、Express Node 服务和 Vite 前端开发服务器协同运行；生产环境使用 Express 托管前端构建产物，并在 Node 服务中将业务接口与瓦片请求代理至 FastAPI。如需统一域名、HTTPS 证书或负载均衡，可在外层增加 Nginx 等反向代理组件。

系统通过环境变量配置数据库连接，避免在代码中写入真实凭据。数据库可通过定期逻辑备份保障数据安全，基础数据和种子数据可通过建库流程与导入流程重建。

系统建设任务

“寻味地理”风味博物志 WebGIS 系统的建设是实现饮食文化空间化表达、风味特征定量化表达和历史传播可视化表达的基础。本系统并非传统的业务管理系统，也不是单纯的美食展示网站，而是以空间数据和数字人文叙事为核心的 WebGIS 应用。后续若向产业化方向演进，系统建设任务还应服务于数据沉淀、内容复用和场景交付，而不只是完成页面功能。

系统建设任务可以划分为以下几个子任务。

系统总体构架

系统总体构架包括前端展示层、后端服务层和数据层三个部分。展示层负责地图渲染、页面交互和图表展示；服务层负责业务 API、瓦片服务、矢量图层服务和启动时数据加载；数据层负责 PostGIS 空间数据、风味业务数据和栅格底图文件的存储管理。

开发环境中，浏览器访问 Vite 开发服务器，Vite 将业务接口和瓦片接口请求代理到 FastAPI。Express 服务在开发时同步启动，用于验证 Node 服务链路；生产模式下，Express 托管前端构建产物，并将业务接口与瓦片请求代理至 FastAPI。系统总体上保持前后端分离，前端不直接访问数据库，所有数据均通过后端接口获取。

空间数据库体系建设

空间数据库体系建设包括生态区空间数据库和风味业务数据库两部分。生态区空间数据库以 WWF TEOW 生态区为核心，记录生态区名称、编码、生物地理界、生物群系、面积和边界几何。风味业务数据库围绕风味基因型记录、食材、传播事件和叙事内容展开，记录地方食物的风味向量属性、坐标位置、食材构成、传播路径和章节文案。

在风味业务数据库中，风味基因型记录、食材、传播事件和叙事章节属于核心实体；食材原产地、配方连结和基因型溯源链属于关联实体。关联实体通常以两个外键共同组成主键，用于将多对多关系转化为可存储、可查询、可扩展的数据表；当关系本身具有业务含义时，还可在关联实体中记录关系属性，如食材在某一地方风味中的角色、贡献度和本地性等。

系统采用单版本数据库设计，不再沿用传统 GIS 项目中的现势库、历史库和工作库划分。其原因在于本系统目前以展示、科普和批次导入为主，数据更新频率较低，且当前数据规模较小。后续如需保留历史版本，可在传播事件、风味节点和叙事章节表中扩展版本字段或建立独立历史表。

基础数据管理系统开发

基础数据管理系统主要由后端启动流程和数据库导入流程承担。后端启动时检查 HYP 栅格文件是否存在，加载 Natural Earth 物理矢量图层，并在数据库连接池初始化后从 PostGIS 中读取生态区边界。数据库初始化、种子数据导入和后续 JSON 批量导入均通过项目数据导入工具完成。

该任务的重点不是提供复杂的可视化后台管理界面，而是保证系统在开发和演示环境中能够稳定重建数据、加载图层和输出接口。对于课程设计或原型系统而言，这种脚本化管理方式更轻量，也更贴合当前数据规模。

WebGIS 应用系统开发

WebGIS 应用系统开发是本项目的核心任务。地图页需实现 L0 至 L3 四层空间图层的叠加展示，支持地图缩放、漫游、平面/球面切换、节点点击、路线点击、生态区点击、图层开关和右侧详情面板展示。系统通过 Pinia 保证节点、路线、生态区三种选择状态互斥，避免多个对象同时占用详情面板造成交互混乱。

除地图页外，系统还需要建设风味基因库、时空叙事馆和关于方法论页面。风味基因库提供结构化卡片浏览，适合用户横向比较不同风味节点；时空叙事馆提供线性章节阅读，适合展示典型传播事件及其历史材料；关于方法论页面说明数据来源和数据清洗思路，增强系统可信度。

数据智能化生产管线建设

数据智能化生产管线是系统后续扩展的重要任务。当前系统已在数据库结构和数据导入流程上为其预留承接方式，但完整自动化采集、提示工程、审核和批量导入流程仍需后续建设。该管线应包括语料收集、LLM 结构化抽取、人工审核、格式校验和数据库导入几个环节。对于面向外部场景输出的数据，还应补充版权状态、可展示范围和编辑版本信息，确保后续内容交付有据可查。

在风味向量生产中，需保证同一食物类型内标度一致，避免不同菜品之间数值不可比。在传播事件生产中，需保证地名、历史时期和生态区映射准确，避免因古今地名差异导致路径错误。LLM 输出不能直接入库，应经过校验和人工复核。

数据资产化指将风味节点、传播事件、食材、来源和叙事内容按统一结构沉淀，使其能够被查询、审核、复用和对外服务。
TEOWTerrestrial Ecoregions of the World，世界自然基金会发布的全球陆地生态区划数据。
HYPNatural Earth Cross Blended Hypsometric Tints，高程分层设色的自然地形栅格底图。
COGCloud Optimized GeoTIFF，云优化 GeoTIFF 格式，支持按需读取局部栅格区域。
PostGISPostgreSQL 的空间数据库扩展，支持几何类型、空间索引和空间分析函数。
MVTMapbox Vector Tile，一种紧凑的矢量瓦片格式。
LRULeast Recently Used，最近最少使用缓存淘汰策略。
MapLibre GL JS开源 WebGL 地图渲染引擎，本系统用于底图和生态区边界渲染。
Deck.glWebGL 数据可视化框架，本系统用于传播路径、流光粒子和风味节点光晕展示。
PiniaVue 3 状态管理库，本系统用于管理节点、路线、生态区选择状态。
SPASingle Page Application，单页应用。前端页面切换由 Vue Router 管理，无需整页刷新。
LLMLarge Language Model，大语言模型。本系统将其作为后续风味数据和传播事件结构化生产的辅助工具。
